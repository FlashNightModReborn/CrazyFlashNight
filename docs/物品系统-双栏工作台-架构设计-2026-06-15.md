## 2. 核心抽象：双栏工作台

把「物品面板」升维为**通用双栏工作台**：双栏只是两个 view-slot，背包/技能/仓库/商店都是可插拔 **View**；物品界面、技能界面只是不同的 View 配置。

```
DualPaneShell（壳：双 view-slot + PanelScale 缩放 + 换栏 + 跨栏中介）
   ├─ slotL ─┐
   ├─ slotR ─┴─ 各挂一个 View（任一槽皆可换）
   └─ TransferBroker（能力撮合：source × sink → AS2 命令）

View（可插拔）
   ├─ CatalogView      商店目录（非容器：购买意图的来源）
   ├─ GridContainerView  ← 网格容器 primitive（见 §6）
   │     背包 / 仓库 / 战备箱 / 资源箱 / 购物车 / 已购买 全是它的配置
   ├─ EquipView        装备槽 + 装备属性
   ├─ SkillView        技能栏
   ├─ EnhanceView      强化 / 强化度转换 / 插件改装
   └─ CraftView        合成
```

**第一性原则**：壳与 View 互不知道对方是谁，只通过 Broker 的**能力契约**对话。这是避免 N² 耦合、不留债的核心。

**视图单例不变量**（壳职责）：同一个 View 默认不允许同时挂在两个槽。用户在 A 槽选了一个已在 B 槽显示的 View 时，壳**自动对调**——B 槽让位，回落到它的默认 View。这条通用规则取代了任何"避免双购物车"之类的视图专属状态机，且对后续多视图（背包+仓库同屏腾挪等）自动适用。displaced 槽的去向因此是良定义的，不是特判。

## 3. 两种转移模态（关键区分）

| 模态 | 例子 | 特征 | 发出的命令 |
|---|---|---|---|
| **①a 同质 owned 转移** | 背包↔仓库/战备箱 | 即时、逐件、落点即提交 | `inventoryMove`/`inventoryMerge`/`inventorySwap`（§4.1，仅稳定同质 owned 容器；资源箱瞬态、暂排除，见 §4.3/§6） |
| **①b 显式领域动作** | 装备/卸装、领取(claim)、出售 | 即时，但**含专属业务逻辑**（装备=等级检查+外观刷新；claim=移除待领记录+入包+强制存盘；出售=定价+金钱） | 各自显式命令 `equip`/`unequip`/`claim`/`sell`，**不是**通用转移 |
| **② 暂存转移** | 商城购买（目录→购物车→结账→已购买→claim）、将来合成/批量出售 | 累积进缓冲 → 单次 commit → 常带代价（K点/金钱/配方） | view 专属 `checkout`/`craft` 批量提交 |

双栏语言**中心是模态①**；模态② 的 UI（购物车/结算）**保持视图本地**，直到第二个视图（合成/批量出售）真的需要同款暂存，再机械式抽取到壳级 staging 托盘。命名好抽取点，但不提前付泛化的钱。

## 4. 权威与命令模型

- **AS2 永远是权威态**，Web 只是投影。沿用 snapshot + command + `dirtyMark`。写操作必须由 AS2 权威路径重新校验价格/权限/槽位/状态（见迁移护栏 §7）；Web 端禁用按钮只是 UX，不是安全校验。

### 4.1 命令分层（修正 itemTransfer 过宽）

不存在一个吃下一切的 `itemTransfer`。命令按模态分层：

- **模态①a 同质 owned 转移**——仅用于 owned↔owned 的稳定同质容器（背包/仓库/战备箱）：`inventoryMove` / `inventoryMerge` / `inventorySwap`。（资源箱是瞬态容器，暂不入此列，见 §6。）
- **模态①b 显式领域命令**——`equip` / `unequip` / `claim` / `sell` / `enhance` 等各自携带专属业务（等级检查、外观刷新、移除待领记录、强制存盘、定价、强化），**不得**塞进通用转移。
- **模态①c 整容器写**——`inventorySortAndMerge`（真实重排槽位 + 合并堆叠 + dirtyMark，权威实现见 [ItemSortUtil.as:37](../scripts/类定义/org/flashNight/arki/item/ItemSortUtil.as#L37)）：既非逐件转移、非领域动作、非批量提交，而是整容器重组写，亦走本节信封；与 Web 本地 `displaySort` 严格区分（§6）。
- **模态② 批量提交**——`checkout` / `craft`。

**命令信封（向后兼容）**。信封要求**仅适用于新增 inventory-domain 及后续模态① owned 命令**；既有 KShop `shop*` 命令是模态② view 专属命令，首战**不强制套信封、不改其形状**（见末尾"信封适用范围"）。

```
{
  v: 1,                       // 信封版本，便于后续字段演进
  domain: "inventory",        // 领域（非展示槽）
  op: "inventoryMove",        // 命令名
  source: { containerId, slot, expectedRevision, expectedFingerprint },
  target: { containerId, slot, expectedRevision, expectedFingerprint },
  count: 1,
  ctx: { ... }                // 可选扩展上下文，未知字段 AS2 忽略（向后兼容）
}
```

关键约束：

- **`containerId` 是稳定的领域容器标识**（如 `背包`/`仓库`/`战备箱`），**不是可换栏的展示槽（slotL/slotR）或 View 实例**。Web 换栏不影响命令寻址。
- **双端并发校验**：跨容器 move/merge/swap 时，目标容器的变化**不影响源 revision**，故 source 与 target **都**必须携 `expectedRevision + expectedFingerprint`，由 AS2 在**同一临界区**内对两端一并比对、通过后才执行；任一端不符即拒绝回 `stale_state`，Web 重拉相关 snapshot。（先前只在源端带 token 是漏洞：目标格仍会被陈旧命令覆盖。）
- **指纹是真正的防线，revision 只是可选快路径**。现有集合**不带 revision**，且 [ItemCollection.as:23](../scripts/类定义/org/flashNight/arki/item/itemCollection/ItemCollection.as#L23) 返回可变物品引用、[:37](../scripts/类定义/org/flashNight/arki/item/itemCollection/ItemCollection.as#L37) 暴露底层 items——游戏逻辑可绕过集合方法**就地改装备属性（等级/tier/mods）**，"每次 mutate 自增 revision" 无法保证。故 v1 的权威并发防线是**执行时双端 fingerprint**（由 AS2 在执行点按当前内容现算并与 expected 比对，捕获任何来源的改动）。`expectedRevision` 仅作为低成本快路径，且只有当 revision 维护在能观测**全部** mutation 的层（领域注册表或集合基类改造，且封堵就地改引用）时才可信——此为后续课题，未完成前不得把 revision 当唯一依据。
  - **fingerprint 定义**：堆叠/dict 格 = `name|count`；背包/仓库等装备格 = `name|count|level|tier|mods` 的稳定序列化（就地改属性也会变指纹）。
- **AS2 最终裁决**：所有合法性（容量、use 匹配、背包-only、金钱/K点、等级）在执行时由 AS2 领域服务重判；Web 侧 capabilities 仅预检。

**信封适用范围**（解决"统一信封"与"首战不改 KShop 协议"的冲突）：信封 + 双端 token 是**新命令族（inventory-domain、后续模态①）的契约**；KShop 的 `shopBulkQuery/saveCart/checkout/claim` 首战**原样保留**，不回填信封。若日后要给 shop* 套信封，单列为协议迁移项。

### 4.2 渐进

- **v1 用 round-trip 确认**（快车道低毫秒级，够用）；乐观更新 + snapshot 对账作为第二步增量，不在首版背回滚复杂度。
- **重投影**：AS2 mutate 后，受影响 collection（携当前 fingerprint，revision 可选）重新下发 snapshot 片段，Web 重渲染。

## 5. View 契约

每个 View 向壳/Broker 声明四件事：

1. **数据需求**：订阅哪些 snapshot 键（如背包订阅 `inventory.背包`）。
2. **作为 source 的能力（UX 预检）**：能"吐出"什么。例：`BackpackView` 声明自己可作 equip/sell/enhance/move 的来源——用于灰显/提示。**注意：这是展示层投影，不是权限 SOT**；"背包-only"等真实约束由 AS2 领域服务持有并最终裁决（§1.2），Web 能力来自 snapshot 下发的 capabilities。
3. **作为 sink 的能力（UX 预检）**：能接收什么 + 落点提示。例：`EquipmentSlotView` 预检 `use` 匹配；`CartView` 只收目录来的购买意图。同样仅预检。
4. **渲染与局部交互**：自身网格/列表/页脚，以及不经 Broker 的本地动作（翻页、displaySort/filter、tooltip）。

Broker 职责**仅交互撮合**：拖拽/点击发起时用 source × sink 能力做**本地预检**（不合则不发命令、给即时反馈），通过则按 §4.1 分层发对应命令（①a 发 `inventoryMove` 等、①b 发显式领域命令、②发批量命令）。**最终合法性永远由 AS2 重判**；Broker 不是权限或业务规则 SOT。Web capabilities 过期时，AS2 拒绝并回错误码，Web 重拉 snapshot。

## 6. GridContainerView primitive（复用基石）

一个网格容器组件，参数化三件：

- **item-model**：`owned`（真实拥有实例）/ `intent`（购买意图 `{catalogIdx, qty}`）/ `owned-待领`。
- **itemActions（逐格）**：单格出口/编辑动作。如 owned 格 = `move`/`equip`/`claim`；intent 格（购物车）= `changeQty`/`remove`（走 `saveCart`，**非** checkout）。
- **viewCommitActions（整 View 批量）**：模态② 的整体提交。如购物车 = `checkout`、合成 = `craft`。**勿与逐格动作混为一谈**：逐格改的是缓冲内容，commit 才是带代价的整体结算。
- **sink-accept**：作为落点接受什么。

通用能力：网格渲染 + 拖拽/点击 + 筛选 + 批量动作 + 分页。

**排序必须拆成两个语义，勿混**（现有 `sortInventory` = `rebuildOrder` 重排槽位 + `mergeStackables` 合并堆叠，是权威写，见 [ItemSortUtil.as:37](../scripts/类定义/org/flashNight/arki/item/ItemSortUtil.as#L37)）：

- **`displaySort` / `filter`**：Web 本地展示重排/过滤，**不改真实槽位**，无命令、无存档。
- **`inventorySortAndMerge`**：AS2 权威写命令（真实重排槽位 + 合并堆叠 + dirtyMark），走 §4.1 信封。Web 端"整理"按钮触发的是这个，不是 displaySort。

### 纠正线（必须守住，否则留债）

**只在视图/交互层统一，数据/所有权 + 出口命令按实例保持区分。**

仓库格子里是**已拥有的真实物品实例**；购物车格子里是**购买意图**（尚未拥有）。两者长得像、交互像，但语义与出口命令不同。"购物车是仓库的变体"——**是组件复用意义上的变体，不是数据所有权意义上的变体**。绝不可把购物车内容物建模成 owned 槽位，否则把"购买意图"语义漏进所有权模型。

### 容器分类法

| 容器实例 | item-model | itemActions（逐格） | viewCommit | sink-accept |
|---|---|---|---|---|
| 背包 | owned + 主库特权 | equip/sell/enhance/move | — | move / claim / 购买结果 |
| 仓库 / 战备箱 | owned（容量/分页规则） | move | — | 背包来的 owned |
| **购物车** | **intent** | changeQty/remove（→`saveCart`） | **checkout** | 目录来的"加购意图" |
| **已购买** | owned-待领 / 回执 | claim→背包（模态①b）/ 只读 | — | — |

战备箱页数派生见 [物品栏UI.as:501](../scripts/展现/UI交互/UI交互_lsy_物品栏UI.as#L501)。

**资源箱暂排除**：资源箱库存数据在 [地图元件.as:62](../scripts/逻辑/关卡系统/关卡系统_lsy_地图元件.as#L62) 瞬态 `new ArrayInventory(null, cap)`（cap=row×col）即时构造，**库存数据对象无稳定 containerId、不入任何领域注册表**（其 UI MovieClip 经 [物品栏UI.as:518](../scripts/展现/UI交互/UI交互_lsy_物品栏UI.as#L518)→UI管理 attachMovie 加载，UI 层虽有 `外部UI列表` + 场景转换时 `卸载外部UI()` 粗粒度清理，但数据对象本身无 per-container 寻址能力），不满足 §4.1 命令寻址前提。近期**不纳入**可寻址转移模型；将来纳入须设计 session-scoped containerId + 失效错误（容器已关闭时拒绝命令）+ 关闭清理。其视觉网格仍可复用 GridContainerView 渲染，但**不接入 Broker 转移**。

## 7. 交互模型：点击为主 + 拖拽加速

- **点击为主**：点选源格 → 点目标栏/格（或双击直送对侧、Ctrl 快移）。可靠、round-trip 语义干净、与既有命令模型同构。
- **拖拽加速**：pointer 式手写拖拽作为锦上添花，失败可降级回点击。

### CDP 可行性（已确认）

输入经 InputShield → CDP `Input.dispatchMouseEvent` 注入。**结论**：

- HTML5 原生拖放（`draggable`/`dragstart`/`drop`）在合成事件下**不触发**——全仓零使用，禁用路线。
- **pointer 式手写拖拽完全可用**，生产级证据：lockbox `setPointerCapture`（[lockbox-panel.js:773](../launcher/web/modules/minigames/lockbox/lockbox-panel.js#L773)）、jukebox 音量滑块 mousedown→mousemove→mouseup（[jukebox.js:516 `initSlider`](../launcher/web/modules/jukebox.js#L516)）/ 进度条 seek 同模式（[jukebox.js:573](../launcher/web/modules/jukebox.js#L573)）、map pointermove 命中（[map-hotspot-hitcapture.js](../launcher/web/modules/map/map-hotspot-hitcapture.js)）。
- 拖拽实现走 `pointerdown → setPointerCapture → pointermove(更新拖拽镜像) → pointerup → releasePointerCapture`；无 touch 事件（CDP 只模拟鼠标）。
- 注意 InputShield 在 hitRect 收缩后有 **~800ms move 抑制窗口**（源码称「收起冷却」：hitRect 缩小后 800ms 内直接不下发 CDP `mouseMoved`，防 mouseenter 重新展开振荡，`InputShieldForm.cs:136/332-335/488-491`；是全量抑制而非限流），开合 UI 时勿在拖拽中触发。

拖拽镜像**已定走 Web 自绘**（不续用 AS2 `_root.鼠标.物品图标容器`），与"AS2 UI 全面外迁"长期战略一致。

## 9. 首战：KShop 加重版

### 9.0 现有 KShop 真实数据生命周期（确权，勿凭直觉）

现有 KShop（[kshop.js](../launcher/web/modules/kshop.js)）内部已是两区：目录网格 + sidebar（购物车 + 结账 + 已购买），固定画布 1024×576。其权威生命周期见 [商城系统_WebView.as](../scripts/逻辑系统分区/商城系统_WebView.as)：

| 阶段 | gameCommand | 改什么 | 注意 |
|---|---|---|---|
| 加载 | `shopBulkQuery` | 只读：目录 + `_root.商城购物车` + `_root.商城已购买物品` + K点(`_root.虚拟币`) + **玩家等级 `_root.等级` + 逆向被动等级 `_root.主角被动技能.逆向`**（[:103-104](../scripts/逻辑系统分区/商城系统_WebView.as#L103)，CatalogView 等级锁所需，勿漏） | 购物车有**服务端恢复影子**（供重开恢复），会话内意图主居 Web |
| 存购物车 | `shopSaveCart` | 写：用 `params.cart` 覆写恢复影子 `_root.商城购物车`（handler [:202](../scripts/逻辑系统分区/商城系统_WebView.as#L202)，覆写写点 [:207](../scripts/逻辑系统分区/商城系统_WebView.as#L207)） | |
| 结账 | `shopCheckout` | 写：结算 **Web 上送的 `params.cart`**（[:117](../scripts/逻辑系统分区/商城系统_WebView.as#L117)，**不读**持久影子）+ 扣 K点 + →`商城已购买物品` + 清空影子 + `强制存盘` | **不碰背包**；K点判定**严格 `_root.虚拟币 > total`**（[:134](../scripts/逻辑系统分区/商城系统_WebView.as#L134)，非 `>=`）——恰好等额时回 `insufficient_kpoints` 失败，迁移须原样照搬勿静默改 |
| 领取 | `shopClaim` | 写：`商城已购买物品`→背包 + 强制存盘；**两条失败路径**：背包无空位 `getFirstVacancy()==-1`→`inventory_full`（[:179](../scripts/逻辑系统分区/商城系统_WebView.as#L179)）、`singleAcquire` 返回 false→`acquire_failed`（[:194](../scripts/逻辑系统分区/商城系统_WebView.as#L194)）；**回包只含 `purchased`，不含背包** | **真正入背包在此，非 checkout**；方案 A 重投影失败需同时覆盖两错误码 |

由此修正先前文档错误：① 购物车并非"纯 Web 本地"也非"纯服务端持久"——它是 **会话内 Web 主态 + saveCart 恢复影子**，且 **checkout 结算的是 Web 上送的 `params.cart` 而非读取影子**（二者可能不一致）；② **背包改变发生在 claim，不是 checkout**；③ "批量领取"语义属于**已购买 View**（owned-待领 → 背包），购物车（intent）的逐格动作是增删/改量，不是领取；④ **claim 回包不含背包**，背包刷新需另建通道（§9.1/§9.2）。

**首战不改这套生命周期**（仍是"请求 cart 结账"模型），只在其上叠加双栏壳 + 容器视图 + 背包并排；沿用 `shopBulkQuery/saveCart/checkout/claim` 原语义。**是否迁移为"AS2 持久 cart 结账"**（checkout 改读 `_root.商城购物车`）留作单独立项——两者的并发、恢复、失败语义不同，首战不动。

### 9.1 加重版叠加（不改既有协议）

- **槽位**：左∈{目录, 购物车} / 右∈{背包, 购物车}，视图单例不变量产三配置（§11）；默认进面板态 = (目录, 购物车)。
- **视图**：CatalogView=现有目录逻辑包成 View；CartView=GridContainerView(model=intent, itemActions=changeQty/remove→`saveCart`, viewCommit=`checkout`)；**已购买是独立容器**（model=owned-待领, itemActions=claim），v1 UI 可沿用现有 sidebar 把 购物车/已购买 两区并置于同一槽（CartView 复合），也可拆独立可换视图——布局自由，但**数据与 egress 必须分开**。
- **目录→购物车** = sink 接受"加购意图"（可拖可点，**理想的低风险拖拽沙盒**：错了零成本、不碰真实库存）→ 走 `shopSaveCart` 写恢复影子。
- **背包视图刷新通道（高优先，必须显式选定）**：现有 `shopClaim` 回包**只含 purchased、不含背包**，且 kshop.js 现仅刷新已购买区——**claim 后背包重投影没有现成闭环**。v1 选 **方案 A：Web 在 claim 成功回调里链式补发一次 `domain=inventory snapshot`**（claim 成功 → 触发 inventory snapshot → `inventory_response` → 背包视图刷新）。优点：**不改 AS2 claim 回包形状**（守"不改既有协议"）、不建推送通道；代价：一次额外 round-trip（v1 可接受）。备选 B（claim 回包带背包 delta）/ C（建 inventory push 通道）留作后续优化，**不在首战**。
- **背包满时领取的功能边界**：claim 需背包空位，满时**拦截+明确提示**，不静默失败（沿用 AS2 现有 `getFirstVacancy()==-1` 判定）。(购物车/已购买, 背包) 同屏即为此场景。v1 腾格子工具箱仅限背包自身 egress（至少"扔到垃圾箱"）；卖出/挪仓库需对应目标/视图存在，留到 NPC 商店、仓库步骤。
- 拖拽镜像 Web 自绘（§7）。

定位诚实声明：KShop 首战是**壳 + GridContainerView + 模态② + claim 重投影**的试车；模态①双向 Broker（owned↔owned）的完整证明在第 2 步（背包↔仓库）。

## 10. 债务防线小结

1. 容器只在视图层统一，所有权/命令按实例区分（§6 纠正线）。
2. 背包-only 约束的 SOT 在 **AS2 领域服务**；视图能力声明只是 snapshot 下发的 UX 预检，不是权限 SOT（§1.2、§5）。Broker 仅交互撮合，不裁决业务规则。
3. 模态② UI 视图本地，第二消费者出现才抽壳级托盘，命名抽取点但不提前泛化（§3）。
4. 命令分层（§4.1）：①a 同质 owned 用 `inventoryMove/merge/swap`；①b 装备/卸装/claim/sell 用显式领域命令（带专属业务）；②批量用 checkout/craft；不存在吃下一切的 `itemTransfer`。逐格 `itemActions` 与整 View `viewCommitActions` 分开（§6）。
5. 命令寻址用**稳定 containerId**（非展示槽）。并发校验 **source 与 target 双端**都带 token、同一临界区比对；**fingerprint 是权威防线**（集合泄漏可变引用、装备就地改属性，revision 无法保证自增，仅作可选快路径），fingerprint 由 AS2 执行点现算（§4.1）。信封带版本 `v` 向后兼容，**仅适用新命令族，不回填 KShop shop\***。
6. 排序拆 `displaySort`(Web 本地) vs `inventorySortAndMerge`(AS2 写)，勿混（§6）。
7. AS2 权威不动；Web 禁用钮≠安全校验；写操作 AS2 重校验（§4、护栏 §7）。
8. 共享背包走独立 `InventoryTask`/`inventory_response`，**`domain=inventory` 顶层先于 panel 路由判断**（否则裸 snapshot 落入 MapTask），勿绑进任何单面板回包通道（§9.2）。
9. 跨通道刷新显式建链：claim(shop_response) → Web 链式 inventory snapshot(inventory_response)；AS2 claim 回包不含背包，不得假定其自动刷新（§9.1）。
10. 资源箱等瞬态容器**暂不接入 Broker 转移**（库存数据对象无稳定 ID、不入领域注册表；UI 层虽有粗粒度清理，数据对象本身不可寻址，§4.3/§6）；纳入前补 session-scoped containerId + 失效错误 + 关闭清理（§6）。

## 11. 已定细节（2026-06-15）

- **双栏槽位 UX**：**左∈{目录, 购物车} / 右∈{背包, 购物车}**，由**视图单例不变量**（§2）自动产出且仅产出三种配置，双购物车被排除：
  - (目录, 背包) 逛+看空间 ｜ (目录, 购物车) 逛+攒单（**面板初始进入态**）｜ **(购物车, 背包) 背包满时腾格子领取**。
  - **「默认」消歧**（两义勿混）：右槽的**结构默认 View = 背包**（displaced 时的回落目标）；**面板初始进入态 = (目录, 购物车)**（进面板首屏，右槽初值=购物车）。
  - 当右=购物车、在左点"购物车"，单例规则触发对调→右槽 displaced 后回落其**结构默认 View（=背包）**→正好落到 (购物车, 背包) 腾格子态。比购物车专属状态机更省，对后续背包+仓库同屏腾挪同一条规则适用。
  - 取舍：切右栏不打断浏览；背包+购物车同屏是**背包满才能领取**的刚需（非锦上添花）。早期"左=目录⇄购物车 / 右=背包"的窄方案已被本方案取代。
- **设计画布**：**1024×576**，对标既有面板/Flash（Stage 尺寸本作动态，落到实处即沿用 PanelScale 标准画布）。好处：AS2 网格几何可原样搬（图标 28px、背包 10 列=280px）。约束：每栏可用约 480–500px，施工保持布局紧凑。
- **乐观更新**：v1 全程 round-trip 确认；**功能完成后再补**乐观更新 + snapshot 对账。
- **拖拽镜像**：**Web 自绘**（不续用 AS2 `物品图标容器`），与"AS2 UI 全面外迁"长期战略一致。
