## ADR-001: 技术栈选择

### 状态
已接受 (2026-03-06)

### 背景
需要选择一套技术栈来实现数值平衡工具，要求：
1. 类型安全，便于维护
2. 支持CLI和GUI两种模式
3. 与AS2代码概念映射容易
4. Agent开发友好

### 决策
采用 **TypeScript + Node.js + Electron + React** 技术栈：

| 组件 | 选型 | 理由 |
|------|------|------|
| 语言 | TypeScript (strict) | 与AS2概念接近，类型系统辅助纠错 |
| 运行时 | Node.js >=18 | 生态丰富，tsx直接执行 |
| 测试 | Vitest | 输出结构化，Agent解析友好 |
| XML处理 | fast-xml-parser | preserveOrder保留格式 |
| Excel | SheetJS (xlsx) | 仅用于legacy导入 |
| GUI | Electron + React + Vite | 独立桌面应用 |
| Schema | Zod | 运行时校验+类型推导 |

### 后果
- **正面**: 类型安全，开发体验好，社区支持强
- **负面**: 需要Electron打包，体积较大

## ADR-002: 数据源单一化

### 状态
已接受 (2026-03-06)

### 背景
存在多个数据来源：Excel表格、XML配置文件、散落的md文档。需要确定单一数据源。

### 决策
**XML是真正的数据源**，Excel仅作为legacy import通道。

```
工作流:
游戏XML数据 <──双向读写──> 平衡工具 <──公式引擎──> 计算内核
                                 |
                   CLI / Electron GUI / Agent 均可操作
```

### 后果
- **正面**: 消除数据不一致，Agent可直接修改XML
- **负面**: 需要实现完整的XML round-trip保格式

## ADR-003: 公式引擎与XML分离

### 状态
已接受 (2026-03-06)

### 背景
公式引擎计算的值（如DPS）是否应该写回XML？

### 决策
**公式引擎计算值为参考指标，不回写XML**。

```
XML item (baseData) -> 提取字段 -> 公式引擎计算 -> 参考值(仅展示)
                                        |
                                        v
                               用户/Agent根据参考值
                               决策后修改XML字段
```

### 后果
- **正面**: 避免计算值与基础数据不一致
- **负面**: 用户需要理解"参考值"vs"实际值"的区别

## ADR-004: 分层XML解析器

### 状态
已接受 (2026-03-06)

### 背景
不同品类XML结构差异巨大：装备用`<item>`，插件用`<mod>`，怪物用中文节点名。

### 决策
**采用分层解析器架构**，而非单一adapter。

```typescript
// 三层解析器
interface EquipmentParser   // <item> 武器/防具
interface ModParser         // <mod> 插件
interface ConsumableParser  // <item> 消耗品
interface EnemyParser       // 中文节点名 怪物
```

### 后果
- **正面**: 每个解析器职责单一，易于测试
- **负面**: 代码量略增，需要维护多个解析器

## ADR-005: 插件系统计算范围

### 状态
已决定 (2026-03-06) - 待实施验证

### 背景
插件系统有复杂数值建模（495行文档），需要决定P1阶段做到什么深度。

### 决策
**分三阶段实现**:

| 阶段 | 计算深度 | 说明 |
|------|----------|------|
| P1 | 单插件效率 | 回本等级/基础增益计算 |
| P2 | 多插件组合 | 叠加效果估算 |
| P3 | 射线建模 | MC仿真复现 |

### 原因
- 插件数值已在`射线插件数值建模_2026-02.md`中有详细设计
- 但射线伤害涉及游戏内命中检测，需逐步验证

## ADR-006: 怪物属性支持策略

### 状态
已决定 (2026-03-06) - 待用户确认

### 背景
怪物属性XML结构与物品完全不同，需要决定是否支持及何时支持。

### 决策
**延后到P3或更晚**，理由：
1. 结构完全不同（中文节点名、min/max区间）
2. 需要独立解析器
3. Excel有712个怪物公式，复杂度高
4. 优先保证装备系统完善

### 替代方案
如需提前支持，建议：
- 仅做CRUD不做公式计算
- 使用完全独立的解析器模块

## ADR-007: 双轨公式体系

### 状态
已发现 (2026-03-06) - 需明确

### 背景
发现存在两套武器计算公式：
- **Excel公式**: 25列精细计算（DevSpec描述）
- **Workflow公式**: 简化快速估算（weapon_weighting_workflow.md）

### 决策
**工具同时支持两套体系**：

| 体系 | 用途 | 实现优先级 |
|------|------|------------|
| Excel公式 | 精细化平衡计算 | P0 - 必须 |
| Workflow公式 | 快速估算筛选 | P2 - 可选 |

### 说明
两套公式服务于不同场景，不是替代关系。

## ADR-008: 版本控制策略

### 状态
待定 (2026-03-06) - 需用户决策

### 背景
工具生成的changelog如何与Git协作？

### 选项

| 选项 | 描述 | 优缺点 |
|------|------|--------|
| A | changelog作为独立文件提交 | 历史清晰，但增加文件 |
| B | 仅本地存储，git diff追踪 | 简单，但缺少语义信息 |
| C | 自动生成changelog commit | 自动化，但可能污染commit历史 |

### 建议
**B + C 结合**：
- 平时开发用git diff追踪XML变更
- 批量平衡调整时生成changelog commit

## ADR-009: 多阶继承展开策略

### 状态
已决定 (2026-03-06)

### 背景
防具和部分刀类有多阶数据(data_2/3/4)，继承data基础值。

### 决策
**解析时展开，内部存储完整值**：

```typescript
// 解析阶段
baseData: { level: 20, hp: 100, defence: 50 }
tierData[0] (data_2): { modslot: 3, level: 28, hp: 150 }  // 继承后: defence: 50
tierData[1] (data_3): { modslot: 3, level: 35, hp: 200 }  // 继承后: defence: 50

// 回写时还原为差异
data_2只写变化的字段: { modslot: 3, level: 28, hp: 150 }
```

### 原因
- 内部计算需要完整字段集
- 回写时保留原始差异格式，最小化diff

## ADR-010: 校准通过标准

### 状态
已决定 (2026-03-06)

### 背景
需要将Excel公式翻译成TS，需要定义"翻译正确"的标准。

### 决策

| 模块 | 目标通过率 | 可接受偏差 | 处理方式 |
|------|------------|------------|----------|
| 枪械DPS | >95% | <0.1% | 强制修复 |
| 防具总分 | >90% | <1% | 强制修复 |
| 近战锋利度 | >90% | <1% | 强制修复 |
| 经济定价 | >85% | <5% | 记录偏差 |
| 药剂效果 | >90% | <1% | 强制修复 |

### 偏差处理流程
1. 标记偏差公式和具体数值
2. 检查是否是Excel缓存值精度问题
3. 检查是否是公式理解偏差
4. 使用`CALIBRATION_TODO`注释标记
5. 继续推进，不阻塞开发
