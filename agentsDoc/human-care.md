# 人类节奏与会话健康

**文档角色**:长会话节奏 / agent 主动行为 canonical doc。  
**最后核对代码基线**:commit `d4f31beee`(2026-04-20)。

本文是 **agent 在长会话中主动行为的约束**。项目主维护者经常单次 5-8 小时连续工作,这个时长会显著影响代码与决策质量;agent 应承担一部分让节奏可持续的责任。

## 1. 推荐节奏(默认参考,可被覆盖)

| 时段 | 默认 |
|------|------|
| 单段专注 | 60-90 min |
| 段间休息 | 5-15 min(离开屏幕) |
| 单次会话总长 | ≤ 3h 为宜,> 4h 进入软停窗口 |
| 连续高强度日 | 不超过 2 天,第 3 天起降低改动复杂度 |

## 2. 该停的信号(任一触发,主动建议收一手)

- 同一段代码反复改三次仍不满意
- 调试同一 bug 超过 45 min 没进展
- 对话出现「先这样吧」「无所谓了」「随便」
- 同会话内出现明显矛盾的决策
- 准备开新方向,但当前方向有未提交中间态
- 对话变短、变敷衍、命令变模糊

## 3. Agent 主动行为约束

agent **不知道** wall clock,以**对话轮次显著**、**任务连续完成多个**、**上下文使用接近压缩阈值**作为代理信号。

| 阶段 | 触发动作 |
|------|----------|
| ~2h(轻量) | 自然停顿点提一次「先停 5-10 分钟?」,被否决就不再提 |
| ~4h(软停窗口) | 下次任务收尾时显式说「建议先归档进度,下一段开新任务」 |
| ~6h(强提示) | 对**新增大改动**给一次明确反馈「建议下一段再做,原因是 …」,坚持仍执行 |

### 主动建议归档(任一触发)

- 一次会话完成 ≥ 2 个独立任务
- 即将开始上下文消耗较大的新方向
- 维护者表达疲态或准备结束
- 跨子栈的中间态尚未稳定

### 主动建议拆任务(任一触发)

- 单次请求出现 ≥ 3 个「同时」「以及」「顺便」
- 涉及子栈数 > 2 且不是显式跨栈协议变更
- 单次新增预估 > 400 行
- 横跨 bug 修复 + 重构 + 新功能

详细工程阈值见 [agent-harness.md §1](agent-harness.md)。

### 拒绝沉默放大(疲劳期高风险请求,必须显式确认)

大范围 `replace_all` / 删文件 / 删分支 / 重置历史 / 跳过 hooks(`--no-verify`)/ 强推(`--force`)/ 改共享配置或 CI / 改权限。

## 4. 软停收尾流程

进入 4h+ 窗口、维护者表达「差不多了」「再做最后一件」时:

1. **收口而非开新坑** — 把当前最小任务收完
2. **强制归档** — 走 [self-optimization.md §4](self-optimization.md) 收尾检查
3. **commit 现场可提交的** — 避免下段会话面对脏状态
4. **写一句下段开场** — plan 或 memory 里记「下段从 X 开始」

「最后再做一件」几乎从不是最后一件。出现这类语言时,agent 应主动复述当前状态,而不是直接接活。

## 5. 表达边界

- 不贴标签(不说「你太累了」「状态不对」)
- 不数据化人类(不说「健康风险 X%」)
- 用工程语言:「现在改的是核心路径,建议明天看一眼再合」
- 被否决就接受,不连续提示同一件事
- 不在每条回复里挂健康提示;只在节点提
