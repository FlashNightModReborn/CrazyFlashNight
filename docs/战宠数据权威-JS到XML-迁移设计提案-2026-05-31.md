# 战宠配置数据权威回收设计提案（JS → XML/AS2）

**状态**：✅ 已施工（A1+B1，2026-05-31）。AS2 编译 0 错误；**欠游戏内端到端手测**。下方"施工序列"保留为设计记录。
**日期**：2026-05-31
**关联**：[as2-web-panel-migration.md](../agentsDoc/as2-web-panel-migration.md) §7 数据权威与转录
**背景**：战宠面板 Web 迁移期间，`launcher/web/modules/pet-data.js` 把价格/等级门槛/进阶次数上限/经验公式等硬抄进了 JS 静态文件，已与 AS2/XML 真源发散。本提案确定如何把数据权威收回，JS 只做展示。

---

## 1. 核心结论：这不是"JS→XML 重写"，而是"JS→删除"

调研发现一个关键事实：**`PetPanelService` 的 snapshot / adoptList / preview / tooltip 回包，已经在运行态把全部权威数据下发给了 JS**。

| JS 当前抄写块 | 真源 | AS2 是否已下发 |
|---|---|---|
| `PET_LIBRARY`（110 宠物全字段，pet-data.js:185-307） | `data/merc/pets.xml` → `_root.宠物库` | ✅ `handleSnapshot.petLib`（:137-156）已完整下发 |
| `SCHEMES` 数据字段（gold/kpoint/maxTier/unlockLevel/buttonText，:22-182） | `战宠进阶函数`（AS2 字面量） | ✅ `handlePreviewAdvance` / `handleTooltip` 已下发真值 |
| `SCHEMES.desc/effect`（静态串） | `战宠进阶函数.描述/详情页描述`（**function**） | ✅ preview/tooltip 执行真函数后下发真文案 |
| `CATEGORIES`（商城分类，:9-19） | `data/merc/pets.xml <PetStore>` → `_root.宠物商城列表` | ⚠️ `handleAdoptList` 下发了 adoptable 列表，但缺"分类名 + 网格行结构" |
| `calcXpForLevel`（写死 50→400，:337-340） | `引擎_lsy_战宠系统.as:41` + `_root.敌人属性表[兵种]`（经验区间来自敌人属性 XML） | ✅ `snapshot.xpNeeded` 已是真值 |

**含义**：pet-data.js 的三大数据块都是冗余双写。把权威收回的最小动作不是"再搭一条 XML→JS 管线"，而是**删掉 JS 副本，让 JS 直接吃已经存在的回包**。真源链路 `pets.xml / 战宠进阶函数 / 敌人属性表 → AS2 → snapshot → JS` 已经打通，只是 JS 没用它、而是平行维护了一份会腐烂的拷贝。

这也顺带修掉两个已知 bug：
- **maxTier 语义错误**：pet-data.js 给 基础训练/强化药剂/超级血清 都写 `maxTier:3`，但 AS2 `次数上限` 是累进的 **1/2/3**（三者共写同一个 `次数` 计数器）。snapshot/preview 下发的是每方案正确的 `次数上限`。
- **经验公式发散**（即 Flash-Night 自述的"读取战宠数据方面有问题"）：JS 写死 50→400，脱离兵种真实经验区间；snapshot 的 `xpNeeded` 是 `计算战宠升级所需经验(兵种,等级)` 真值。

## 2. 需要你拍板的两个决策点

### 决策 A：JS 怎么拿数据？

| | 方案 A1：snapshot 驱动（**推荐**） | 方案 A2：构建期生成 pet-data.js |
|---|---|---|
| 做法 | 删除 pet-data.js 的 `PET_LIBRARY/SCHEMES/calcXpForLevel`，JS 纯靠回包渲染；AS2 snapshot 补齐缺的少量字段（分类名等） | 写 `tools/export-pet-data.js` 从 pets.xml + 进阶字面量生成带 `AUTO-GENERATED` 头的 pet-data.js |
| 真源 | 运行态唯一真源 = AS2 回包（背后是 XML/敌人属性表） | 构建期快照，仍是副本 |
| 漂移风险 | 不可能漂移（没有副本） | 忘记重生成就漂移，需 build gate 兜 |
| 改动量 | 删 JS + 小幅扩 AS2 snapshot | 新增导出脚本 + build gate + 改 JS 引用 |
| 先例 | kshop.js / intelligence-panel.js（权威全走回包） | stage-select-data.js（XFL→JS 生成） |
| dev harness | harness 用独立 mock（已有） | harness 可直接吃生成物 |

**推荐 A1**：snapshot 已经下发 90% 的字段，A1 是最小且零漂移的改动，与本仓"权威面板（kshop/map/intelligence）数据全走回包"的主流模式一致。stage-select 走 A2 是因为它的数据来自 XFL 符号矩阵、运行态拿不全；战宠没有这个约束。

### 决策 B：进阶方案的"数值字段"要不要也搬进 XML？

进阶方案的**逻辑字段**（`条件/执行/单位进阶执行/描述/详情页描述`）是函数，**永远留在 AS2**，无可争议。争议在**数值字段**（`消耗金币/解锁等级/次数上限/称号/罗马数字`）：

| | 方案 B1：数值留在 AS2 字面量（**推荐**） | 方案 B2：数值搬进 pets.xml |
|---|---|---|
| 做法 | 数值继续写在 `单位函数_aka_战宠进阶.as` 的方案对象里，与其逻辑函数同处；snapshot 下发给 JS | 在 pets.xml 增 `<PetPromotion>` 段存数值，AS2 加载时注入方案对象 |
| 权威 | AS2 代码 = 进阶数值权威 | XML = 进阶数值权威（真正"回 XML"） |
| 利 | 数值与逻辑同处，不会"数值改了逻辑没改"；改动小 | 策划可不碰代码调进阶数值 |
| 弊 | 数值仍在代码里，非策划可编辑 | 数值与逻辑分家，25 方案需逐一拆解，易出"数值/逻辑错配" |

**推荐 B1**：进阶数值与其判定/执行逻辑强耦合（如 `次数上限` 直接被 `条件` 函数读），拆进 XML 收益小、错配风险高。**真正要紧的"消灭 JS 双写"无论 B1/B2 都靠 snapshot 达成**。若将来需要策划可视化调进阶数值，再单独做 B2。

> 注意区分：宠物库/商城数据（id/价格/解锁等级）**本来就在 pets.xml**，已是"XML 权威"，本决策只针对进阶方案的数值。

## 3. 推荐方案（A1 + B1）的施工序列

1. **扩 AS2 snapshot 覆盖缺口**（`PetPanelService.handleAdoptList`）：在下发 adoptable 时补 `categoryName` 与保留商城网格行结构（`_root.宠物商城列表[c].Name` + `List` 二维含 null 占位），让 JS 不再需要本地 `CATEGORIES`。
2. **修正 snapshot 的进阶字段语义**：确认 preview/tooltip 下发的 `maxTier` = 各方案真实 `次数上限`（1/2/3），并对"基础训练系共享 `次数` 计数器"的累进语义在回包里表达清楚（如下发 `currentTier` + `maxTier`）。
3. **删 pet-data.js 数据块**：移除 `PET_LIBRARY`/`define()`、`SCHEMES` 数据字段、`calcXpForLevel`；保留纯查询/展示 helper（`getCategoryForPet` 之类改为对 snapshot 数据做查找，或并入 panel 模块）。
4. **改 pet-panel.js 取数来源**：所有读 `PetData.*` 的渲染点改读 snapshot/preview/tooltip 回包字段；强化/删除灵石预览改用 `snapshot.xpNeeded`（删掉本地公式预算）。
5. **dev harness**：保留独立 mock（标注为 mock，不与生产取数路径混用）。
6. **文档同步**：按迁移护栏 §9，更新 `launcher/README.md` 战宠条目（标注数据全走回包、无 JS 静态副本）。

## 5. 不在本提案范围

- 进阶**逻辑**（条件/执行函数）：永久 AS2，不动。
- 宠物初始战斗属性：来自 `敌人属性表[兵种]`（敌人属性 XML），战宠系统不自存，不动。
- 佣兵/任务面板的数据权威：另案（佣兵 merc-data.js 经核对是显示用、写操作已走 AS2 校验，风险低）。

## 6. 审阅修订（2026-05-31，提交 a2b94839d 后续）

迁移把"完成度"判定收归 AS2 后，审阅发现按"消耗金币>0 即一次性购买"的启发式会误伤**开关/反复型**方案。修订引入显式分类，停止启发式推断。

### 6.1 进阶类型显式建模（核心）

`战宠进阶函数` 各方案新增可选字段 `进阶类型`：

| 进阶类型 | 语义 | 示例 | 完成度 | 商店/进阶按钮 |
|---|---|---|---|---|
| （缺省） | 一次性购买 / 三件套累进 | 钙化、武器升级、基础训练→强化药剂→超级血清 | 钙化等按布尔标志；三件套按 `基础训练.次数 ≥ 次数上限(>0)` | 价门槛；完成后禁用 |
| `开关` | 纯开关，无前置购买；`消耗金币` 是运行时(每张图)扣费而非升级价 | 常驻淬毒、切换发型 | **永不完成** | 始终可点（`可切换`），不显示价、不做金币门槛 |
| `购买后开关` | 首次=一次性付费购买，购买后=免费启用/停用开关 | 影子刺客 | **永不完成**；另发 `purchased` 表示前置购买是否完成 | 未购买→价门槛购买；已购买→免费切换 |

实现：
- `PetPanelService.isSchemeRepeatable/isSchemePurchased`；`isSchemeCompleted` 对反复型早退 `false`，并把 `次数上限` 守卫收紧为 `> 0`（排除占位项 `凑数组的.次数上限:false`）。
- `snapshot.pets[].schemeStatus[方案]` 增 `repeatable`/`purchased`；`snapshot.schemes[方案]` 增 `type`。
- `pet-panel.js renderPromotions` 增 `freeToggle = repeatable && purchased` 分支：免价、免门槛、始终可点。
- 修复回归：常驻淬毒/影子刺客首次操作后被误判"已完成"而锁死、常驻淬毒按钮误显"500金"。

### 6.2 次数上限累进计数契约（审计结论）

审计 `战宠进阶函数` 全表：`次数上限>0` 仅 `基础训练(1)/强化药剂(2)/超级血清(3)` 三者，且均以 `基础训练.次数` 为权威累进计数——**当前假设成立**。已在 `isSchemeCompleted` 注明隐含契约：新增 `次数上限>0` 方案须沿用 `基础训练.次数` 同源计数，否则须改造判定。

### 6.3 其他边界修订

- **商店页签**：`requestSnapshot` 回调补 `_currentPage==='store'` 时 `renderStoreCategories()`，避免 snapshot 晚于进店时分类页签永久空白。
- **escapeHtml**：补转义单引号（纵深防御；当前动态值均进双引号属性/textContent 且源自可信 XML，无实际越权面）。
- **PetTaskTests.cs**：补传输层底线测试（cmd→action 映射、双层 callId 往返、断线/未知命令回包）。业务完成度逻辑在 AS2 侧，C# 测试不覆盖。
