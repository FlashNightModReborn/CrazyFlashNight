# 任务-成就 判定层共享（A 已施工）+ V8 判定下沉（B 草案，未开工）— 2026-06-11

> 背景：成就 A 轮（docs/成就系统-A轮-设计-2026-06-10.md）带来的条件表达力（curOf/check/进度），
> 任务系统未受益。本轮决策：**二者不强行同构**——任务保留自己的生命周期与奖励链，
> 仅在【判定条件 + 进度 UI】层与成就共享同一套基础设施。
> B（V8 下沉）经评估**只留草案不开工**（§8），动机与边界见该节。

## 1. 动机与边界

施工前事实：成就 `curOf` 已单方面读任务域状态（taskFinished/chainProgress/itemOwned/skillLevel），
与任务 `specialRequirements`（task/skill/infrastructure）+ `checkItemRequirements` 是**平行双实现**。
任务原生条件只有 关卡/交物/持有/特殊 四类，表达不了「击杀 N 敌人」「花费 N 金币」类目标，
且面板完成态是二值（无进度）。

**共享边界（本轮不变量）**：
- 共享：指标读数（rawOf）+ 谓词合取（checkConditions）+ 进度口径（{cur,target} 封顶）。
- 不共享：任务生命周期（接取/NPC 交付/链自动续接）、奖励发放（FinishTask 原子链）、
  成就锁存/认领（unl 棘轮/claimed）、各自的基线**策略**（见 §4）。

## 2. 共享层：ObjectiveEvaluator.rawOf

`org/flashNight/arki/achievement/ObjectiveEvaluator.as`（新）：
- `rawOf(type, params):Number` — 指标**原始读数**，无基线扣减、零副作用、全分支防御 undefined、
  未知类型返 0。原成就 curOf 的逐类型现算 1:1 迁入。
- 类型枚举（9）= `tools/lib/objective-types.js` OBJECTIVE_TYPES 单源（两个 derive 共用，禁副本）：
  `infraLevel / infraBuiltCount / killTotal / taskFinished / chainProgress / skillLevel / itemOwned / economyCount / itemCount(新)`
  - `itemCount`（新增）：`ItemUtil.getTotal(item)` 实数计数，供进度条；`itemOwned` 保持 0/1 布尔语义不变。
  - 成就域启用新类型需在 derive-achievement-catalog 的 params switch 显式加 case（default fail，刻意双重门）。
- 「无通用求值器」纪律延续：rawOf 仍是封闭枚举分发，不做 DSL/AST。
- `AchievementService.curOf` 改为：killTotal → `rawOf - a.base.kt`（D1 终身基线）；其余直通 rawOf。行为 1:1。

## 3. 任务 conditions 字段（opt-in，照 finish_remote 模式）

```jsonc
// data/task/*.json 任务条目可选字段；缺省 = 行为与历史完全一致（零成本直过）
"conditions": [
  { "type": "killTotal",    "params": {},                       "target": 50,
    "label": "击杀敌人",     "sinceAccept": true },
  { "type": "economyCount", "params": {"counter": "商城花费K点"}, "target": 10000,
    "label": "商城消费K点",  "sinceAccept": true },
  { "type": "itemCount",    "params": {"item": "明矾"},          "target": 20, "label": "持有明矾" }
]
```
- `label` 必填（面板进度行直显写手文案，避免 AS2 模板与数据漂移）。
- 与老字段（finish_requirements/物品/special_requirements）**合取**：`taskCompleteCheck` 在原三检后追加
  `TaskUtil.checkConditions`。
- derive-task-catalog.js build gate 校验：类型枚举 / target 整数≥1 / label 非空 / per-type 必填 params /
  economyCount counter 命中 AchievementMetrics.VALID（共享解析器 parseEconomyWhitelist）/
  taskFinished 存在性闭包+禁自引用 / **sinceAccept 仅限单调类型**（killTotal/economyCount——
  非单调指标减基线语义混乱，校验直接拒）。反例 9 类全红已验证。
- 复审补强（§9 修复 2，三轮收敛）：**布尔型 taskFinished/itemOwned 的 target 必须=1**（rawOf 返
  0/1，target>1 永不可达）；**itemOwned count≥1**（count=0 时 containTaskItems 恒真=错误达成）；
  **chainProgress 基础闭包**——须为有序号链（无序号链永不写 task_chains_progress）+ target≤链最大
  seq。**死锁校验 = 依赖图环检测**（局部规则两轮被击穿后收敛到标准做法）：所有 gating lower 成
  任务级依赖边——①有序号链按 seq 升序完成（链前驱边；实测 225 相邻转换 171 显式依赖前序）
  ②get_requirements ③条件边（taskFinished→引用任务；chainProgress(C,t)→C 链 seq≥t 的最小 seq
  锚点任务）——条件边的锚点经图可达回宿主 = 死锁红并打印环路径。统一覆盖：自链引用、seq 缺口、
  跨链循环、taskFinished 互锁、含链前驱/get_req 的长环。边界：只证「无结构性死锁」不证「可完成」
  （物品/经济/运行态归 playtest）；顺序模型偏保守（合法乱序设计会误杀→显式放宽点）；只审计
  条件边参与的环，存量 get_requirements 历史问题不在此门。
- conditions **不进 web task-catalog**（cur 是运行态读数，必须 AS2 现算回传；catalog 只有静态 target 无意义）。

## 4. 基线策略（per-consumer，不在共享层）

| 消费者 | 语义 | 基线 |
|---|---|---|
| 成就 killTotal | 终身（D1） | `_saveExt.成就.base.kt`（ensureInit 唯一拍点） |
| 任务 sinceAccept | **接取后窗口** | `AddTask` 时逐条 `rawOf` 快照进 `requirements.condBase[i]`（非 sinceAccept 条目记 0 占位对齐下标）；随 tasks_to_do 进存档透传 |

- 老档已接取任务无 condBase → conditionCur 跳过扣减 = 终身语义降级（宽松方向，不会卡死任务）。
- 弃任务重接 → AddTask 重新拍基线（窗口重置，符合直觉）。

## 5. UI 复用（检测-UI 层）

- `TaskPanelService.handleDetail` 回 `conditions: [{label,cur,target}]`（TaskUtil.conditionsProgress，
  与成就 progressOf 同口径：cur 封顶 target、永不 NaN）+ 权威 `satisfied`（taskCompleteCheck 实时算，
  §9 修复 1）。C# 桥通用透传零改动。
- web 详情在需求区后渲染 `.task-cond-row`（label + cur/target + 进度条，done 态提亮；黑白灰功能层）。
- **运行态字段不缓存固化**（§9 修复 1）：web 详情缓存只该固化静态字段；缓存命中时
  `maybeRefreshRuntime` 对含 conditions（或满足+非远程+不可前往）的任务后台复查一次，
  `patchCondRows`/`patchDeliverButton` 就地修补（satisfied 翻转时全量重渲徽章/列表角标）。
  无 conditions 且无导航复查需求的任务保持纯缓存命中，不打 AS2。

## 6. A2：扫描心跳联动

含 conditions 的任务，其达成态可能**无事件**翻转（如击杀到 50——没有物品/存档事件触发
`是否达成任务检测`）。借成就 `scanTick`（10s 循环）：`TaskUtil.anyActiveConditions()` 为真才调
`是否达成任务检测()` 刷新红点/td 信号。不注册第二个循环任务；无 conditions 任务时零额外成本。
**该联动位于 scanTick 的 `_dataReady`/`ensureInit` 门【前】**（§9 修复 3）：任务判定生命周期
不依赖成就目录可用性，成就目录加载失败不得连带停掉任务条件刷新。

## 8. B 草案：context→C#→TS/V8 判定下沉（**未开工，仅预留**）

动机：launcher 刻意保留 V8（ClearScript，`V8Runtime.cs`，已跑 hit-number bundle；TS 与 AS2 同 ES 血缘），
条目上量后把批量判定从 AS2 单线程解放到多核/现代引擎。

**评估结论（2026-06-11）**：当前检测负载非热点（scanTick 10s 级、任务检测低频事件驱动），
B 收益在「成就 90+ 条目 + 任务 conditions 普及 + 触发加密」之后。**先测量后施工**
（protocol-latency-baseline.md 范式；引以为戒：task_texts 体积估错一个数量级的教训）。

预留设计（落地时按此走，三刀切准）：
1. **按"权威 vs 通知"切，不按系统切**：NPC 交付门控 / FinishTask / taskFinish 服务端硬门控
   **永留 AS2 同步**（异步化=刷奖窗口+点 NPC 误判"路过"）。可下沉 = 扫描型/通知型输出
   （红点 td:、成就 toast、面板进度）。模式 = **乐观镜像 + 权威重验**：V8 回 advisory 信号驱动 UI，
   提交点 AS2 廉价重验。
2. **传增量不传快照**：AS2 侧 LiteJSON.stringify 完整 context 的成本可能超过检测本身。
   注入点已现成：`AchievementMetrics.record()`（经济计数唯一写点）顺发 `Δ|counter|delta`；
   killStats / FinishStage / 物品增减各有单点。V8 维护镜像态做全量谓词扫描，回
   `satisfied:{taskIds,achIds}` 信号。镜像在 换档/删档/loadAll 时必须 reset 重灌（SaveManager 钩子位）。
3. **单一声明式 SOT，两端解释器**：conditions/achievements json 即数据（§3 已就位），
   AS2（权威小解释器）与 TS（镜像全量扫描）解释同一份数据；金样本向量交叉测试
   （xUnit 直接喂 V8 跑向量；AS2 走 QA 向量），防双实现漂移。
4. 宿主注意：V8Runtime 现为单 bundle + 单 lock + FrameTask 帧驱动——判定扫描须低频批量，
   不得挤帧驱动通道（10s 级无碍）。

**开工门槛（全部满足才立项）**：① 实测 AS2 侧检测耗时在目标机出现帧预算占用证据；
② conditions/成就条目规模 ≥ 数十；③ A 的声明式层稳定一轮无 schema 变更。

## 9. 复审修复（2026-06-11，施工后同日）

复审发现三处 P2，已修：

1. **conditions 运行态进度被详情缓存永久固化**——web `_detailCache` 命中后只复查 finishNavigable，
   conditions 的 cur 与 satisfied 停在首次加载值（直至重开面板）。修复：`handleDetail` 增回权威
   `satisfied`；web 缓存命中对含 conditions 任务后台复查就地修补（见 §5；harness ui42=进度行
   就地刷新、ui43=satisfied 翻转→徽章/按钮/列表角标纠偏，ui11 改用纯缓存任务）。
2. **derive 允许永不可达/错误达成的条件**——taskFinished/itemOwned target>1（rawOf 返 0/1）、
   itemOwned count=0（containTaskItems 恒真）、chainProgress 引用不存在链/超界 target 均曾通过校验。
   修复：objective-types.js 收紧 + derive-task-catalog chainProgress 可达性闭包（见 §3 复审补强）。
   二轮复审再收紧：自链死锁初版用「存在其它任务 seq≥target」存在性检查，会把依赖前序的后续任务
   误判为可达（B#2 要求进度 3，C#3 存在但 C 依赖 B）→ 改保守规则 target<自身 seq。
   三轮复审彻底收敛：target<自身seq 仍漏 **seq 缺口**（A#1、A#5 链上 A#5 要求进度 4——4<5 通过但
   进度只能到 1）与**跨链循环**（A#1↔B#1 互要对方链进度）。教训定性：**死锁是依赖图的全局性质，
   逐条局部不变量必然留组合盲区**——废弃局部规则，lower 成任务级依赖图 + 环检测（业界标准做法，
   见 §3），三类死锁统一归约为「条件锚点可达回宿主」。验证时图检测还在真实数据中抓到测试假设
   之外的传递依赖（支线#1 经 get_requirements 链依赖主线#2，反向条件即真死锁），证明全局分析的
   必要性。
3. **任务条件心跳被成就 _dataReady 绑死**——scanTick 首行 `if (!_dataReady) return` 在 A2 联动之前，
   成就目录加载失败 → 任务 conditions 的 10s 刷新永久停摆。修复：联动块移到门前（见 §6）。

已知残留（接受）：成就 tab 领奖发放物品后切回「我的任务」tab 不重拉 snapshot，旧 satisfied 需
重选任务（触发复查）或重开面板才纠正——tab 切换重拉会重置选中态，性价比低，暂不做。
