## 1. 已完成改动（工作树现状）

### 阶段0
- **删除** `scripts/类定义/org/flashNight/arki/bullet/BulletComponent/Queue/HitContext.as`（前次拆分残留 orphan，全仓仅自引用）。
- **收窄注释**：`BulletQueueProcessor.as` C2 契约（约 L2617）+ `scripts/类定义/org/flashNight/arki/component/Damage/ShadowBookkeeping.md` L169，把"击中时触发函数仅发射新子弹"收窄回 C2 范围"不写 `霰弹值`"（原断言过度泛化，已证伪：实际加 buff/停弹/gotoAndPlay爆炸/让射手硬直，见 毒液蜘蛛侠.as:37/雾人:115/键盘镰刀.as:202）。

### 阶段1：`settleHit`
- 新增 `private static function settleHit(ctx, hitTarget, collisionResult, dmgMult, dodgeState):Object`（在 `settleRayHit` 之后）。职责：同步命中引用(幂等)→`ctx.Damage.calculateDamage`→`hitCount+=`→`publish("hit")`→kill/death(带 `_killed` 守卫)→`triggerDisplay`。**不**内算闪避/触发/附加层重置。
- 新增静态 `_hitCtx:Object`（主循环 per-bullet 不变量容器，对齐 `_rayHitCtx`）。
- 主循环 `processQueue` 命中段（原 L2354-2397）：保留 `附加层重置 + 命中对象 + 闪避 + 击中触发`（原位、原序），原 L2372-2397 的伤害→飘字替换为 `settleHit(_hitCtx, hitTarget, collisionResult, collisionResult.overlapRatio, dodgeState)`（**丢弃返回值** —— 主循环 `damageResult` 局部是 `DamageResult` 型，settleHit 返 `Object`，不接收避免类型不匹配）。
- per-bullet 填 `_hitCtx`（原 `isNormalKill` 预计算处）：bullet/shooter/Damage/flags/meleeMask。
- **死变量清理**：`processQueue` 删 `isNormalKill/damageResult/targetDispatcher/targetX/targetY` 5 个死声明（均已下沉进 settleHit）。

### 阶段2：`settleRayHit` 委托
- `settleRayHit` 体改为：写复用 `finalResult`(overlapCenter/overlapRatio/tEntry)→算 dodgeState→`settleHit(ctx, hitTarget, finalResult, dmgMult, dodgeState)`→FX。删掉原内联的 命中对象同步/calculateDamage/publish/kill-death/triggerDisplay（移入 settleHit）。
- **副作用 = 修 D1**：射线原 kill 块无 `_killed` 守卫，委托 settleHit 后获得守卫（防 UnitBullet 重复 kill/death + killStats 按 `_name` 膨胀）。10 个 settleRayHit 调用点**零改动**。

### 阶段3：`injectHit`（旁路原语）
- 新增 `public static function injectHit(bullet, shooter, target, hitX, hitY, dmgMult):Object`。不经几何，直接对指定目标结算一次命中→`settleHit`。
- 合法性兜底：`target==bullet || !(target.hp>0) || target.防止无限飞==true` → 返回 `null`。
- 用独立 `_injectCtx`（不碰主循环 `_hitCtx`，防装备层在主循环 settleHit 进行中调用时互相覆盖）。
- `FLAG_MELEE/FLAG_EXPLOSIVE` 经 `#include "../macros/FLAG_*.as"`（它们是宏 include，非类静态）；Dodge 经 `var Dodge:Object = DodgeHandler` 调用（绕 typed 返回类型不匹配）。
- 复用静态 `_rayHitResult` 作 collisionResult。
- **设计契约**：视觉无关（不重定向射线/不 spawn 子弹）；仅首发直结算；后续机制由 caller 自行下一帧调度；不进影子记账窗口（bullet.霰弹值 须正，MultiShotDamageHandle 直接模式）。

### 阶段3-builder：共享"可结算子弹"构造（2026-06-23）
**动因（关键架构发现）**：injectHit 的 `bullet` 形参虽号称"轻量 Object"，但 `calculateDamage` 强依赖 `bullet.damageManager`（伤害处理器链）+ 有效 `子弹威力/固伤/百分比/伤害类型/flags/霰弹值`。全仓扫 `getDamageManager`/`calculateDamage` 调用面，**唯一给 bullet 挂 damageManager 的地方就是 spawn 流程**（`子弹区域shoot传递`→`BulletFactory.createBullet`→`BulletLifecycle.bindLifecycle` L42）。故 injectHit 消费者必须先把一份属性模板初始化到"可结算"程度——这是 injectHit / 未来 C# 注入共同的前置原语，值得一次做对而非逐消费者手搓（手搓=复制 init 逻辑=静默数值漂移）。
- 新增 `BulletFactory.prepareBulletData(Obj, shooter):Void` —— 抽取自 `子弹区域shoot传递` 内联的 6 步 init（setDefaults / initializeBulletProperties / setTypeFlags / setFlagDependentDefaults / inheritShooterAttributes / calculateKnockback）。spawn 路径与可结算路径**共用**此函数。
- 新增 `BulletFactory.createSettlementBullet(Obj, shooter):Object` —— `prepareBulletData` + 忠实复刻 `createBulletInstance` 的"伤害相关尾部"：霰弹值正整数化（联弹 floor 正值 / 非联弹恒 1）→ `initializeNanoToxicfunction` → `additionalEffectDamage=0` → `damageManager = DamageManagerFactory.Basic.getDamageManager`。返回可直接传 injectHit 的 Obj。**不**含运动/碰撞器/生命周期/stateFlags 访问器（injectHit 旁路这些）。
- BulletFactory 加 import：`BulletComponent.Type.*`（BulletTypesetter）+ `component.Damage.*`（DamageManagerFactory）。
- `子弹区域shoot传递` 把 6 行内联 init 换成 `BulletFactory.prepareBulletData(Obj, shooter)`；spawn 的 finalize（含 nanoToxic）在 `createBulletInstance` 原位不动。
- ⚠ **淬毒双扣陷阱（已规避）**：`initializeNanoToxicfunction` 有 `shooter.淬毒 -=` 副作用，必须按"每发实际子弹一次"计。故它**不在** `prepareBulletData`（共享段），而是各 finalize（createBulletInstance / createSettlementBullet）各调一次。
- 正确性核对：`MultiShotDamageHandle` 只读 `霰弹值/最小霰弹值/flags/是否为敌人`（**不读 `联弹霰弹值`**），全部由 prepareBulletData + 霰弹值正整数化覆盖。

## 3. 待施工 + 最新决策（2026-06-23，射线贪心重写）

### 3.1 决策（已评估成立，方向已定，子项待定）
把射线从"**单帧批量收集全命中 + 排序 + 一次性 chain/fork**"改成"**每帧贪心找下一个命中 → injectHit 结算 → 挂触发函数下一帧续作**"的流式模型。射线管线简化成可插拔组件（`findNextHit 策略 + injectHit + 续帧`）。

**动因（用户域知识）**：
- 射线是新机制、消费者少、行为变更可接受；当前迁移难点多在射线原语翻译。
- 强化射线视觉打击反馈不足，本就想"每帧只判一个"用变相卡肉强化视觉。
- **当前真射线模型对矮碰撞箱敌人手感极差**；**要扩展"有宽度的射线模组"被当前批量管线卡死**——`findNextHit` 可插拔正是解法（线最近→带宽最近→锥最近，续作循环复用）。

**性能评估结论**：总吞吐与当前**持平**（结算 k 次两边相等、是大头；贪心省排序但每帧重扫，洗白）；**帧时间稳定性明显更优**（每帧封顶 1 命中，消除批量尖刺）——宽度射线×人堆正是尖刺最严重处，贪心赢在要害。额外白拿：早退（不为看不到的命中做功）、动态正确性（每帧重扫处理帧间移动/死亡）。
**细化建议**：别把"每帧 1 个"写死，做成 `每帧命中数 N` 参数（N=1 最强卡肉/矮箱；N=小常数防快射线发黏；N=∞ 退化回当前批量、作兼容对照）。

### 3.2 isLockon 纠偏（重要：设计稿 §6.4/§8 阶段3 旧描述已废）
isLockon **不是** bypass，是"`setRayFast` 重定向射线 + 把锁定目标注入几何模式管线"，4-5 触点：setup ray-redirect(L719)、combo-pierce 数组插入(L797)、combo-nearest 覆盖(L1086)、pure-pierce 数组插入(L1342)、single/chain/fork 覆盖(L1506)。锁定目标随后**作为几何命中流入完整管线**（chain/fork/pierce 从它继续）。故"坍缩成单一 injectHit"会破坏后续机制 + 视觉。**injectHit 真正的消费者是 G1111 hack + 锁定跨行首发**，不是 isLockon。

### 3.3 消费者迁移

## 4. 关键陷阱（load-bearing，施工必对照）
- **AABB 伤害乘数恒为 1**：`_aabbResult.overlapRatio` 从不被重写、沿用 AABBCollider 构造的 1；polygon=真实重叠比、ray=衰减。统一命中记录若给 AABB 填真实 overlapRatio → 伤害静默漂移（编译/订阅者都不报错）。三态必须复刻。
- **`命中对象` 赋值 load-bearing**：`击中时触发函数` 读 `this.命中对象`（毒液蜘蛛侠:38/雾人:116），故 BulletQueueProcessor L1486 + 主循环 L2359 的 `命中对象=hitTarget` **删不得**。
- **霰弹值影子窗口**：`__init/__commitDeferScatter` 留主循环调用点、不进 settleHit；命中管线不碰 `bullet.霰弹值`；`sc < -65535` 严格 `<`（NaN 陷阱）勿改 `<=`。
- **AS2 私有访问**：同包测试经 `var x:Object = 类名` 绕编译期 private 检查（AVM1 运行期不强制）。
- **FLAG_* 是 `#include` 宏**，非类静态；新方法用到须自行 include。
- **新建 .as 必须 UTF-8 BOM**：禁从零 Write（丢 BOM 被 #include 时编译器静默跳过/中文乱码）；正路=`cp` 现有 .as 再 Edit，或 Write 后 PowerShell 补 BOM 字节并校验。
- **D1 `_killed` 守卫**：kill/death 块统一带 `&& !hitTarget._killed`。
- **injectHit 的 bullet 必须经 `createSettlementBullet` 合成**：裸属性模板（如 G1111 的 chargedProp）无 `damageManager`/无效 `子弹威力`/无 `flags`，直喂 injectHit 会算错/报错。唯一正路 = `BulletFactory.createSettlementBullet(模板, shooter)`。
- **淬毒双扣**：`initializeNanoToxicfunction` 有 `shooter.淬毒 -=` 副作用 → 只能"每发实际子弹一次"。故它不在共享的 `prepareBulletData`，而在各 finalize（createBulletInstance / createSettlementBullet）各调一次；勿把它挪进 prepareBulletData，否则 spawn 路径双扣淬毒。
- **帧脚本（asLoader 单帧聚合单元）引用类**：用短名靠该帧通配 import 并集解析；跨包叶名碰撞才需 FQN。改 `tools/lint-frame-imports.js` 核对包是否在并集 + 是否碰撞，别盲加具体 import（具体 import 在聚合单元里会污染 sibling）。

## 6. 射线 deferral=B 持久化重写 —— 设计（2026-06-23，实现蓝图）

**目标**：`processRayBullets` 从"单帧批量结算全部命中 + 排序 + 一次性 chain/fork"改为"**持久射线：每帧贪心找下一个命中 → injectHit 结算 → 跨帧存活携带 state**"，并简化成 `findNextHit` 可插拔 + "命中→下一帧命中"的统一循环。用户域知识动因见 §3.1（矮碰撞箱手感差、强化射线打击反馈不足想变相卡肉、带宽射线模组被批量管线卡死）。

### 6.0 现状关键事实（Explore 实测，2026-06-23）
- 注册：`preCheckRay(bullet)` L380-384 → push 进静态 `_rayBullets`。
- 处理：`processRayBullets()` L670-1834，每帧对每个 ray **批量**结算全部命中（combo per-node L818 / per-phase L1124；非 combo pierce L1358 / single·chain·fork L1544）。
- 销毁：L1829 `bullet.removeMovieClip()` + L1833 `_rayBullets.length=0` —— **单帧销毁、零驻留**。
- `TeslaRayLifecycle`：不挂 onEnterFrame，`shouldDestroy()` 恒 true，`bindFrameHandler` 仅调 preCheckRay。
- reset()/SceneChanged：L291 `_rayBullets.length=0`。
- 射线 bullet 是 spawn 出来的 → **本就带 `damageManager`**（不需 createSettlementBullet）；`injectHit(ray, shooter, target, x, y, mult)` 直接可用 —— 这正是"射线天然适配 injectHit"（消费者侧 bullet 已存在）。
- settleRayHit 共 10 个调用点；5 处 lockon 注入（setRayFast L788 + combo L866 + combo non-pierce L1155 + pure-pierce L1411 + single/chain/fork L1575）。
- TeslaRayConfig：rayMode single/pierce/chain/fork + 组合；MASK_PIERCE=1/CHAIN=2/FORK=4；chainRadius 默认 200、damageFalloff 默认 1.0、rayLength 默认 900、pierceLimit=目标数。

### 6.1 生命周期改动（核心断点）
- 未结束的 ray **留在 `_rayBullets`**（不 removeMovieClip、不整体清空）；每帧推进至多 N 个命中。
- 该帧 budget 耗尽 / 连续 K 帧找不到命中 → 帧末 `removeMovieClip` + 从 `_rayBullets` 移除（存活重建：原地 swap-pop 或扫一遍重建存活数组，替代 `length=0`）。
- §5.5 队列语义：**暂停**（`_root.暂停`）→ ray 不推进、保留 state；**清空**（reset/SceneChanged）→ 遍历 `_rayBullets` 全部 removeMovieClip 再 `length=0`（防 MC 泄漏，比现状 `length=0` 更严）。

### 6.2 持久 state（挂在 ray bullet 上，随 `_rayBullets` 跨帧存活）
- `_rayInited:Boolean` — 首帧 setup（lockon 解析 / 方向 / budget / visited）完成否。
- `_rayBudget:Number` — 剩余命中名额（init = pierceLimit）。
- `_rayVisited:Object` — 已命中 `_name` 去重（跨帧）。
- `_rayCursorX/Y/Z:Number` — 上一命中点（chain 下一跳搜索中心；pierce 沿线推进基准）。
- `_rayLastTEntry:Number` — pierce 沿线已推进到的 tEntry（下一命中取更大者）。
- `_rayDryFrames:Number` — 连续未命中帧数（≥K 判链断/无目标 → 销毁）。
- `_rayDmgMult:Number` — 当前累计衰减乘数（每命中 ×= damageFalloff）。
- 方向/几何已在 `aabbCollider`（RayCollider，经 setRayFast）。

### 6.3 每帧算法（每个存活 ray）
```
if (!_rayInited) {
    解析 lockonTarget；若有 → setRayFast 重定向；init budget/visited/cursor/dmgMult；
    附加层伤害计算=0；hitTarget/命中对象=首目标；首发调一次 击中时触发函数()；
    若 lockon → 首发 = injectHit(ray, shooter, lockonTarget, ...)（首发直伤；约束允许时序变更）；
              visited[name]=true; budget--; cursor=lockon点;
    _rayInited = true;
}
if (_root.暂停) continue;                       // 暂停保留 state
hitsThisFrame = 0;
while (hitsThisFrame < N && _rayBudget > 0) {
    next = findNextHit(ray, cursor/方向, _rayVisited);   // 可插拔策略，见 6.4
    if (next == null) { _rayDryFrames++; break; }
    _rayDryFrames = 0;
    injectHit(ray, shooter, next.target, next.hitX, next.hitY, _rayDmgMult * lockonDmgMult);
    _rayVisited[next.target._name] = true;
    _rayBudget--; hitsThisFrame++;
    cursor = next 命中点; _rayLastTEntry = next.tEntry; _rayDmgMult *= damageFalloff;
}
if (_rayBudget <= 0 || _rayDryFrames >= K) markForRemoval;
// VFX：每命中即时 spawn 段（chain/fork 段逐帧出，正好配合卡肉）；射线骨干首帧出
```
- **首命中触发函数仅首发一次**（对齐原 L899/1167/1594）；附加层伤害计算=0 在首发前（陷阱：§4 命中对象 load-bearing）。
- injectHit 已带 `_killed` 守卫 + 合法性兜底 + AABB/ray dmgMult 语义；ray 用衰减乘数（非 1），与 §4 AABB 恒 1 不冲突（injectHit 的 cr.overlapRatio = 传入 dmgMult）。

### 6.4 findNextHit 可插拔策略（按 rayMode）
- **single**：首发即唯一命中（lockon 或最近 tEntry），budget=1，一帧完事（无 while）。
- **pierce（线）**：沿射线 AABB，找 `tEntry > _rayLastTEntry` 的最近未 visited 碰撞（用 cache 窗口裁剪 + bsearchScanStart，复用现 L1367 扫描骨架，但只取"下一个"不取全部）。
- **chain**：从 cursor 搜 chainRadius 内最近未 visited（复用现 L1619 chain 单跳搜索逻辑，抽成"找一个最近"）。
- **fork**：⚠ 多分支不契合"每帧一命中"。一期降级：fork → "从首命中点 chainRadius 内最近的逐帧各打一个"（视觉=多帧逐个分裂，天然卡肉）；或一期不做、combo/fork 暂留旧批量路。
- **带宽射线模组**（用户痛点解法）：findNextHit 的"线最近"替换为"带宽最近"（射线加宽度的碰撞），矮碰撞箱手感修复。这是组件化后**新增策略、不改主循环** —— 正是 deferral=B 解锁的能力。

### 6.5 N 参数
- `config.hitsPerFrame`，默认 **1**（最强卡肉/矮箱）；小常数防快射线发黏；∞ 退化回批量（兼容/对照）。

### 6.6 分期（rough → 组件化，§3.4「先粗糙再组件化、别固化第三套」）
- **一期（rough 能跑）✅ 已施工+compile 双绿（2026-06-23）**：**pierce + chain**（非 combo）走持久贪心路径；**single + fork + combo 暂留旧批量路**（最小化首版风险——single 一帧即结无需持久；combo/fork 多分支不契合每帧一命中）；lockon 跨行 = 首发 injectHit + 沿线/链下一帧。⏳ 待真机验。施工细节见 §6.8。
- **二期（组件化）**：`findNextHit` 抽成策略（函数指针/对象：line/band/cone），主循环统一；single/fork/combo 用多策略叠加重表达或废弃旧批量路（清理 §6.8 标记的失活批量分支）。
- **三期**：带宽射线模组（findNextHit=band）落地，验矮碰撞箱手感。

## 7. no-death 等价 Gate（RayPipelineEquivalenceTest，2026-06-24）

**动机**：手感跑偏 → 用户要先拿基线（batch）对比、确认 greedy selection 是否分叉，再开二期（防未侦测遗漏）。结论=**放弃 batch-then-defer、保留 greedy**：序列播放期间单位会死（射线边打边杀，自己序列就杀后续目标；且批量伤害数学 `B=hp/损伤值` 杀伤封顶假设了命中时 hp），deferred 必漂移 → batch-then-defer 一旦有死亡就退回每帧校验。greedy 每帧重扫天然对死亡正确。

**机制**（`scripts/类定义/.../Queue/RayPipelineEquivalenceTest.as`，注册进 TestLoader）：
- 加 `BulletQueueProcessor._rayPersistent`（public 开关，默认 true；false=全回退批量，生产回退 + 测试 A/B 同场景）。
- 加 `BulletQueueProcessor._raySettleProbe`（public，生产恒 null）：settleHit 顶部 `if(_raySettleProbe!=null){_raySettleProbe(hitTarget,dmgMult);return null;}` —— batch settleRayHit 与 greedy injectHit 唯一汇聚点，录 (target,dmgMult) 并短路（纯比 selection，不跑真伤/事件/几何）。
- harness：mock cache + scripted checkCollision（几何抽象成"目标在 tEntry 处碰撞 / 离束不碰撞"）+ 目标 hp 拉满不死；静态重赋值 stub `TargetCacheManager.acquireAllCache`/`RayVfxManager.spawn`/`EffectSystem.Effect`；bullet `stateFlags=STATE_FRIENDLY_FIRE` 跳 FactionManager、`伤害类型=真伤` 跳 DodgeHandler。
- **⚠ 陷阱（本轮踩中）**：`bullet.rayConfig` 必须是**真 `TeslaRayConfig` 实例** —— processRayBullets 用 `TeslaRayConfig(bullet.rayConfig)` 做类型转换，普通 mock Object 会被转成 **null** → config=null → rayMode 退化 "single" → 所有模式只命中最近一个。makeConfig 必须 `new TeslaRayConfig()` + 设 `rayModeMask`（isCombo/hasPierce 由它真实计算）。

**结论（实测）**：

| 模式 | batch | greedy | 判定 |
|---|---|---|---|
| pierce | T0:1,T1:.85,T2:.722 | 同 | ✅ SET+ORDER EQUAL |
| chain | T0:1,T3:.85,T1:.722,T4:.614,T2:.522 | 同 | ✅ EQUAL |
| fork | T0:1,T3:.85,T1:.85,T4:.85 | 同（两侧批量）| ✅ EQUAL |
| **combo** | T0,T3,T1,T4,T2,T5（逐节点序）| T0,T1,T2,T5,T3,T4（先抽 pierce 再泛洪）| ❌ **SET DIVERGE** |

combo 同一目标集 {T0..T5} 但**伤害分布不同**（falloff 按命中序累乘）：batch T1=.722/T3=.85，greedy T1=.85/T3=.522。= 铁枪手感跑偏精确根因。**pierce/chain/fork 已证 greedy 无需改**。

**二期 combo 修法（已定方向）**：greedy combo 从"frontier 全局泛洪"改成"**增量复刻 batch per-node 遍历**"——有状态迭代器（pierce 节点游标 + 当前节点的 chain/fork 子游标），每帧吐 batch 序列的下一个命中，死目标跳过续走。no-death 下由构造 ≡ batch（harness combo 转 EQUAL），有死亡时自动跳过（死亡正确）+ 卡肉。harness 即该修法的回归门。

### 7.1 二期 combo per-node 迭代器 —— 已实施 + harness 全绿（2026-06-24）
- **替换**：combo greedy 由 frontier 泛洪 → `findNextComboPerNode(ctx, bullet)` 状态机（删 frontier 相关 `_rayFrontier`/`findNearestToFrontier`）。loop 中 combo 用 `next.dmgMult`（迭代器按 phase 算）、且**不**走统一 `_rayDmgMult`/`_rayLastTEntry`/`_rayCursor` 推进（迭代器自管）。
- **状态机**（挂 bullet）：`_rayComboPhase`(node/chain/fork) / `_rayPnDmgMult`(节点累乘) / `_rayNodeTEntry` / `_rayChainCX·CY·CZ` + `_rayChainDmgMult` / `_rayForkBaseX·Y·Z` + `_rayForkDmgMult` / `_rayForkList` + `_rayForkIdx`。每次调用吐"批量 per-node 序列的下一个命中"：node→其 chain 走链→其 fork→下一 node。
- **dmgMult 对齐 batch**：node=pnDmgMult(P0)，node 后 pnDmgMult\*=falloff(P1)；chain 起于 P1 逐跳\*=falloff；fork 全用 P1。逐字复刻 batch per-node（L966-1123）。
- **新辅助**：`findNearestAt(ctx,visited,cx,cy,cz)`（findNearCursor 现为其 wrapper）+ `collectForkList(ctx,visited,cx,cy,cz,maxCount)`（复刻 batch fork 插入排序收集）。
- **结果（RayPipelineEquivalenceTest，2 场景 × 4 模式 = 8 组全 SET+ORDER EQUAL，compile test+publish 0 错）**：
  - cluster（紧密簇）：combo `T0,T3,T1,T4,T2,T5`(@1/.85/.722/.614/.522/.444) batch=greedy。
  - **sparse（稀疏，链断→fork→跨节点）**：combo `T0,T3,T4,T1`(@1/.85/.85/.85) batch=greedy —— 验证了 cluster 未覆盖的 fork + 跨节点路径。
- **lockon combo 已知小差异（非 harness 覆盖，待真机）**：lockon 首发走通用 init 直伤（满足"首发直伤"要求），随后 per-node 迭代器从 node phase 起（lockon 不作为 per-node 序列的节点）；与 batch 把 lockon 注入 comboAllHits 略有差异，但首发直伤 + 后续机制保留。
- ✅ **真机已验（2026-06-24）：手感对**（用户："行为感觉差不多"）。combo 逐帧扩散 + 伤害分布回到原版。

## 11. 带宽射线模组 v1（2026-06-24，§8.1 落地）：矮碰撞箱手感

§8.1 的原始痛点（细线射线常打不中矮碰撞箱敌人）落地。闭环 CS6 test+publish 双绿、未提交；真机手感待用户验。

### 11.1 几何根因（实测 RayCollider）
RayCollider 是**细线**（Kay-Kajiya slab：线 vs 目标 AABB）；2.5D Z 经 `getAABB(zOffset)` 把目标 AABB 的 top/bottom 在屏幕-Y 平移 zOffset。**矮目标 = 小 `(bottom-top)`** → Z-shift 后是细线难穿的窄窗 → 屏幕-Y 线-vs-窄AABB 的几何 miss（**非** zoff 门问题：放宽 zoff 只放进更多深度候选、细线照样错过，还可能命中邻 lane）。故对的轴是给射线加**屏幕-Y 厚度**。

### 11.2 实现（**子类隔离**：严格细线零带宽开销 —— 用户隔离诉求落地）
带宽**不进** RayCollider，独立子类承载（先做"退化进 RayCollider"v1 跑通，再隔离，同"先粗糙再组件化"）：
- **新建 [`BandRayCollider`](../scripts/类定义/org/flashNight/arki/bullet/BulletComponent/Collider/BandRayCollider.as) extends RayCollider**：`_halfWidth` + setHalfWidth/getHalfWidth；**override checkCollision = 零复制委托**——把目标 AABB（已 Z-shift）的 top/bottom 各膨胀 ±_halfWidth 装入**复用静态代理 AABBCollider**，再 `super.checkCollision(proxy, 0)`（proxy.getAABB(0) 不二次平移）。**零 slab 复刻、零父类私有访问**（super 用自己的 _ray/_isValid；`this` 仍是 BandRayCollider 故几何正确）；先读目标 4 边进局部再写代理，规避"目标也是 AABBCollider 时与代理共享同一静态 AABB"的别名覆盖。
- **RayCollider / RayColliderFactory / RayColliderTest 回归 pristine**（`git diff` 空 = 与本特性出现前逐字节相同）：严格细线路径**零带宽开销、零回归**，且**删掉了 v1 的池重置补丁**（细线池纯净，无泄漏）。
- **TeslaRayConfig**：`rayWidthFactor`（默认 0，FIELD_MAP 一行）——保留。
- **TeslaRayLifecycle.bindCollider**：`rayWidthFactor>0` → `new BandRayCollider`（**非池化**；稀有，getFactory()=null → bindSafeRemove 的 releaseCollider 静默 no-op，随 Object GC，不污染细线共享池）+ setHalfWidth(zRange*系数*0.5)；否则走 RayColliderFactory 共享池细线。多态 `areaAABB.checkCollision(...)` 自然分派 —— **细线永不触碰带宽逻辑**。

### 11.5 关键修正（2026-06-24 实机反馈）：Z = 屏幕 _y → 带宽必须并入深度门
**实机现象**：镇暴射线装霰弹枪，rayWidthFactor 0.8→1.5 **仍打不中低矮血鼠**（角色套装强制显示碰撞箱，确认血鼠 AABB 确实矮）。
**根因（代码实测）**：`Z轴坐标 = this._y`（[敌人模板:691](../scripts/逻辑/单位函数/单位函数_lsy_敌人模板迁移.as#L691) / [玩家模板:2108](../scripts/逻辑/单位函数/单位函数_fs_aka_玩家模板迁移.as#L2108)）——**深度就是屏幕 _y**。射线命中 = 两道串联：① 深度门 `|zoff| ≥ 纵向攻击范围(42) → continue`（**先**，连 checkCollision 都不调）；② 屏幕-Y 线相交（**后**，带宽在此）。血鼠屏幕上低 ~150px → `_y`(=Z) 差 ~150 ≫ 42 → **深度门先挡掉**，带宽代码**从未执行** → 拉系数无效。⚠ 用户域知识补充：深度按脚底定，且**敌人 AI 自动保持 Z 一致**（[CombatModule](../scripts/类定义/org/flashNight/arki/unit/UnitAI/behavior/CombatModule.as) 的 上行/下行 = diff_z 收敛）——故稳态本是同深度（带宽该管），血鼠系 AI 未对齐瞬间/被截图。
**修正**：既然 Z=_y，带宽厚度与深度是【同一根轴】→ 把半宽并入深度门（[processRayBullets](../scripts/类定义/org/flashNight/arki/bullet/BulletComponent/Queue/BulletQueueProcessor.as)：`bulletZRange += areaAABB.getHalfWidth()`，无类型探测 BandRayCollider；非带宽射线无此方法 → 门不延伸 → 零影响）。**效果**：rayWidthFactor 现在真正控制光束的【深度+高度统一 reach】—— 有效门 = `纵向攻击范围 × (1 + 系数/2)`（系数 1.5 → ×1.75 = 73.5px；够到 150px 外目标需系数 ≈ 5 = 高大扫）。粗光束 = 宽扫，正是"宽射线扫矮敌"的本意。
**验证**：test —— RayEq 18 不变（mock 射线无 getHalfWidth → 门不延伸 → 非带宽路径零影响）+ RayColliderTest 61/61 + BandRayColliderTest 15/15（38s）；publish 0 错（876270B）。⏳ **真机调系数**：AI 拉敌人靠近后入门即中；远目标按需提系数（系数越高=光束越高大、扫越宽一片深度+高度，对 镇暴 riot sweep 是合理手感）。
