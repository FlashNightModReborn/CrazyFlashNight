## §1. gotoAndStop 后续代码销毁（最高优先级）

### 行为定义

AS2 中 `mc.gotoAndStop(label)` / `gotoAndPlay(label)` 会**立即卸载当前帧上的所有 MovieClip 元件**。如果调用链当前正处于被卸载 MC 的 `onEnterFrame` / `onClipEvent(load)` 体内，**跳转后这一帧脚本的后续代码不会执行** —— 执行上下文随 MC 销毁。

见 [agentsDoc/as2-anti-hallucination.md](../agentsDoc/as2-anti-hallucination.md) §1：
> `gotoAndStop`/`gotoAndPlay` 会立即卸载当前帧的 MovieClip，调用链处于被卸载 MC 的 `onEnterFrame` 中时，帧跳转后的代码不会执行（执行上下文被销毁）。

### Routing 系统中的依赖

`__stateTransitionJob` 机制就是为绕过这个黑箱设计的。Producer 写 job，consumer 在 `状态改变` 后跳的新帧（已是新执行上下文）调 `executeStateTransitionJob` 取出 callback。

代码现状（**已端到端测试通过**，但仅在 fake unit 上）：
- producer-set：[RoutingIntent.createStateTransitionJob](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingIntent.as):67
- consumer apply：[StateTransition.apply](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/StateTransition.as):21 第 8 步
- callback 跳起执行：[RoutingIntent.executeStateTransitionJob](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingIntent.as):98

### Mock 接口建议

**核心问题**：在 testloader 里没有真的 mc 时间轴帧。fake unit 是 plain object，gotoAndStop 是 spy 方法，本身不会真销毁上下文。

**夹具策略：分两层契约（[DECIDED]）**

> 不做"全局 poison self"。区分两类上下文：
> (a) **旧帧上下文**（被 gotoAndStop 卸载的 man/container/frame clip + 旧帧脚本闭包捕获的引用）—— poison
> (b) **新帧/状态机 executor 上下文**（StateTransition.apply 内 gotoAndStop 后续的 self method 调用）—— 放过
>
> 强 mock 验证的是"旧帧脚本后续代码不能被依赖"，**而不是**否定 StateTransition.apply 作为状态机 executor 的 post-goto 编排。

**实现机制：frameEpoch / contextToken**

```as
class MockMovieClip {
    public var __frameEpoch:Number;  // 每次 gotoAndStop ++
    public function gotoAndStop(label):Void {
        this.__lastLabel = label;
        this.__frameEpoch++;
        // 不动 self 上的 method 引用
    }
}

// 业务代码若错误地从旧帧闭包持有引用：
var oldMan = self.man;
var oldEpoch = oldMan.__frameEpoch;  // 显式 snapshot token
self.gotoAndStop("X");                 // __frameEpoch++，token 失效
// 现在断言 helper：
MockAssert.requireCurrentEpoch(oldMan, oldEpoch);  // → trace warning + 测试 FAIL
```

- routing class 内的 `self.读取当前飞行状态()` / `executeStateTransitionJob(self)` 走 *当前* self method，**不持 token**，因此不会被 poison —— `apply` 第 8 步合规序保持工作
- 业务代码（路由 .as / 玩家模板迁移 .as / 装备函数）若需要持有跨 gotoAndStop 的引用，**必须显式调 token 检查**才会触发 poison —— 反过来强制业务代码做正确的 epoch 守卫

### 断言策略

- ✅ producer-set → state-change → gotoAndStop fires → callback executes 顺序（弱契约即可，已有）
- ✅ gotoAndStop 之后 `self.<method>` 调用合法（不 poison）—— apply 第 8 步 `gotoAndStop → 读取当前飞行状态 → executeStateTransitionJob` 通过
- 🟡 显式 epoch token：跨 gotoAndStop 持有的 mc 引用 + epoch 比对 → 旧 token 失效（新增）
- 🟡 callback 内闭包捕获 self —— 反向断言：若 callback 闭包捕获跳转前的 self ref + 显式 epoch token，应 FAIL（防御 routing 业务调用方误用）

### 已知陷阱

- [[feedback-as2-detached-mc-signature]] —— removeMovieClip 后引用持有的 mc 退化签名（`toString()=""`, `_parent=undefined`, 子级 undefined）。MockMovieClip 应模拟该退化（被 remove 的 mc 而非 gotoAndStop 后的 self）。
- 同帧二次 gotoAndStop（mark/is 同帧跳转保护）—— [RoutingIntent.markWeaponSameFrameJump](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingIntent.as):35 已纯函数化，但本黑箱契约要求 mock 也支持"同一帧内连续 gotoAndStop 应只生效一次的语义验证"

### 后续验证（独立 task）

frameEpoch 模型是契约假设；真实 Flash 行为待用最小影片剪辑 + JPEXS 反汇编验证：
- gotoAndStop 之后**在同一同步代码块继续调 `self.X`** 实际调到老上下文还是新帧的同名 method？
- 跨 gotoAndStop 持有的子级 mc 引用何时变 detached（gotoAndStop 之后立即 vs onEnterFrame 之后）？

若验证结果与本契约偏离，更新契约不影响夹具实现框架（frameEpoch 机制本身仍成立）。

## §2. attachMovie + initObject 同步 enumerate + copy

### 行为定义

`parent.attachMovie(linkage, name, depth, initObject)` 在 AS2 中：
- **同步** enumerate `initObject` 的 own properties，逐项 copy 到新创建的 MovieClip 上
- 调用返回后 `initObject` 立即可复用（不持有引用）
- 如果 `linkage` 符号不存在，返回 `undefined`（不抛异常）
- `name` 重复会覆盖同名 child（同深度位置）

见 [ContainerInitScratch.as](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/ContainerInitScratch.as):17：
> AS2 attachMovie(linkage, name, depth, initObject) 同步 enumerate initObject 的 own 属性 copy 到新 MovieClip，调用返回后 initObject 立即可复用。

### Routing 系统中的依赖

**5 个 attachMovie 调用点**（详见 [路由系统-旧API使用清单-2026-05-19.md](路由系统-旧API使用清单-2026-05-19.md) §0）：
1. 兵器攻击路由.as:150（KIND_WEAPON）
2. 空手攻击路由.as:52, 367（KIND_UNARMED ×2）
3. 技能路由.as:75（KIND_SKILL）
4. 战技路由.as:90（KIND_BATTLE_SKILL）

**initObject 构造**：
- [ContainerInitScratch.assembleFromMap](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/ContainerInitScratch.as):36 — self-replacing scratch + RoutingFieldMap 装配
- [RoutingLifecycle.buildPublicContainerInit](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingLifecycle.as)（待 grep 确认入口）

**Linkage 拼接**：[ContainerSpec.buildLinkageName](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/ContainerSpec.as):60

### Mock 接口建议

```as
class MockMovieClip {
    public var __children:Object;  // name → MockMovieClip
    public var __initObjectsReceived:Array;  // 每次 attachMovie 的 initObject 快照
    public var __missingSymbols:Object;     // 注入 missing symbol 列表

    public function attachMovie(linkage:String, name:String, depth:Number, init:Object):MockMovieClip {
        if (this.__missingSymbols[linkage]) return undefined;  // 模拟 missing
        var child = new MockMovieClip();
        for (var k in init) child[k] = init[k];  // 同步 copy
        this.__children[name] = child;
        this.__initObjectsReceived.push({linkage: linkage, init: snapshot(init)});
        return child;
    }
}
```

**[DECIDED] 引入 RoutingRuntime.attachMovie + 高层 ContainerAttachAction，且提前到二阶段后期**

性能上 attachMovie 本身就是低频重操作，多一层 static call 相比黑箱开销可忽略。收益明确：5 个调用点统一可 mock / missing symbol 可注入 / initObject copy 可端到端断言 / 第三阶段 MockMovieClip 不需要侵入业务路由。

**两层 adapter 设计**

低层（RoutingRuntime 扩 frozen surface）：
```as
RoutingRuntime.attachMovie(parent:MovieClip, linkage:String, name:String, depth:Number, initObj:Object):MovieClip
RoutingRuntime.setAttachMovieAdapterForTest(adapter:Object):Void
RoutingRuntime.clearAttachMovieAdapterForTest():Void
```
生产路径走 `parent.attachMovie(...)`；测试路径走注入 adapter。

高层（容器化专用，新类）：
```as
ContainerAttachAction.attach(unit:MovieClip, kind:String, actionName:String, initObj:Object):Object
// → 返回 { man, fallback: ABORT|SILENT_CONTINUE }
// 内部：ContainerSpec.buildLinkageName(kind, actionName) → RoutingRuntime.attachMovie(...) → 看 man undefined? → ContainerSpec.getMissingFallback(kind)
```
调用方拿 result 自决 `if (result.fallback === ABORT) return ...;`（参考 #2 task 之前讨论）。

**5 个生产调用点切换**（属二阶段后期工作，非第三阶段）：
- 兵器攻击路由.as:150 / 空手攻击路由.as:52, 367 / 技能路由.as:75 / 战技路由.as:90

### 断言策略

- ✅ attachMovie 收到正确 linkage（ContainerSpec.buildLinkageName 输出对照）
- ✅ initObject 在 attach 之后立即被复用（ContainerInitScratch 已断言 "两次调用返回同一 scratch + transform 刷新"）
- 🟡 missing symbol 注入 → ContainerSpec.getMissingFallback 策略真正生效（ABORT vs SILENT_CONTINUE），调用方按 result.fallback 行为对
- 🟡 initObject 字段完整性 → 与 RoutingFieldMap.X_FIELDS 装配契约对照（端到端路径，补充已有的纯函数 sentinel 断言）

### 已知陷阱

- ContainerInitScratch self-replacing trampoline：首次调用 vs 后续调用 fast-path closure 路径不同；mock 测试通常在 boot 后跑，已经走 closure 路径 —— 但夹具应能强制 reset 测装配契约本身（已通过 `assembleFromMap` 公开 API 解决）。
- 跨容器跳转（空手 3 处 XML）：在 onClipEvent(load) 内调 `跨容器标签跳转` —— 当前生产路径走 `__stateTransitionJob.arg_containerName + arg_targetLabel` 传参，夹具应能模拟"attachMovie 之后子级 MC 的 onClipEvent(load) 立即执行"语义。

### 已确认风险点（2026-05-19 closed）

**SILENT_CONTINUE 路径下 undefined man 会污染 unit — 已在生产入口短路**

技能 / 战技的 missing symbol fallback 是 SILENT_CONTINUE。Pre.3 切换前的旧代码（`unit.attachMovie(...)` 直接调）和 Pre.3 切换后的中间版（`var man = ContainerAttachAction.attach(...).man` 不检查 status）都会把 undefined `man` 传给下游：

- `RoutingLifecycle.handleFloat(man, unit, "技能浮空")`
- `RoutingLifecycle.bindEndCleanup(man, unit, ...)`

III.2 端到端夹具揭示的关键观察（见 `RoutingEndToEndTest.testBlackbox_HandleFloat_UndefinedMan_TempYPositive_PollutesUnit`）：

| 调用 | 真实 AS2 行为 |
|---|---|
| `undefined.落地 = true/false` | silent no-op |
| `undefined.onUnload = function() { ... }` | silent no-op（closure 也被丢弃） |
| `unit[floatFlag] = true` | **写入生效（unit 是真对象）** |
| `unit._y = unit.temp_y` | **写入生效** |
| `unit.起始Y = unit.Z轴坐标` | **写入生效** |
| `unit.浮空 = true` | **写入生效** |

也就是说："SILENT_CONTINUE 不只忽略 man，handleFloat 内对 unit 的副作用照样发生" — 这是被 AS2 silent no-op 掩盖的半副作用污染。

**修复（2026-05-19，user 决策）**：在生产调用点显式短路，不让 missing 容器走到 handleFloat / bindEndCleanup。

短路放在业务调用点而非 ContainerAttachAction 本体（ContainerAttachAction 继续只做"拼 linkage / attach / 返回 status"，不做策略决议）：

```as
var attachResult:Object = ContainerAttachAction.attach(unit, ContainerSpec.KIND_SKILL, 技能名, initObj);
if (attachResult.status !== ContainerAttachAction.STATUS_OK) {
    return;   // missing 容器：不报错、不 fallback、不执行依赖 man 的 lifecycle
}
var man:MovieClip = attachResult.man;
RoutingLifecycle.handleFloat(man, unit, "技能浮空");
RoutingLifecycle.bindEndCleanup(man, unit, "战技", "技能结束", "技能浮空");
```

落地位置：
- [scripts/引擎/引擎_fs_技能路由.as:72](../scripts/引擎/引擎_fs_技能路由.as) 载入后跳转技能容器
- [scripts/引擎/引擎_fs_战技路由.as:87](../scripts/引擎/引擎_fs_战技路由.as) 载入后跳转战技容器

语义更新：
- `MISSING_ABORT` — 业务路径必须返回（兵器 / 空手，旧行为）
- `MISSING_SILENT_CONTINUE` — **不报错、不 fallback、但不执行依赖 man 的 lifecycle**（语义从"继续跑下游"收紧为"静默跳过容器后续"；status 名暂不改）

测试锁定（`RoutingEndToEndTest`）：
- 黑箱观察：`testBlackbox_HandleFloat_UndefinedMan_TempYPositive_PollutesUnit` 记录裸调 handleFloat 会污染 unit 的事实（防止未来在别处误用）
- 生产路径锁修复：`testProduction_MissingSkill_ShortCircuits_NoUnitPollution` / `testProduction_MissingBattleSkill_ShortCircuits_NoUnitPollution` 复现路由入口算法，断言 missing 容器 + tempy>0 时 unit.浮空 / unit.技能浮空 / unit._y 全保持默认
- 正向回归：`testProduction_HappySkill_StillRunsLifecycle` — happy 路径仍完整跑 handleFloat + bindEndCleanup

## §3. onUnload chain（多层叠加 / 父级 unload 触发）

### 行为定义

AS2 MovieClip 的 `onUnload` handler 在以下时机触发：
- `mc.removeMovieClip()` 时
- `parent.gotoAndStop()` 把当前帧的 mc 卸载时
- `parent.removeMovieClip()` 级联触发

多次写 `mc.onUnload = fn` 会**覆盖**（不会叠加 / 不会 chain）。Routing 系统手工实现 chain：
```as
var prev = mc.onUnload;
mc.onUnload = function() {
    if (prev != undefined) prev.apply(this);
    // 新逻辑
};
```

### Routing 系统中的依赖

- [RoutingIntent.bindContainerEndState](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingIntent.as):51 — 显式 chain 实现，prev.apply(this) 后写 `UpdateBigSmallState`
- [RoutingIntent.suppressOldManUnload](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingIntent.as):29 — 设为空函数（覆盖）
- [RoutingLifecycle.bindEndCleanup](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingLifecycle.as) — 包含浮空清理逻辑

### Mock 接口建议

```as
MockMovieClip.removeMovieClip = function() {
    if (typeof this.onUnload === "function") {
        this.onUnload.apply(this);
    }
    // 级联 children
    for (var k in this.__children) {
        this.__children[k].removeMovieClip();
    }
    this.__removed = true;
    this.__signature = "detached";  // 后续访问退化
};
```

**断言 helper**：
- `assertOnUnloadChainOrder(mc, [stepA, stepB, ...])` —— 触发 unload，断言 chain 中各 prev 按 LIFO 顺序触发

### 断言策略 [DECIDED]

**必须测**：

- ✅ 原始 A → bindContainerEndState → unload，顺序：A → containerEndState
- ✅ 原始 A → bindContainerEndState → bindEndCleanup → unload，顺序：A → containerEndState → endCleanup（后绑定 wrapper 调 prev，再执行自己）
- ✅ 父级 unload / remove 级联触发子级 onUnload
- ✅ suppressOldManUnload 后旧 prev 不再触发

**out of scope（不测，标 undefined）**：

- prev 内部自修改 `this.onUnload = newer` —— 生产代码当前不依赖此自修改 chain 语义。若未来出现需求再讨论。

### 已知陷阱

- onUnload 中 `this` 指向被卸载的 mc 本身（已不在 parent 子级列表）—— MockMovieClip 应保留 `this` 但 detach 父子关系
- onUnload 内调用 `this.removeMovieClip()` 是 no-op（已在 remove 中）—— 幂等
- gotoAndStop 之后 onUnload 与 callback 的执行顺序：onUnload 先 vs gotoAndStop 后续代码先？关联 §1 frameEpoch 模型 —— 夹具中应 documented + 加 ordering 断言

## §4. removeMovieClip 幂等 / 时序

### 行为定义

- `mc.removeMovieClip()` 之后再调一次：no-op（不抛错）
- removeMovieClip 之后 mc 引用持有时：[[feedback-as2-detached-mc-signature]] 描述退化（toString=""、_parent=undefined 等）
- removeMovieClip 是 **同步** 操作，调用返回时 mc 已卸载，onUnload 已触发

### Routing 系统中的依赖

- [JumpDeriveAction.execute](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/JumpDeriveAction.as):28 — 派生触发后 `man.removeMovieClip()`
- [StateTransition.apply](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/StateTransition.as):44 — `if (plan.removeDynamicMan) self.man.removeMovieClip();`
- [兵器攻击路由.as / 空手攻击路由.as 的 attachMovie 失败 fallback](路由系统-旧API使用清单-2026-05-19.md) — 见 §3 missing fallback

### Mock 接口建议

已在 §3 提到 `MockMovieClip.removeMovieClip`，再加：
- 已 remove 的 mc 再调 removeMovieClip → 仅 trace warning，不抛
- 已 remove 的 mc 上访问字段 → 返回 undefined（detached 签名）

### 断言策略

- ✅ removeMovieClip 触发 onUnload 一次（已有覆盖）
- ✅ 双调 removeMovieClip 幂等（新增）
- 🟡 detached signature 验证：访问 _parent / 子级返回 undefined（新增）
- 🟡 调用顺序：JumpDeriveAction 内 `状态改变(targetState)` 在 `man.removeMovieClip()` 之前 —— 这个顺序生产代码已写死，夹具需断言（防止未来重构漂移）

### 已知陷阱

- [[feedback-as2-detached-mc-signature]] —— `if (!ref || !ref._parent) return;` 是订阅者 idiom
- removeMovieClip 在 timeline-attached（非 attachMovie 创建）mc 上**不生效** —— 生产代码 ContainerInitScratch 装的 mc 都是 attachMovie 出来的（`__isDynamicMan: true` flag 区分），fake 应模拟此区分

## §5. Key 输入快照

### 行为定义

`Key.isDown(keyCode)` 是 host native 调用，**每次调用都是当前帧实时键盘状态**。AS2 不提供"上一帧按下状态"原生 API（需要业务层缓存）。

### Routing 系统中的依赖

3 处 派生跳跃判定（已通过 [JumpDeriveAction.tryDerive](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/JumpDeriveAction.as) 统一）：
- 兵器上挑：`Key.isDown(unit.B键)`
- 空手升龙拳 ×2：`Key.isDown(u.A键) && Key.isDown(u.B键)`

调用者在调 `tryDerive` 之前**自己读 Key.isDown**，结果作为 `keyComboPressed:Boolean` 形参传入。JumpDerivePredicate / JumpDerivePlan 完全不调 Key —— **快照已发生在调用方**。

### Mock 接口建议

**零额外 mock 需要**。fake 测试只需直接传 `true` / `false` 作为 `keyComboPressed`，覆盖真值表。已实现：
- [JumpDerivePredicateTest](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/JumpDerivePredicateTest.as) 11 断言
- [JumpDerivePlanTest](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/JumpDerivePlanTest.as) 10 断言

### 断言策略

- ✅ keyComboPressed=true + 其他条件满足 → triggered=true
- ✅ keyComboPressed=false → triggered=false（已覆盖）
- 🟡 调用方在 routes/* 内的 Key.isDown 调用本身（不在 Routing class 中）—— 夹具不覆盖，由人工 review

### 已知陷阱

- AS2 `Key.isDown` 性能 417ns/次（[agentsDoc/as2-performance.md](../agentsDoc/as2-performance.md) S06）—— 已被 routing 收敛到调用站点一次读取传入

## §9. 浮空标记协议（routing ↔ 空中控制器）

> routing 内部的数据协议，**不是** AS2 原生黑箱。改 `RoutingLifecycle.completeAnimation` /
> `bindEndCleanup` / `handleFloat`、改空中控制器的标记清除、或新增 floatFlag 取值前，
> 必读本节。源码依据：[RoutingLifecycle.as](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Routing/RoutingLifecycle.as) +
> [单位函数_fs_空中控制器.as](../scripts/逻辑/单位函数/单位函数_fs_空中控制器.as)。

### 9.1 三类"浮空标记"——别混

| 标记 | 类型 | 谁写 | 含义 |
|---|---|---|---|
| `unit.浮空` | Boolean | 空中控制器 `_tick` 每帧 | 通用"在空中"（`_y < z - tol`）。瞬时量，**任何系统都不应长期持有它当状态** |
| `unit.技能浮空` | Boolean | handleFloat / completeAnimation / 动画完毕 / 空中控制器 | skillFloat 源（空中放技能/战技维持重力）的"外部可见标记" |
| `unit.飞行浮空` | Boolean | 升空函数 / 喷气背包 / 空中控制器 | jetpack（喷气背包）源的标记，**独立子系统** |
| `__技能浮空任务ID` / `__跳跃浮空任务ID` / `__自然落地任务ID` | task id | 玩家模板各浮空函数 | 调度任务句柄；清理时须 `removeTask` + 置 null |

空中控制器是 **source 模型**：`unit.__air.sources` = `{ skillFloat / jumpFloat / naturalFall / fastFall / knockback / jetpack }`。`技能浮空` / `飞行浮空` 是这些 source 在 unit 上的**外部可见布尔标记**，不是 source 本身 —— source 增删走 `设置源` / `清除源`。

### 9.2 `__preserveFloatFlagOnUnload` 语义

**用途**：解决"技能在空中结束 + 二段跳"。技能容器卸载时 `技能浮空` 标记必须**存活**到下一个跳跃状态，由 `启动跳跃浮空` 当二段跳消费；否则会被 onUnload 清掉。

- **写**：`RoutingLifecycle.completeAnimation(man, unit, enableDoubleJump)` 当 `shouldPreserveDoubleJump`（即 `enableDoubleJump==true && inAir`）→ `unit.技能浮空 = true` + `unit.__preserveFloatFlagOnUnload = "技能浮空"`（**字面量硬编码**）。
- **读**：`RoutingLifecycle.bindEndCleanup(..., floatFlag)` 的 onUnload，`needReset` 时：
  - `__preserveFloatFlagOnUnload == floatFlag` → `delete` 该 marker，**不清** `unit[floatFlag]` → 标记存活。
  - 否则 → `unit[floatFlag] = false` → 清标记。
- **也被读**：动画完毕(998) `wantsDoubleJump = (__preserveFloatFlagOnUnload == "技能浮空")`；B3 持枪族分支消费后 `delete`。
- **不变式**：marker 是**一次性**的 —— 被 bindEndCleanup / 动画完毕 读到即 delete。残留未 delete 的 marker = bug 信号（preserve 意图没被正确消费）。

### 9.3 合法 floatFlag 取值

当前生产路径 `handleFloat` / `bindEndCleanup` / `空中控制器.启用技能浮空` 的 `floatFlag` 实参**只用 `"技能浮空"`**（兵器 / 空手 / 技能 / 战技 路由 + 路由基础 facade 全部传它）；`completeAnimation` 写的 preserve marker 也硬编码 `"技能浮空"`。

→ **当前唯一合法 floatFlag = `"技能浮空"`**。新增取值前，必须同步确认 `completeAnimation` 的 marker 字面量与之一致（见 9.4）。

### 9.4 不可混用规则（关键）

1. **`completeAnimation` 写的 marker 值 必须 == `bindEndCleanup` 收的 `floatFlag` 实参。** 两者目前都是 `"技能浮空"`，一致。
2. 若分叉（如某路由给 `bindEndCleanup` 传 `"飞行浮空"`，而 `completeAnimation` 仍写 `"技能浮空"`）：onUnload 的 else 分支 `unit["飞行浮空"] = false` —— **清错标记**；同时 `__preserveFloatFlagOnUnload` 残留不删 → `技能浮空` 被错误保留（卡浮空）、`飞行浮空` 被错误清零。
3. `技能浮空`（skillFloat 源）与 `飞行浮空`（jetpack 源）是**两个独立子系统**，preserve marker 永远不该命名对方的标记。
4. **回归哨兵**：`PlayerTemplateUnitFixtureTest.testEndCleanup_CompleteAnimationPreserve_MismatchedFlagClearsWrong` 把"混用 → 清错"固化成可观察事实 —— 它是 **negative contract / 风险 tripwire**，**不是目标行为**。任何让该测试行为改变的改动，都要回本节重新评估。

### 9.5 `技能浮空` 标记的三条清除路径

谁负责把 `技能浮空` 清回 `false`：

- (a) **空中控制器 `_tick` 落地收尾**：skillFloat 源在 `_y >= z - tol` 时 `unit[sf.floatFlag] = false`。
- (b) **`bindEndCleanup` onUnload else 分支**（非 preserve 场景）：`unit[floatFlag] = false`。
- (c) **动画完毕(998) B3 分支**：持枪族（长枪/双枪/手枪/手枪2）`技能浮空 = false`。

**例外**：跳跃接管时空中控制器 `_tick` **故意不清** `unit[floatFlag]`（`单位函数_fs_空中控制器.as` 状态切换块注释）—— 把标记让渡给跳跃状态。这条例外正是 §9.2 preserve 协议存在的理由。

## §10. 空中控制器审视 —— 下一步开拆入口

> 玩家卡死集中在浮空转换。路由侧已补回归网（见 [[project-routing-classification]]）；
> 下一步主战场是 `_root.空中控制器` 本体（[单位函数_fs_空中控制器.as](../scripts/逻辑/单位函数/单位函数_fs_空中控制器.as)）。
> 本节列开拆时优先看的风险面，作为 north star。

**第一等边界：空中控制器不可靠** —— 实例可能被替换 / 置空、unit 可能已卸载。routing 侧
`runJumpFloat` 已用 `airProvider` live-read 还原此语义；空中控制器本体的审视须以此为前提。

待审视风险面：

1. **source 并发与优先级** —— `__air.sources` 可同时挂 skillFloat / jumpFloat / naturalFall / fastFall / knockback / jetpack；`_tick` 对 fastFall 与 knockback 有 early-return，会跳过同帧其余 source 的处理。多 source 共存时的落地回调（`动画完毕`）有被吞的风险。
2. **jumpFloat 自动让出** —— `_tick` 在 `state != "空手跳"/"兵器跳"` 时立即 `delete sources.jumpFloat`。若状态被别处改写，jumpFloat 的落地回调 `unit.动画完毕()` 永不触发 → 疑似卡死路径之一。
3. **落地回调扇出** —— 落地块从 jumpFloat / naturalFall(`__自然落地接管`) / knockback 多处调 `unit.动画完毕()`；`动画完毕` 缺失 / 异常 / 重入 的后果待审。
4. **与 `fly_*` 槽位耦合** —— jetpack 落地时 `_tick` 直接写 `_root.fly_isFly1/2 = false`，与 `存储 / 读取当前飞行状态` 用的是同一组寄存器（见 §9.1 + [[project-routing-classification]] gap2）。
5. **任务生命周期** —— `__air.step` 是 33ms 周期任务，经 `EnhancedCooldownWheel`；`停止` / `启动` 的幂等、unit 卸载后的孤儿任务。
6. **`__air` 状态持久化** —— `确保` 惰性建 `unit.__air`；unit 若被对象池回收 / 复用，残留 `sources` 的隐患。
