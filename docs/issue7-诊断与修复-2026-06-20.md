## Bug #1 — 佣兵第 3 个起扣钱不入可用列表 ✅ 已修（compile 0 错）

**根因**：历史「空洞档」+ 新版 `handleHire` 用 `同伴数据.push` 追加。
- 旧版解雇用 `[]` 占位且不压缩 → `同伴数据` 中段是墓碑、`.length` 停在曾经的峰值（满编=5）。
- 新版 `handleHire` 用 `push`（落位=`length`），快照/进场读窗口固定 `[0,佣兵个数限制=5)`。
- 空洞档下 push 落到 index≥5（窗口外）→ 扣钱但快照/进场都读不到 → 「只能拥有 2 个，从第 3 个起失效」。

**修复**（2 文件）：
- [MercPanelService.as](../scripts/类定义/org/flashNight/arki/merc/MercPanelService.as)
  - `handleHire`：弃 `push`，改在 `[0,佣兵个数限制)` 找首个空槽写入（对齐老版 Symbol 2035 语义），落点恒在读窗口内；`同伴数` 用 `countCompanions()` 重算。
  - 新增 `countCompanions()`：统计 `[0,cap)` 内有效项（`[i][0]!=undefined`），作 `同伴数` 权威口径。
- [SaveManager.as](../scripts/类定义/org/flashNight/neur/Server/SaveManager.as) `unpackGameState`
  - 新增 `normalizeCompanionData()`：读档点把 `同伴数据`/`佣兵是否出战信息` 严格并行压实、`同伴数`=有效数，并**回收越界尾项**（修复已被旧 bug 写坏的存档，最多保留 cap 个）。对干净档幂等。

两层叠加：live 雇佣自身已修；读档归一化额外修复已损坏存档。

**真机验证点**：拿一个带空洞的老存档（曾满编 5 → 解雇到 2），进游戏 → 第 3 次雇佣应正常入列；读档后 `同伴数==同伴数据.length==有效数`。

## Bug #3 — 受击时位移技能（闪现/小跳/一瞬千击）反向 ✅ 已修（纯 AS2，compile 0 错）

**根因**：受击翻面 + 位移技能实时读 `方向`。
- [HitUpdater.as:71](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Updater/HitUpdater.as) 命中时（`!锁定方向`）`方向改变` 翻面朝攻击者。
- 三个位移技能经 `技能标签跳转_旧` 进 FLA 帧标签，位移每帧实时读 `parent.方向`（[主角技能.as:94](../scripts/逻辑/单位函数/单位函数_lsy_主角技能.as)）。
- 脱困位移帧里挨一发 → `方向` 被翻 → 这一帧朝敌堆冲。「有概率」= 是否恰好在位移帧窗口内命中。

**修复**（纯 AS2，**无需 FLA**，2 文件）：
- [引擎_fs_技能路由.as](../scripts/引擎/引擎_fs_技能路由.as) `技能标签跳转_旧`（所有施法唯一收口）：对 闪现/小跳/一瞬千击 开 `unit.位移锁向截止帧 = 当前帧数 + 30` 窗口。
- [HitUpdater.as](../scripts/类定义/org/flashNight/arki/unit/UnitComponent/Updater/HitUpdater.as)：翻面前加门控 `!(位移锁向截止帧 > 当前帧数)`。窗口内不翻面 → 位移读到的 `方向` 保持玩家意图。
- 帧戳**自动过期**，永不会把单位方向锁死；窗口仅覆盖这三招。

**待调**：窗口 `30` 帧为初值（FLA 里 dash 实际帧长静态取不到）。偏短=漏修一部分；偏长=受击短暂不转面向（基本无感）。真机按手感微调。

**根因**：`handleDeploy` 只改 `佣兵是否出战信息` 标志，实体只在场景切换的 `加载佣兵` 里生成。

**修复**（2 文件）：

## Bug #2 — 军阀前线基地 场景3「资料」无限循环动画 ⏳ 诊断完成 + 起测试面板

**根因**（引擎层）：拾取触发的全屏过场没有任何「结束/卸载/跳过」路径。
- [军阀前线基地.xml](../data/stages/基地房顶/军阀前线基地.xml) SubStage id=2 的 Pickup 事件只挂 `<Animation>`（`前线基地过场动画.swf`，13.6MB），**无 `<Pause>`、无 `<StageProgress>`**。
- [StageEvent.as:59](../scripts/类定义/org/flashNight/arki/scene/StageEvent.as) 只 `最上层加载外部动画`（loadMovie 贴顶层壳），不暂停、不推进、无卸载/跳过。
- 对照：StageManager 进/出图过场都绑场景生命周期（切场景时壳被清/下一图覆盖）；拾取过场是唯一不绑生命周期的调用 → 盖屏循环且无法跳过。

**方案（与用户讨论收敛）**：过场迁 WebView 面板（天然带生命周期 + 跳过），双路渲染：
- `<video>`（mp4/webm）= 主/保底路径，HW 解码，重资产（前线基地 13.6MB）转 mp4 收益最大；
- Ruffle（WASM）= 零重制兼容老 SWF。**「卡就转视频」是闭环兜底**，故 Ruffle 性能永远不是 blocker。
- 唯一前置：把 31 个过场按「不透明全屏 vs 半透明转场」分类（mp4 不支持 alpha；转场 stinger 走 WebM-alpha 或留 Flash）。

**本轮已起测试面板**（见下「动画测试面板」），用于真机验证 Ruffle 兼容性/性能。bug #2 的**生产整合**（重接 StageEvent/StageManager）继续推迟。

## Bug #5 — 慢速行走 + 拖拽背景版面 抽搐/掉帧 ⚠️ 静态分析未能定位

- 仓库内**无「慢速行走模式」字符串**：移动默认走路，Shift/双击方向=奔跑（[帧计时器.as:464](../scripts/通信/通信_fs_帧计时器.as)）。

- 最可能：拖**承载窗口**触发 Windows 模态 move 循环 → 消息泵占用 → Flash 渲染/帧计时器被饿死 → 抽搐+掉帧；慢走小步进时丢帧更明显。
- **需向报告者澄清**「背景版面」具体所指 + 操作方式 + 录屏；基本要真机复现才能定位。
