## 1. 背景与目标

竞技场现在已经具备三条可用基础：

1. 标准人形佣兵对战。
2. Web 本地采样下发 `roster`，AS2 走 `commitRoster` 生成非人形怪。
3. 势力爬升模式，用 `benchLevel` 驱动押金 / 奖励 / 波奖励基准。

真正缺口是：`benchLevel` 仍主要靠策划主观判断或粗公式回退。`arena-factions.js` 已明示“廉价非人形怪远弱于同级人形 merc，原始怪物等级会难度低奖励高”，但当前多数势力 `benchLevel:null`，运行时回退 `levelMax`，这会直接影响分组难度、奖励和后续赔率。

**目标**：搭建一个内部平台，用真实战斗采样把“某斗兽组合在指定对照阵容前相当于哪一档难度”转成可追溯数据，服务：

- 竞技场阶段分组。
- 非人形怪 / 势力卡 `benchLevel` 标定。
- 标准卡、堕落卡、爬升卡奖励倍率校准。
- 低样本条件下的主动剪枝与下一批采样规划。
- 后续斗蛐蛐赌场玩法的可行性评估。

**非目标**：

- 不在首期做玩家投注 UI。
- 不在首期做完整离线战斗模拟器。
- 不在首期要求所有随机数完全可复现。
- 不在首期自动写正式玩法配置；只产出建议与可追溯证据。

## 2. 现状勘察（依赖图谱）

### 2.1 Web / 数据侧

[arena-factions.js](../launcher/web/modules/arena-factions.js) 是当前势力卡人工元数据入口，`benchLevel` 设计意图已经明确：它是“等效挑战等级”，不是怪物原始等级。`arena-panel.js` 当前按 `benchLevel` 计算堕落卡和爬升卡押金 / 奖励；如果为空，就回退 `levelMax`。

[tools/derive-arena-meta-teams.js](../tools/derive-arena-meta-teams.js) 已能从 `data/stages/**` 派生 `data/arena/meta_teams.json` 和 Web rosters。当前 `powerRating` 只是 `Σ count×level` 的初版代理，适合作候选池排序和初筛，不足以作为最终难度标定。

[data/arena/arena_config.xml](../data/arena/arena_config.xml) 当前文件头明确写着：**本文件目前未被代码加载**，运行时真值在 `launcher/web/modules/arena-panel.js`。因此第一期标定结果不能假定“改 XML 即生效”。

### 2.2 AS2 竞技场运行态

[ArenaController.as](../scripts/类定义/org/flashNight/arki/merc/ArenaController.as) 已有可复用入口：

- `prepareArenaStage(deposit, reward, difficulty)`：预载 `DEATH MATCH角斗场`，设置押金、奖金、难度。
- `commitRoster(squad)`：提交非人形怪 roster，走 `_root.加载角斗场怪物`。
- `commitEscalation(...)`：提交势力爬升模式。

[关卡系统_lsy_关卡回调函数.as](../scripts/逻辑/关卡系统/关卡系统_lsy_关卡回调函数.as) 的 `角斗场加载` 会按 `_root.角斗场对手类型` 分叉；`_root.测试角斗场怪物` 已经能用 4 个盗贼兵种做 dev 验证，证明非人形怪通路可跑通。

[关卡系统_lsy_场景转换.as](../scripts/逻辑/关卡系统/关卡系统_lsy_场景转换.as) 的 `_root.加载角斗场怪物` 已经清空 `掉落物`，并按实际 attach 成功数修正 `_root.敌人同伴数`，这是标定通路的重要基础：标定不应刷装备，也不应因缺 linkage 卡死判胜。

首期观战式斗兽不能只复用 `commitRoster`：现有 `commitRoster` 只生成“敌方非人形怪”，默认仍让主角/同伴作为玩家方参战。新平台必须新增 calibration-only 的**双边刷怪路径**：蓝方斗兽作为玩家方/友方单位，红方斗兽作为敌方单位，主角本人不进入战斗或被移出战斗可达区域。这样标定对象才与后续斗蛐蛐玩法一致。

### 2.3 C# / 总线 / 日志侧

[ArenaTask.cs](../launcher/src/Tasks/ArenaTask.cs) 是现有竞技场 Web 面板桥，10 秒 timeout，适合 snapshot / enter，不适合作长时间批量标定。

[TaskRegistry.cs](../launcher/src/Bus/TaskRegistry.cs) 是活跃 task 的注册真值；如果新增 `arena_calibration` 控制入口、`arena_calibration_response` 回包入口或其他 HTTP 可调用任务，必须在此登记并在 status JSON 中声明。

[HttpApiServer.cs](../launcher/src/Bus/HttpApiServer.cs) 的 `/logBatch` 只是把 body 解码后写 `[LogBatch] ...` 到 `launcher.log`。这适合诊断，不适合作标定数据仓库。

[LogManager.cs](../launcher/src/Guardian/LogManager.cs) 的 `launcher.log` 会按 2MB / 日期轮转，仅保留一份 `.1`。长批次标定必须写独立 JSONL。

### 2.4 随机性现状

[LinearCongruentialEngine.as](../scripts/类定义/org/flashNight/naki/RandomNumberEngine/LinearCongruentialEngine.as) 的单例默认用当前时间初始化，且热路径方法直接内联读写 `seed`。项目内还存在 AS2 原生 `random()` / `Math.random()` 调用。虽然有 [SeededLinearCongruentialEngine.as](../scripts/类定义/org/flashNight/naki/RandomNumberEngine/SeededLinearCongruentialEngine.as)，但它主要适合测试实例，不等于当前全战斗链路可统一 seed。

结论：首期标定接受随机性，用多次采样和 HP 残值降低噪音；严格复现 seed 作为二期调试能力，不阻塞 MVP。

## 3. 技术路线决策

| 方案 | 优点 | 问题 | 结论 |
|---|---|---|---|
| 纯公式 / Node 离线评分 | 快、便宜、易批量 | 无法覆盖 AI、碰撞、技能、子弹、召唤、场景事件 | 只做候选池排序和初筛 |
| 复用现有 arena Web 面板跑批 | UI 现成，进入链路现成 | Web 面板桥 timeout 短，状态恢复与批量日志不适配 | 不作为内核，可用于人工复核 |
| 新增真实运行态 calibration service | 准确度最高，可复用真实战斗 | AS2/C#/Node 三栈施工，需防副作用 | **首期推荐** |
| 完整 headless Flash / 离屏模拟 | 理想批量能力强 | 当前项目仍依赖 Flash CS6/真实 SWF 运行态，投入过高 | 暂不做 |

最终架构：

```text
候选组合池
  -> case_manifest.json
  -> Launcher/C# calibration task
  -> AS2 ArenaCalibrationService 双边斗兽真战斗采样
  -> result.jsonl
  -> Node summary/analyzer
  -> rule / agent planner
  -> next_batch.json
  -> 人工确认 benchLevel / 阶段分组建议
```

## 4. 数据契约

第一期先定义文件契约，再施工运行态。所有文件放在独立目录，建议：

```text
tmp/arena-calibration/
  batches/<batchId>/case_manifest.json
  batches/<batchId>/summary.json
  batches/<batchId>/next_batch.json

logs/arena-calibration/
  <timestamp>-<batchId>.result.jsonl
```

### 4.1 `case_manifest.json`

一批要执行的战斗计划。示意：

```json
{
  "schema": "arena-calibration.case-manifest.v1",
  "batchId": "pilot-2026-06-29-a",
  "createdAt": "2026-06-29T00:00:00+08:00",
  "buildCommit": "d34391d2b0",
  "planner": { "type": "manual", "reason": "首批锚点验证" },
  "arenaMode": "spectator-beast-vs-beast",
  "blueBench": {
    "benchId": "thief-lv30x4",
    "description": "蓝方对照斗兽阵容；不是主角/同伴"
  },
  "cases": [
    {
      "caseId": "fallen-duoluocheng-lv30x4",
      "source": "manual-anchor",
      "blueRoster": [
        { "type": "兵种44", "level": 30 },
        { "type": "兵种45", "level": 30 },
        { "type": "兵种48", "level": 30 },
        { "type": "兵种49", "level": 30 }
      ],
      "redRoster": [
        { "type": "兵种44", "level": 30 },
        { "type": "兵种45", "level": 30 },
        { "type": "兵种48", "level": 30 },
        { "type": "兵种49", "level": 30 }
      ],
      "repeat": 5,
      "timeoutFrames": 5400,
      "tags": ["anchor", "expected-mid"],
      "plannerReason": "复用现有 _root.测试角斗场怪物 默认盗贼组作为通路锚点"
    }
  ]
}
```

关键约束：

- `caseId` 必须稳定，后续 summary / planner / 人工备注都引用它。
- `blueRoster` / `redRoster` 都只描述斗兽单位，不能携带奖励、押金、经济字段。
- `blueBench` 是标定对照阵容，不是玩家角色档。首期可以从人工锚点开始，后续逐步扩成多档蓝方锚点。
- 玩家不作为 case 输入，工具不得读取当前存档主角属性来构造战斗力基准。
- `repeat` 首批建议 `3~5`，边界组合再追加到 `20~50`。
- manifest 使用工具侧中性字段 `{type, level}`；C# 下发给 AS2 前必须归一化为 AS2 本地字段 `{兵种, 等级}`，或 AS2 入口第一步统一归一化。不要让两种字段形状同时流入刷怪函数。
- `startBatch` 时把 manifest 规范化并冻结到批次目录（若输入已在该目录则原地校验），计算 `manifestHash`；每个 case 计算稳定 `caseHash`，写入 result / summary。只靠 `caseId` 不足以区分同名 case 被改写后的样本。

### 4.2 `result.jsonl`

一局一行，append-only，不做人工改写。示意字段：

```json
{
  "schema": "arena-calibration.result.v1",
  "batchId": "pilot-2026-06-29-a",
  "buildCommit": "d34391d2b0",
  "manifestHash": "sha256:...",
  "caseId": "fallen-duoluocheng-lv30x4",
  "caseHash": "sha256:...",
  "runId": "fallen-duoluocheng-lv30x4#003",
  "sampleIndex": 3,
  "startedAt": "2026-06-29T00:01:10+08:00",
  "endedAt": "2026-06-29T00:02:43+08:00",
  "frames": 2790,
  "status": "finished",
  "winner": "blue",
  "blue": {
    "startHp": 18320,
    "startMaxHp": 18320,
    "endHp": 4120,
    "endMaxHp": 18320,
    "remainHpRatio": 0.225,
    "startCount": 4,
    "endAliveCount": 2
  },
  "red": {
    "startHp": 15600,
    "startMaxHp": 15600,
    "endHp": 0,
    "endMaxHp": 15600,
    "remainHpRatio": 0,
    "startCount": 4,
    "endAliveCount": 0
  },
  "deaths": [
    { "side": "red", "name": "红方3", "frame": 1022 },
    { "side": "blue", "name": "蓝方1", "frame": 2180 }
  ],
  "errors": []
}
```

必采指标：

- 追溯字段：`buildCommit`、`manifestHash`、`caseHash`、`caseId`、`runId`。
- 胜负 / `winner`。
- 战斗帧数 / 超时状态。
- 双方开局 HP 总和与最大 HP 总和。
- 双方结束 HP 总和与最大 HP 总和。
- 双方存活数。
- 死亡时间序列，首期可以只采 side/name/frame。
- 异常：timeout、stage_failed、spawn_failed、bridge_lost、abort。

运行级枚举口径：

- `status`：`finished | timeout | aborted | stage_failed | bridge_lost | invalid_case | spawn_failed`。
- `winner`：`blue | red | draw | null`；只有 `status:"finished"` 时才要求非 null，timeout / abort / bridge_lost 默认 null，若后续要按 HP 判定超时优势必须另加字段，不复用 winner。
- `errors[]`：对象数组，最小形状 `{ "code": "spawn_failed", "side": "red", "unit": "兵种44", "message": "..." }`。部分单位生成失败但双方仍有至少 1 个单位时，`status` 可以继续为 `finished`，失败细节只进 `errors[]`；任一方有效生成数为 0 时整局 `status:"spawn_failed"`。

HP 口径：以“单位完成生成后、开战前”的 `hp / hp满血值` 为 start；以 StageFinished / StageFailed / timeout 时刻为 end。若后续发现召唤、回血、复活会显著影响统计，再扩字段 `summonCount` / `postStartHpCreated` / `reviveCount`，不要在首期阻塞。

命名口径：`blue` 是对照方，`red` 是被测方或当前 case 主体。若某批次需要反向验证，新增镜像 case，不要在同一条 result 里改变 `blue` / `red` 语义。

### 4.4 `next_batch.json`

planner 输出，必须结构化并经过 schema 校验后才能执行。agent 输出也必须落成该形状：

```json
{
  "schema": "arena-calibration.next-batch.v1",
  "sourceBatchId": "pilot-2026-06-29-a",
  "planner": { "type": "rule", "version": 1 },
  "decisions": [
    {
      "caseId": "fallen-duoluocheng-lv30x4",
      "caseHash": "sha256:...",
      "decision": "append",
      "appendRepeat": 10,
      "reason": "边界样本，追加样本提高置信度"
    },
    {
      "caseId": "fallen-low-lv10x4",
      "caseHash": "sha256:...",
      "decision": "prune",
      "reason": "蓝方高胜率且高残血，红方明显低档"
    }
  ]
}
```

执行 `next_batch.json` 前必须校验 `caseHash` 仍命中 source summary 中的 case；不命中时拒绝执行，要求重新分析，避免 agent 或人工基于旧 case 内容追加样本。

## 5. 采样指标与粗判模型

胜率是主指标，但首期样本很小，不能只看胜负。HP 残值、战斗时长、存活数能把“险胜”和“碾压胜”拆开，是第一版比随机数复现更有工程意义的部分。

推荐第一版局内优势分：

```text
advantage =
  winLossBase
  + winnerRemainHpRatioWeight
  + survivorWeight
  + durationWeight
  - timeoutPenalty
```

不要过早追求一个完美公式。第一版只要求 summary 能稳定标出：

- `red_too_weak`：蓝方高胜率且高残血，红方明显低档。
- `red_too_strong`：蓝方低胜率且红方高残血，红方明显高档。
- `candidate_boundary`：胜负接近，胜方残血低，战斗时长合理。
- `unstable`：胜负或 HP 方差大，需要追加或排除。
- `timeout_abnormal`：战斗过长，不进入普通挡位。

初始剪枝规则：

| 条件 | 决策 |
|---|---|
| 蓝方胜率 `>= 0.8` 且蓝方平均剩余 HP `>= 0.6` | 红方明显过弱，停止追加 |
| 蓝方胜率 `<= 0.2` 且红方平均剩余 HP `>= 0.6` | 红方明显过强，停止追加 |
| 胜率 `0.4~0.6` | 高价值边界，追加 |
| 胜负偏一边但胜方平均剩余 HP `< 0.25` | 可能接近目标档，追加 |
| 超时率 `> 0` | 标记异常，先不归普通难度 |
| 样本方差高 | 追加少量样本或交给 agent 复核 |

## 6. AS2 侧设计

新增建议类：`org.flashNight.arki.merc.ArenaCalibrationService`。

职责：

- 注册 `_root.gameCommands["arenaCalibrationRun"]`。
- 接收单局 case：`caseId`、`runId`、`blueRoster`、`redRoster`、`timeoutFrames`。
- 入站 roster 先归一化为 AS2 本地字段 `{兵种, 等级}`，再进入双边刷怪；缺字段、未知兵种、非法等级要回 `spawn_failed` / `invalid_case`，不要静默当空阵容。
- 设置 `_root.斗兽标定模式 = true`，并缓存需要恢复的全局状态。
- 调用 `ArenaController.prepareArenaStage(0, 0, "")`，但不走普通 `commitRoster(squad)` 作为完整战斗入口；首期需要 calibration-only 的双边刷怪入口。
- 生成蓝方斗兽为玩家方/友方单位，生成红方斗兽为敌方单位，记录双方单位句柄用于 HP 和死亡采样。
- 主角本人不进入斗兽场战斗。实现可以先把主角出生点移到观战席/不可达区域并禁用参战影响，后续再做专用观战 UI。
- 在战斗结束、失败或超时时采集指标并回包。
- 清理 `_root.斗兽标定模式` 与 arena 临时字段。

不建议复用现有 `ArenaTask` 直接跑批，因为它面向 Web 面板，timeout 只有 10 秒；斗兽单局可能几十秒到数分钟，需要新的长任务语义。

安装入口不要依赖 AS2 类文件“存在即执行”。按现有 `竞技场系统_WebView.as -> ArenaPanelService.install()` 模式新增显式 install 接线，例如在竞技场 WebView 分区或相邻标定分区调用 `ArenaCalibrationService.install()`；新增类被引用处按 AS2 约束显式 `import`。

### 6.0 双边刷怪口径

现有 `_root.加载角斗场怪物` 只生成敌方单位，并服务普通竞技场判胜。斗兽标定第一版应新增专用函数，例如 `_root.加载斗兽标定单位(side, roster, x, y)`，不要把双边逻辑硬塞进普通竞技场路径。

首期建议：

- `side:"blue"`：`是否为敌人=false` 或按现有阵营系统等价设置为玩家方/友方；命名使用 `蓝方0...N`。施工时必须验证非人形单位在友方阵营仍会主动攻击红方；若 AI 只会围绕玩家或不主动索敌，应新增标定专用阵营 / target override，不要继续假设 `是否为敌人=false` 足够。
- `side:"red"`：`是否为敌人=true`；命名使用 `红方0...N`。
- 两边都清空 `掉落物=[]`。
- 两边都按 attach 成功数记录 `startCount`，缺 linkage 的单位写 `spawn_failed`。
- 判胜不要复用只看 `_root.敌人同伴数` 的普通角斗场 finishRequirement；标定服务自己轮询蓝/红存活数，决定 `status` 与 `winner=blue/red/draw/null`。
- 主角/同伴不计入蓝方 HP、存活数或胜负，也不得进入红方或蓝方 AI 的有效 target set。若第一版用“移到观战席 / 不可达区域”实现，验收必须确认敌方不会追主角、远程技能不会打主角、主角/同伴死亡或受击不会改变胜负。

### 6.1 副作用隔离

首期必须守住：

- 押金 `deposit=0`，奖金 `reward=0`。
- 不发玩家奖励，不刷掉落，不写赌场经济。
- 不自动存盘。
- 运行前后记录并恢复 `_root.金钱`、`_root.虚拟币`、`_root.押金`、`_root.角斗场奖金`、`_root.存档系统.dirtyMark`、`_root.killStats`。
- 运行前清理上一场 `_root.角斗场对手类型`、`_root.角斗场roster阵容`、`_root.角斗场爬升` 残留。
- 标定胜负不得调用正式 `_root.FinishStage`、任务完成、成就进度、赌场结算或普通角斗场发奖路径。若首期因复用 `DEATH MATCH角斗场` 无法完全避开普通 `StageFinished` 链，必须在专用标定存档上验证任务/成就/经济状态无污染，并把仍会变动的字段列入 snapshot / restore 或改成标定专用结束链。
- 首期仍建议使用专用标定存档或 disposable save；即便玩家不参战，也不要在正式档上做长批次风险验证。

注意：第一版不刷新玩家入场，因此不应把玩家 HP、弹药、技能冷却、同伴状态纳入标定输入。验收边界改为“专用标定存档 + 主角旁观不参战 + 双边斗兽单位可重建 + 不自动存盘 + 显式恢复经济/dirtyMark”。后续如要在同一场景连续跑很多局，再补更强的场景清场和斗兽 actor reset。

### 6.2 结束检测

优先级：

1. 订阅或轮询关卡状态：`_root.获取关卡状态()` 返回 `Finished` / `Failed` / `Active`。
2. 订阅 `_root.gameworld.dispatcher` 的 `StageFinished` / `StageFailed` / `UnitDeath`，用于死亡时间线。
3. 超过 `timeoutFrames` 强制返回 `timeout`，不要让批次卡死。

首期允许轮询实现，后续再用事件订阅降开销。

## 7. C# 侧设计

新增建议类：`ArenaCalibrationTask.cs`。

职责：

- 对外提供 HTTP `/task` 可调用入口，例如 task=`arena_calibration`，action=`startBatch/status/abort`。
- 对 AS2 下发 `arenaCalibrationRun`。
- 接收 AS2 `arena_calibration_response`。
- 管理批次队列、当前 run、长 timeout、abort。
- 将每局结果 append 到独立 JSONL。
- 只把简短进度写 `LogManager.Log`，不把完整结果写 `launcher.log`。

新增 task 时要改：

- `TaskRegistry.RegisterAll`：注册 `arena_calibration` 控制入口（HTTP `/task` 入站）与 `arena_calibration_response` 回包入口（AS2 出站）。
- `TaskRegistry.ToStatusJson`：声明 `arena_calibration` 为 `httpCallable:true`，`arena_calibration_response` 为 `httpCallable:false`。
- `Program.cs` / 构造注入：按现有 Task 模式接入。
- `HttpApiServer.HandleTask`：无需新增专用 endpoint，优先复用 `/task`。

`/task` 只能承载**控制面**，不能承载整批或单局长等待。现有 `HttpApiServer.HandleTask` 对 async task 固定等待约 15 秒并丢弃超时后的 late callback；因此 `arena_calibration` 的 `startBatch/status/abort` 必须快速返回 ACK / 当前状态，批次运行、单局 timeout、JSONL 写入和 AS2 回包关联都由 `ArenaCalibrationTask` 自己的后台状态机管理。如果未来需要 HTTP 长轮询或流式进度，必须先另起切片调整 `/task` 超时模型。

建议 HTTP 控制面形状：

```jsonl
{ "task": "arena_calibration", "action": "startBatch", "callId": "web-1", "manifestPath": "tmp/arena-calibration/batches/pilot/case_manifest.json" }
{ "task": "arena_calibration", "action": "status", "callId": "web-2", "batchId": "pilot-2026-06-29-a" }
{ "task": "arena_calibration", "action": "abort", "callId": "web-3", "batchId": "pilot-2026-06-29-a" }
```

`startBatch` 成功只表示后台批次已接管，响应应包含 `batchId/state/resultPath/manifestHash`，不表示任何单局已经完成。HTTP `callId` 只用于控制请求响应；下发给 AS2 的单局 `callId` / `runId` 必须由 `ArenaCalibrationTask` 单独生成并映射，不能复用 HTTP 请求的 `callId`。

`manifestPath` 必须解析到项目根下的 `tmp/arena-calibration/`，`resultPath` 必须解析到项目根下的 `logs/arena-calibration/`；拒绝 `..` 穿越、UNC 路径和任意绝对路径。即便 HTTP 端点只面向本机，也不要让标定任务成为任意文件读写入口。

### 7.1 C# 状态机

```text
Idle
  -> BatchLoaded
  -> Running(runId)
  -> WaitingFlashResult
  -> AppendResult
  -> Running(next)
  -> Complete

任意状态:
  abort -> Aborted
  socket disconnect -> Paused/Error
  run timeout -> Append timeout result, continue/stop by policy
```

关键约束：

- 任何时刻只允许一个 active run。Flash 运行态不是并行战斗容器。
- `callId` 必须单调，AS2 回包必须能映射到 runId。
- 断连后不要把批次标成功；保留已写 JSONL，summary 标 partial。
- timeout 也要写入结果，不能只写日志。

## 11. 风险与债务防线

### 11.1 随机数

随机性会影响单局，但不应阻塞首期。首期用小批量重复 + HP 残值 + 方差标记吸收噪音。严格 seed 注入放二期，且必须先梳理 `LinearCongruentialEngine`、AS2 `random()`、`Math.random()` 的入口，不要只改一个引擎就声称全链路可复现。

### 11.2 战斗耗时

真实战斗时间长，样本稀缺。必须把剪枝放进第一版闭环，但只做批次级：

```text
3~5 局 pilot -> summary -> rule/agent planner -> next_batch
```

不要让 agent 每局或每帧介入。

### 11.3 状态污染

斗兽平台是内部工具，不能污染正式玩家档。第一版要求专用标定存档；即使主角不参战，运行态仍要恢复明显经济与 dirtyMark。后续如果要大规模自动化，必须补完整斗兽 actor reset、场景清场或专用战斗沙箱。

### 11.4 配置 SOT

当前 `arena_config.xml` 不是运行时真值。标定建议沉淀后，不要直接写 XML 并误以为生效。若要让标定结果进入正式运行，需另起“竞技场配置 SOT 迁移”切片：C# 启动解析配置，snapshot 下发 cards，Web 删除硬编码。

### 11.5 生成物膨胀

JSONL 可能快速变大。保留原始数据，但 summary / next_batch 只引用必要统计。不要把 JSONL 内容复制进 markdown。

## 12. 斗蛐蛐赌场可行性预告

赌场玩法可行，但必须排在内部标定平台之后。内部平台产出的不是“可直接上架赔率”，而是赔率评估的基础数据。

后续可行性报告至少回答：

- 胜率是否稳定到足以定赔率，还是只能做娱乐展示。
- 是否引入庄家抽水、投注上限、每日限额、冷却和赔率四舍五入。
- 是否隐藏部分组合参数，避免玩家背表套利。
- 是否允许玩家影响参赛阵容；如果允许，赔率必须重新计算或禁止下注。
- 是否把投注收益分成娱乐代币与真实经济，避免破坏金钱曲线。
- 是否使用“平台推荐赛程”而非玩家自选对局，减少套利空间。
- 是否设置异常风控：同一组合高频投注、赔率偏差、版本更新后赔率失效。

进入赌场施工前置门：

1. 内部标定平台能稳定跑批。
2. 至少一个势力/阶段的 benchLevel 已由数据支持并人工确认。
3. 标定结果能解释，原始 JSONL 可追溯。
4. 经济回收模型单独评审，不与难度标定 PR 混在一起。
