## 说明

这个文件只保留测试入口和测试设计说明，不再混入历史跑数日志。

## 当前测试目标

1. 验证 `JSON`、`LiteJSON`、`FastJSON` 的常规正确性。
2. 显式记录三套实现的已知差异。
3. 提供更可信的 parse / stringify 性能基准。
4. 对 benchmark 自身的负载与缓存假设做最小自校验，避免“测得很稳但测错对象”。

## 重点差异

- `LiteJSON` 不支持 Unicode 转义解析。
- `LiteJSON` 不支持 `\b` / `\f` 还原。
- `LiteJSON` 不支持科学记数法。
- `LiteJSON` 会忽略根值后的 trailing token。
- `FastJSON.parse()` 对同一字符串会返回同一对象引用。
- `FastJSON.stringify()` 对同一对象实例会命中身份缓存，可能返回旧字符串。
- `FastJSON` 对“对象尾逗号”有容错行为，但对多数其他非法输入仍会抛错。
- `JSON` 的非法输入测试改为“记录错误但不触发异常上报副作用”的路径，避免测试日志被服务器消息干扰。
- `LiteJSON` 对部分 EOF 类非法输入（如未闭合数组）存在死循环风险，这类用例在套件里显式跳过。
- `1e` 这类“不完整指数”不再归入统一的非法输入断言，而是作为单独的差异行为矩阵记录当前实现语义。

## 性能基准原则

- 三套实现都走等价循环路径，避免把闭包开销混进某一方。
- parse / stringify 都有基线扣除。
- `FastJSON` 明确区分三档语义：`失配`（同实例不同 payload）、`严冷`（每次操作新建实例）、`热`（缓存命中）。
- `FastJSON` 冷路径使用“有限变体循环 + 批量冷操作聚合”的方式维持 cache-miss 负载并放大量级，而不是依赖无限变体。
- 自动放大迭代次数，尽量避开 `getTimer()` 的分辨率陷阱；日志中的 `批/轮 x N = 总次数` 表示该轮把多个冷操作打包测量。
- 冷路径的规模校准基于重复采样的中位数，而不是单次试跑，避免偶然抖动把批次数误锁定在过小量级。
- 在进入性能区前，套件会先自校验变体字符串确实不同、对象变体不共享引用，以及 `FastJSON` 的 `失配` / `严冷` 语义没有被缓存副作用污染。
- 置信度不再只看扣除基线后的 `adjusted` 时间，也会参考原始总时长和基线时长的稳定性；低置信度行会额外打印 `原始/基线` 供排查。
- 如果数据仍接近计时下限，会标记为“低置信度”或“低于计时分辨率”。
- 只有双方都达到高置信度时才输出倍率，避免用低置信度样本得出误导性倍数。
