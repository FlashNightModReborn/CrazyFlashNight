**文档角色**：审计 / 决策沉淀（`docs/*` 层）。一次性调查归档，非 canonical doc。
**最后核对代码基线**：commit `45156f419`（2026-05-20）。

## 0. 背景

大地图卡顿的根因：地图背景的矢量图未被烤进位图、每帧实时渲染。
`_root.贴背景图()` 会把 `gameworld.背景` 烤进 `deadbody` 位图层并 `unloadMovie`
释放矢量内存——但**前提是地图里存在实例名为「背景」的元件**。
commit `0f6c8c4ac` 对 `地图-靶场` 完成了这套样板施工；本调查盘清剩余覆盖面，
并实测了一直被当作天花板的 BitmapData 尺寸上限。

相关工具：[tools/swf-audit/](../tools/swf-audit/)（纯 Python SWF 扫描器 + 审计脚本）。

## 3. BitmapData 容量实测（摘要）

> 完整 canonical 档案见
> [AS2-BitmapData-尺寸上限实测.md](../scripts/优化随笔/AS2-BitmapData-尺寸上限实测.md)
> （方法论、四轮交叉验证数据、API 陷阱、复现入口）。本节只留与解锁决策相关的摘要。

游戏自带 projector **Flash Player 11.2.202.228** 上经四轮独立交叉验证：

- **AVM1 的 `flash.display.BitmapData` 没有 2880 / 8191 边 / 16,777,215 总像素
  限制**——AS2 语言参考里的 2880 是 Flash 8 冻结文档，不绑定本运行时。真实约束是
  内存（`w×h×4` 字节）。
- 失败形态不统一：近 2 GiB 堆顶 → 播放器**挂死**；`≥2^31` 字节 → clean `null`。
  → **不能靠 `if(bd==null)` 兜 OOM。**
- ⚠ 上述数字在 `TestLoader`（近空 SWF）里测得；真实游戏进程可用预算远小于此。

## 4. 决策：解锁 `addBodyLayers` 的位图钳制

[SceneManager.as](../scripts/类定义/org/flashNight/arki/scene/SceneManager.as)
`addBodyLayers()` 旧代码把 `deadbody.layers` 两张 BitmapData 硬钳到 **2880×1024**——
该常量是 Flash 8 时代 BitmapData 上限的遗留，已被 §3 实测推翻，且会让宽度超 2880
的地图背景 / 尸体层在 2880 处**静默截断**。

**改动**：上限钳制 `2880×1024` → `8192×4096`，注释改为反映实测真相。

**取舍依据**：
- 2880 不是真限制；运行时能力远超之。
- 内存不是实际约束：手作（非程序生成）地图受**战斗体验**自限——做不出、也玩不了
  超大地图，所以钳制上限永远不会被真实内容触及，实际内存占用 = 真实地图尺寸。
- 钳制因此降级为**防御性护栏**：仅用于挡住 XML 里的异常 / 笔误尺寸值，
  避免 `new BitmapData` 拿到病态参数（§3.2 的挂死区）。`8192×4096` 对任何
  战斗可行的地图都留有大幅余量。
- 未触及 `贴背景图` 的 null 兜底：在 `8192×4096` 内分配实际上不会失败；
  若未来要做超大图 / 瓦片化，再单独处理失败路径。

> 若未来目标是程序生成的超大世界图，单张大位图模型仍应让位给瓦片化烤图
> （把 `deadbody.layers` 改成 BitmapData 瓦片网格）——原因是工程稳定性，
> 不是 2880 硬墙。
