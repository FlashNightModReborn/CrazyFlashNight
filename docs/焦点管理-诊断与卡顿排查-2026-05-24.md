## 1. 为什么这是个长期问题

Flash SA 嵌入 launcher 的窗口栈本质上有**多个独立信号源**在抢"前台"这个语义：

| 信号源 | 形态 | 工作粒度 |
|---|---|---|
| `WM_ACTIVATEAPP` | 进程级激活 | 同进程窗口间切换**不**触发，跨进程切换才触发 |
| `GetForegroundWindow()` | 窗口级前台 | 任何瞬态窗口都会改变它 |
| `WM_SIZE`/`SIZE_MINIMIZED` | 最小化态 | 与前台正交 |
| `SystemEvents.SessionSwitch` | 会话锁屏 | 默认桌面侧前台恒为 NULL |
| WebOverlay panel 接管 | 同进程前台转移 | 不触发 WM_ACTIVATEAPP，但 `GetForegroundWindow()` 变 |
| 嵌入后的 Flash 子 HWND | PID 归 Guardian，但仍是独立 HWND | `IsFlashForeground` 判定要考虑会话边界 |

外部干扰源更多：

- QQ / Telegram / Discord 等 IM 通知的 toast 窗口（瞬时抢焦再归还）
- 输入法候选窗（搜狗/微软拼音）
- Steam Overlay 弹出
- Windows 系统通知 / 任务栏弹窗
- 杀软扫描时的瞬态窗口
- 后台程序退出后留下的"焦点真空"（`GetForegroundWindow()` 返回 NULL）

**核心难点**：这些信号在时间维度上重叠，单一信号不足以判定"用户是不是真的在玩游戏"。任何用单一信号源的判定都会在某种边界翻车，过去半年的迭代基本就是这件事的"打地鼠"。

## 2. 当前架构（4 层防御，commit `e1d1ee0f6` 收束）

> 详细注释在源码里，这里只画地图。修改前**先**回去读源码 doc-comment。

### 层 ①：`AppActivationState`（权威激活态，去抖）

- **路径**：[`launcher/src/Guardian/AppActivationState.cs`](../launcher/src/Guardian/AppActivationState.cs)
- **信号源**：`WM_ACTIVATEAPP`（进程级）+ `WM_SIZE`/`SIZE_MINIMIZED`
- **喂入**：[`GuardianForm.WndProc`](../launcher/src/Guardian/GuardianForm.cs)
- **去抖窗口**：200ms（曾是 450ms，e1d1ee0f6 收窄）
- **设计意图**：取代散落各处的 `GetForegroundWindow()` 现场轮询。后台 toast 瞬时抢焦再归还会产生 `WM_ACTIVATEAPP(false)→(true)` 抖动，宽限窗内重新激活视作从未失活。
- **线程模型**：UI 线程写、钩子线程/FrameTask 线程读，全 volatile + 单调 TickCount，**无锁**。改这里前先确认你理解 §3。

### 层 ②：`KeyboardHook`（按键归属判定）

- **路径**：[`launcher/src/Guardian/KeyboardHook.cs`](../launcher/src/Guardian/KeyboardHook.cs)
- **核心方法**：`ShouldInterceptForOurApp(bool strict)`
- **判据**（按优先级）：
  1. 非 strict 键：先看 `AppActivationState.IsAppActive`（去抖后）
  2. `GetForegroundWindow() == NULL`（真空）→ 视作我方（真空下按键不会落到别的程序）
  3. 前台窗口 pid 归本进程或 Flash 进程
- **strict 例外**：破坏性键（Ctrl+Q 杀进程）跳过判据 1 的去抖宽限——宽限窗内无法区分"toast 抢焦"与"用户真的切走"，宁可漏触发也不能在别的程序里把启动器干掉。

### 层 ③：`PerfDecisionEngine`（性能决策门控）

- **路径**：[`launcher/src/Guardian/PerfDecisionEngine.cs`](../launcher/src/Guardian/PerfDecisionEngine.cs)
- **门控**：`!IsAppActive || IsMinimized` → 停决策 + 3 秒 cooldown
- **为什么**：Flash 非前台自行降帧，AS2 上报的 FPS 已不可信。继续决策会把后台降帧误判成性能不足，污染 tier——回到游戏后画质卡在低档（这是过去的回归 bug）。
- **panic 兜底**：失活时**不**停 panic 降级（旧实现因 panic 被掐断导致 FPS 探底后无救援）。

### 层 ④：前台看门狗（兜底真空回收）

- **路径**：[`GuardianForm.SetupForegroundWatchdog`](../launcher/src/Guardian/GuardianForm.cs)
- **节奏**：400ms 定时器
- **触发条件**：`GetForegroundWindow() == NULL` 连续 **≥2 tick**（≈800ms）
- **动作**：`WindowManager.RestoreFlashInputFocus("fg_watchdog:vacuum")`
- **为什么需要连续确认**：MSDN 明确 "a window is losing activation" 瞬间 `GetForegroundWindow()` 即为 NULL——单帧采样会与用户主动切窗竞态，误把交接瞬间当真空、把前台抢回 Flash 对抗用户。
- **停转条件**：`SessionLock`/`ConsoleDisconnect`/`RemoteDisconnect`（锁屏 / 安全桌面 / 快速用户切换期间默认桌面侧前台恒为 NULL）。

## 3. 诊断日志体系（2026-05-24 补全）

补这层日志的原因：测试员反馈卡顿仍存在，但 commit `e1d1ee0f6` 之后**抢焦肇事者在日志里完全不可见**——`AppActivationState` 的写入静默、`PerfDecisionEngine` 的门控静默、`[FocusRestore]` 只在真空已成型后才打（fg 已是 NULL）。

### 3.1 `[Activation]` — WM_ACTIVATEAPP 进出对

源：[`GuardianForm.WndProc`](../launcher/src/Guardian/GuardianForm.cs) WM_ACTIVATEAPP 分支

```
[Activation] lost fg=0xABCD pid=1234 class=TXGuiFoundation title="QQ"
[Activation] gained gapMs=87
```

- **`lost fg=`** 是抢焦发生瞬间 `GetForegroundWindow()` 抓到的肇事窗口快照（pid/class/title）。**这是唯一能识别"是谁抢的焦"的实时窗口**——抢焦窗口之后可能消失，再查就晚了。
- **`gapMs`**（lost→gained 时间差）直接区分场景：
  - `< 200ms` → toast 瞬时抢焦，被 `AppActivationState` 去抖吃掉；理论上不影响性能决策
  - `200ms ~ 数秒` → 真失活了，`PerfDecisionEngine` 已停决策
  - 秒级 → 用户切走又切回

### 3.2 `[PerfGate]` — 性能门控转换

源：[`PerfDecisionEngine.Evaluate`](../launcher/src/Guardian/PerfDecisionEngine.cs) 失活门控分支

```
[PerfGate] closed reason=deactivated tier=0 latestFps=27.3
[PerfGate] opened tier=0 latestFps=14.2 cooldownMs=3000
```

仅在【转换】时打，不每帧刷。配 `[Activation]` 一起读就能回答"抢焦时 tier 在哪个档、回来后是不是被卡在低档"。

### 3.3 已有的 `[FgWatchdog]` / `[FocusRestore]`

为完整性记录在此（不是这次新加）：

- **`[FgWatchdog] foreground vacuum detected`**：看门狗确认了 ≥2 tick 真空，准备回收。此时 `GetForegroundWindow()` 已是 NULL——肇事者已退场，**抓不到名字**。
- **`[FocusRestore] <reason> ...`**：所有主动回收前台的路径都打。reason 字段决定归因来源：
  - `fg_watchdog:vacuum` — 来自 §2 层 ④
  - `panel_close:idle:<tag>` / `panel_close:soft:<tag>` — 来自 WebOverlay panel 关闭
  - `set_ready` 等 NativeHud idle 调度

## 4. 日志解读速查表

测试员复现卡顿后，把卡顿时间戳前后 ±5 秒的 `logs/launcher.log` 截出来，对照下表：

| 日志组合 | 解读 | 下一步 |
|---|---|---|
| `[Activation] lost` + 短 `gapMs (<200)` + **无** `[PerfGate]` | 去抖吞掉的 toast 抢焦 — 卡顿成因**不**是性能控制 | 看 `fg=` 里的 class 名锁定骚扰进程；考虑用户能否关掉它的通知 |
| `[Activation] lost` + 长 `gapMs (>200)` + `[PerfGate] closed/opened` 配对 | 真实失活；**重点看 opened 时 tier 是不是 1** | 若卡低档，回去检查 panic 降级路径是否在失活期间被误触发 |
| `[FgWatchdog] vacuum` 但**没有**对应的近期 `[Activation] gained` | 抢焦未归还的真空场景 | 往前找最近的 `[Activation] lost fg=` — 那个 class 名就是嫌疑人 |
| 短时间内多次 `[Activation] lost/gained` 配对 | 后台程序周期性骚扰（自动更新、心跳通知） | `fg=` 的 class 名锁定；判断要不要在 launcher 层"原谅"它（白名单不算失焦） |
| `[FocusRestore] ... pass1=fail pass2=fail` | AttachThreadInput hack 都失败了 — 焦点被某个高权限进程钉死 | 查 `fgBefore=` 的 pid，看是不是 UAC 提升进程 / 反作弊驱动 |

### 4.1 已知嫌疑进程的窗口 class 名速查

- `TXGuiFoundation` → QQ
- `Chrome_WidgetWin_*` → Chromium 系（Discord/Edge/Electron 通知）
- `Shell_TrayWnd` → Windows 任务栏右下角弹窗
- `ChineseIme*` / `SogouImeIE*` / `MSCTFIME*` → 输入法候选窗
- `vguiPopupWindow` → Steam Overlay
- `tooltips_class32` → 通用 tooltip 弹窗
- `Windows.UI.Core.CoreWindow` → UWP 应用通知中心

## 5. 还没回答的疑点（卡顿根因候选）

补完日志后等测试员日志样本，下面这些都是当前**没排除**的候选：

1. **去抖窗口 200ms 仍不够短**：某些 toast（如 Windows 11 通知中心）可能停留 300~500ms，恰好超出去抖、刚好触发 `[PerfGate] closed/opened` 一对，3 秒 cooldown 期间画质会被切到低档；用户感受是"卡了一下"。
   - 怎么验：日志里如果出现"`gapMs` 300~400 + `PerfGate` 配对 + 后续 `[PerfActive]` 出现 tier=1 → 0 的回弹"——就是这个。
2. **看门狗 800ms 确认窗 + 400ms 间隔 + 2 秒节流**：最坏 case 真空持续超过 800ms 才动作；这期间 Flash 已经降帧，玩家感受到的卡顿可能 1~2 秒长。
   - 怎么验：`[FgWatchdog] vacuum detected` 与之前最后一次 `[Activation] gained` 的时间差。如果常见 >1s，可以考虑把 streak 阈值降到 1（接受偶发误抢回风险）。
3. **Flash 子窗口 PID 归属错乱**：嵌入后 PID 归 Guardian，但底层 SetParent 与窗口生命周期的某些边界态（Flash 自重启？）可能让 `IsWindowInFlashSession` 错判。
   - 怎么验：`[FocusRestore]` 的 `fgBefore=` pid 是否在某些时刻出现非 Guardian 且非 Flash 的值。
4. **panel 关闭瞬间的双跳**：WebOverlay panel 关闭会触发 `panel_close:idle` 路径，期间 `GetForegroundWindow()` 可能短暂为 NULL，看门狗误判真空。
   - 怎么验：`[FocusRestore] panel_close:*` 之后 800ms 内是否紧跟 `[FgWatchdog] vacuum detected`。看门狗有 `IsPanelMode` 抑制，但 panel→idle 切换瞬间的窗口期可能漏。
5. **AS2 端的 FPS 上报受失焦节流影响**：Flash SA 非前台时帧率被系统降到 ~8 FPS。如果 AS2 在`[PerfGate] closed` 之前就上报了几个低 FPS 样本，可能已经触发 panic（panic 不被门控）。
   - 怎么验：`[PerfActive]` panic 跳变与 `[Activation] lost` 的时序。

### 已被否决的方案（**不要重新提出**）

- ❌ **把去抖窗口拉长到 500ms+**：会让破坏性热键的安全窗口扩大，Ctrl+Q strict 路径就是为这个加的护栏。
- ❌ **轮询 `GetForegroundWindow()` 取代 `WM_ACTIVATEAPP`**：早期实现，输给了"窗口级前台 vs 进程级激活"的语义差——同进程窗口切换会被误判失活。
- ❌ **看门狗单帧采样真空就动作**：与用户主动切窗竞态，玩家反馈"切出去窗口又被吸回来"。
- ❌ **失焦（非最小化）也停决策**：导致 panic 降级被掐断，FPS 探底后无救援。e1d1ee0f6 之前的 bug。

## 7. 下一步（待测试员日志样本）

收到带卡顿时间戳的 `logs/launcher.log` 之后：

1. 按 §4 速查表分类，初步定位卡顿属于哪种 pattern。
2. 若是 §5.1（去抖窗口边缘命中），可能要在层 ③ 加"刚 opened 的前 N 秒禁止降级"，而不是动去抖。
3. 若是 §5.2（看门狗反应慢），评估把 streak 降到 1 + 增加"真空持续时长"的二次门控。
4. 把验证结论回填到本文档 §5，并在 MEMORY.md 的 focus-management 入口加一行总结。

**不要**直接动架构——4 层防御之间的耦合性很强，过去 5 轮迭代每次想"顺便简化一下"都引入了新的边界 bug。
