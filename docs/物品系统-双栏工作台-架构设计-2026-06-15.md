### 0.0 评审改判决议（2026-06-20，外部双评审 + 代码核查）

本文经两份外部 AI 评审（Gemini 3 Deep Think、GPT-5.5 Pro）独立审阅，二者**收敛到同两处地基问题**；随后经代码核查 workflow 把评审的代码级断言逐条对真机源码复验，确认**外部 AI 看不到实现、其中数条断言被代码证伪或夸大**。最终改判如下；下方各节已按此重写，任何与本块冲突的旧表述以本块为准。

**决议 A — 首战 KShop 不承载"泛化 Broker 已验证"，只冻结 Shell/ViewHost。** KShop 全程未流经通用跨栏转移引擎（目录→购物车=模态②意图暂存、checkout=模态②整体提交、claim/discard=模态①b 单向领域写），无一条是 owned↔owned 的 move/merge/swap。故：①§8 把首战正名为「KShop 双栏壳与可靠性试车」，删除"验证完整 Broker"暗示；②在迁移序列插入明确架构门——**KShop（冻结 Shell，Broker/并发契约标 `provisional`）→ 背包↔仓库最小纵切（真正的 Broker 验收门 Gate A2）→ 冻结 Broker → 后续界面**；③§2 第一性原则重写为四层（Shell 只认 View 生命周期 / InteractionBroker 只处理跨 View Intent / DomainCoordinator 负责命令与并发），避免 Broker 退化成同时认识 `shopSaveCart/inventoryMove/equip/claim/sell/craft` 的中央 switch（N² 耦合换地方没消失）。这些抽象在全仓零既有同名符号，拆分/改名零迁移成本。

**决议 D — 废除"完整存储态 canonical descriptor 作 v1 普遍并发防线"，改 AS2 snapshot-local slot-lease。** full canonical 把"防陈旧槽位误操作"扩大成"任意深层字段完全相等"，语义过度、且在 AVM1 上要递归遍历+键排序+对象分配+字符串膨胀（1200 格仓库放大），是性能自杀。改用 AS2 铸造的 slot-lease（O(1) 按对象引用判定）+ 按命令定义的语义前置条件（见 §4.1 新表）。canonical encoder 降级为**诊断工具/测试 oracle/少数确需 exact-storage-equality 命令专用**，不再是 inventory-domain 全局基础设施。**代码核查给 lease 落地补了关键边界**：`add/getItem/纯 rebuildOrder` 全程零克隆保对象身份（AVM1 `===` 按引用、`InventoryIcon.collection === _root.物品栏.背包` 是生产先例），故 lease 可按 BaseItem 引用锁；但公开排序入口 `ItemSortUtil.sortInventory` 必带 `mergeStackables → setItems → createFromObject`，会**全量重铸所有物品对象身份**（不止被合并项），任何经 `setItems`/排序的路径后 lease 必须**整容器失效**，且 `Inventory.merge`（原地 addValue 保身份）与 `ItemSortUtil.mergeStackables`（铸新对象）语义相反，失效规则须按路径分。

### 1.2 「只对背包栏放行」约束（散点，迁移要收口）

当前散落在多处的源-集合判断，本质都是「只有背包能作为某类操作的来源」：

- 背包图标创建门控 `界面 != "物品栏" → return`：见 [物品栏UI.as](../scripts/展现/UI交互/UI交互_lsy_物品栏UI.as)，以该完整条件搜索定位。

- 样品栏只收背包来的物品：见 [物品栏UI.as](../scripts/展现/UI交互/UI交互_lsy_物品栏UI.as)，定位锚点 `只能从背包拖拽物品到样品栏`。
- 卸装/卸药剂必须背包有空位：`EquipmentIcon` / `DrugIcon` 的 `背包.getFirstVacancy()`

迁移目标：把这些从散点 `if` **集中到 AS2 领域服务的策略层**（一处持有、最终裁决）。Snapshot 可附带"当前 capabilities"供 Web 展示与预检（灰显/提示），但**Web/Broker 的能力声明只是 UX 预检，不是权限 SOT**——真实放行与否由 AS2 在执行命令时重新裁决（见 §5、护栏 §7）。这样 Web 声明过期/错误时不会导致非法跨容器操作。

### 1.3 已建立的 Web Panel 迁移范式（迁移底座）

merc / pet / arena / kshop 已迁移，全部是 **snapshot + command** 模型：开面板 AS2 下发只读 snapshot；用户动作发离散命令；AS2 改权威态，按既有业务契约 `dirtyMark` 或显式强制存盘，再回包。**没有任何已迁移面板做过两个活动库之间的连续跨库转移**——这正是物品系统的核心交互，也是本设计要新建的部分。

复用件：`PanelScale`（[panel-scale.js](../launcher/web/modules/panel-scale.js)；**通用 contain 等比缩放** `scale = min(w/designW, h/designH)`、两轴取小保形不裁切，**非写死 16:9**——画布 1024×576 与 16:9 anchor 同比时无损铺满，异比画布会留 letterbox，后续非 16:9 的 View 须知此语义）；双层 callId 桥（`PanelBridge.BuildFlashCommand` + 保留键守卫）；`Panels.register`；PanelHost open/close 生命周期。

## 2. 核心抽象：双栏工作台

把「物品面板」升维为**通用双栏工作台**：双栏只是两个 view-slot，背包/技能/仓库/商店都是可插拔 **View**；物品界面、技能界面只是不同的 View 配置。

```
DualPaneShell（壳：双 view-slot + PanelScale 缩放 + 换栏 + 冲突原子回滚 + 焦点）
   ├─ slotL ─┐
   ├─ slotR ─┴─ 各挂一个 WorkbenchView（任一槽皆可换；壳只认 View 生命周期接口）
   │
   ├─ InteractionBroker（仅交互撮合：Offer × Accept → 中性 OperationIntent；不知 AS2 命令名/timeout/dirtyMark）
   │
   └─ DomainCoordinator（领域命令 + 并发；OperationIntent → 命令/单飞/对账，AS2 最终裁决）
         ├─ InventoryCoordinator   背包/仓库/战备箱；持 backpack write gate
         └─ ShopCoordinator        saveCart/checkout/claim 单飞 + needs_reconcile

WorkbenchView（可插拔；契约=mount/unmount + render/subscribe + exportOffer(hit) + probeAccept(offer,hit)）
   ├─ CatalogView      商店目录（非容器：购买意图的来源）
   ├─ GridContainerView  ← 网格容器 primitive（见 §6；拆 GridRenderer / ContainerViewAdapter / ViewChrome）
   │     背包 / 仓库 / 战备箱 / 资源箱 / 购物车 / 已购买 全是它的配置（按 binding 区分实例）
   ├─ EquipView        装备槽 + 装备属性
   ├─ SkillView        技能栏
   ├─ EnhanceView      强化 / 强化度转换 / 插件改装
   └─ CraftView        合成
```

**第一性原则（四层职责，2026-06-20 按评审改判）**：**Shell 只认识 WorkbenchView 生命周期契约**（mount/unmount/render/subscribe + slot 元数据；mount/unmount 不绕 Broker）；**跨 View 交互只认识 Offer/Accept**——发起方 `exportOffer` 产出**不含 AS2 命令名**的交互报价，落点 `probeAccept` 返回是否接受 + UX hint，InteractionBroker 据此生成中性 `OperationIntent`；**领域命令与并发只存在于各 DomainCoordinator**——`InventoryCoordinator` 结合当前目标态决定发 `move`/`merge`/`swap`、`ShopCoordinator` 处理购物车意图，AS2 再做最终裁决。原"壳与 View 互不知道对方、只通过 Broker 能力契约对话"的窄表述已被本四层取代：否则 Broker 会同时认识 `shopSaveCart/inventoryMove/equip/claim/sell/craft`，N² 耦合只是从 View 之间搬进一个中央 switch、并未消失。

最小接口（命名空间隔离领域命令）：

```js
// View 产出的交互报价——不含 AS2 命令名
{ subjectKind: "ownedItem"|"catalogEntry"|"purchaseReceipt",
  sourceRef: { containerId, slot, lease },
  offeredOperations: ["inventory.transfer"] }
// 目标 View 的探测结果
{ accepted: true, operationId: "inventory.transfer",
  targetRef: { containerId, slot, lease }, hint: "merge" }   // hint 仅 UX，非最终命令
// Broker 生成的中性领域意图
{ operationId: "inventory.transfer", sourceRef, targetRef }  // 由 InventoryCoordinator 落 move/merge/swap
```

Catalog→Cart 走 `operationId: "shop.addCartIntent"`（ShopCoordinator 处理），只借用同一套手势撮合，**不被包装成 inventory transfer**。代码核查确认：`DualPaneShell / WorkbenchView / InteractionBroker / DomainCoordinator / GridContainerView / TransferBroker` 在全仓零既有同名符号，是纯空地，本次拆分/改名零迁移成本。

**视图单例不变量**（壳职责，**按 binding key 而非组件类型判定**）：单例性按 `instanceKey`（如 `inventory:背包`、`shop:cart`）+ `instancePolicy:"singletonByBinding"` 判定，**不是**按构造函数 / View class / 笼统"同一个 View"——`BackpackView` 与 `WarehouseView` 都基于同一个 `GridContainerView` 类却显然可并存（绑定不同容器），只有同一 binding（如双购物车）才排他。同一 `instanceKey` 默认不允许同时挂两个槽；用户在 A 槽选了一个已在 B 槽显示的 instanceKey 时，壳原子解冲突：先把目标移入 A；B 优先接收 A 原先的 View（仅当 B 允许且不会再次冲突），否则回落到 B 的结构默认 View；若回落仍冲突或不可用，则拒绝本次切换并保持原状态。该规则取代任何"避免双购物车"之类的视图专属状态机，并确保 displaced 槽在后续多视图组合中仍有确定去向。

## 3. 两种转移模态（关键区分）

| 模态 | 例子 | 特征 | 发出的命令 |
|---|---|---|---|
| **①a 同质 owned 转移** | 背包↔仓库/战备箱 | 即时、逐件、落点即提交 | `inventoryMove`/`inventoryMerge`/`inventorySwap`（§4.1，仅稳定同质 owned 容器；资源箱瞬态、暂排除，见 §4.3/§6） |
| **①b 显式领域动作** | 装备/卸装、领取(claim)、丢弃(discard)、出售 | 即时，但**含专属业务逻辑**（装备=等级检查+外观刷新；claim=移除待领记录+入包+强制存盘；discard=确认后从指定 owned 槽删除；出售=定价+金钱） | 各自显式命令 `equip`/`unequip`/`claim`/`inventoryDiscard`/`sell`，**不是**通用转移 |
| **② 暂存转移** | 商城购买（目录→购物车→结账→已购买→claim）、将来合成/批量出售 | 累积进缓冲 → 单次 commit → 常带代价（K点/金钱/配方） | view 专属 `checkout`/`craft` 批量提交 |

双栏语言**中心是模态①**；模态② 的 UI（购物车/结算）**保持视图本地**，直到第二个视图（合成/批量出售）真的需要同款暂存，再机械式抽取到壳级 staging 托盘。命名好抽取点，但不提前付泛化的钱。

## 4. 权威与命令模型

- **AS2 永远是权威态**，Web 只是投影。沿用 snapshot + command；新增 inventory-domain 写入成功后必须 `dirtyMark`，既有领域动作保留其明确的持久化契约（如 KShop claim 的强制存盘）。写操作必须由 AS2 权威路径重新校验价格/权限/槽位/状态（见迁移护栏 §7）；Web 端禁用按钮只是 UX，不是安全校验。

### 4.1 命令分层（修正 itemTransfer 过宽）

不存在一个吃下一切的 `itemTransfer`。命令按模态分层：

- **模态①a 同质 owned 转移**——仅用于 owned↔owned 的稳定同质容器（背包/仓库/战备箱）：`inventoryMove` / `inventoryMerge` / `inventorySwap`。（资源箱是瞬态容器，暂不入此列，见 §6。）
- **模态①b 显式领域命令**——`equip` / `unequip` / `claim` / `inventoryDiscard` / `sell` / `enhance` 等各自携带专属业务（等级检查、外观刷新、移除待领记录、确认后删除、强制存盘、定价、强化），**不得**塞进通用转移。`inventoryDiscard` 是首战“腾格子”的明确写命令，不复刻 Flash UI hitTest 路径。
- **模态①c 整容器写**——`inventorySortAndMerge`（真实重排槽位 + 合并堆叠 + dirtyMark）：既非逐件转移、非领域动作、非批量提交，而是整容器重组写，亦走本节信封；与 Web 本地 `displaySort` 严格区分（§6）。既有算法见 [ItemSortUtil.as](../scripts/类定义/org/flashNight/arki/item/ItemSortUtil.as)，定位锚点 `public static function sortInventory(`；**该工具本身不设置 `dirtyMark`，新领域 handler 必须在成功后显式设置**。
- **模态② 批量提交**——`checkout` / `craft`。

**命令信封（显式版本化）**。信封要求**仅适用于新增 inventory-domain 及后续模态① owned 命令**；既有 KShop `shop*` 命令是模态② view 专属命令，首战**不强制套信封、不改其形状**（见末尾"信封适用范围"）。Web 外层桥接信封与 inventory 领域负载必须分层，避免 `cmd`/`op` 双重真相；版本兼容通过 `payload.v` 明确迁移，不靠未知字段静默透传。

```js
{
  type: "panel",
  panel: "kshop",             // 发起面板，仅用于 Web 生命周期/诊断
  domain: "inventory",        // 领域路由
  cmd: "discard",             // InventoryTask 白名单键
  callId: "...",
  payload: {
    v: 1,                     // 领域负载版本
    source: { containerId, slot, expectedLease } // slot lease（不透明 token）；containerEpoch 可选携带
  }
}
```

InventoryTask 下发给 AS2 时去掉 Web 路由层并展开已验证 payload，例如：

```js
{ task: "cmd", action: "inventoryDiscard", callId: 17, v: 1, source: {...} }
```

关键约束：

- **可信 action 只由 C# 白名单映射产生**：`snapshot→inventorySnapshot`、`discard→inventoryDiscard`、`move→inventoryMove`、`merge→inventoryMerge`、`swap→inventorySwap`、`sortAndMerge→inventorySortAndMerge`。`InventoryTask` 校验 `domain/cmd/payload.v`，按下表为每个 v1 cmd **递归新建仅含白名单字段的 normalized payload**，再调用 `PanelBridge.BuildFlashCommand`；不得把原始 Web 消息或原始 payload 直接透传。这样 `op/action/task/cmd/domain/panel/callId` 及未知字段均不会下发。未知 cmd 回 `unsupported_cmd`，未知版本回 `unsupported_version`，字段非法回 `invalid_payload`。保留键守卫见 [PanelBridge.cs](../launcher/src/Tasks/PanelBridge.cs)，定位锚点 `private static readonly HashSet<string> Reserved`。
- **`containerId` 是稳定的领域容器标识**（如 `背包`/`仓库`/`战备箱`），**不是可换栏的展示槽（slotL/slotR）、View 实例或可拼接的 AS2 属性路径**。Web 换栏不影响命令寻址；AS2 领域注册表显式白名单解析 containerId，未知值回 `unsupported_container`。首战只向 inventory domain 暴露 `背包`。
- **并发防线 = AS2 snapshot-local slot lease（按命令前置条件），不是"全字段相等"**（2026-06-20 按评审改判，废除原"全量 canonical descriptor 现算"方案）。要防的是 **stale intent / 地址前置条件**——用户操作快照里的槽位 A，命令抵达时 A 已被换成别的东西、或目标槽状态已变——而**不是**"整个物品存储态逐字段完全相等"。后者语义过度（耐久、`lastUpdate` 等不影响寻址的字段变化会被误判 `stale`），且在 AVM1 上要递归遍历 + 键排序 + 对象分配 + 字符串膨胀（1200 格仓库放大成数十~数百 ms 定格、易触发 Flash 15s 脚本超时）。改用 AS2 铸造的有状态 lease；跨容器 move/merge/swap 的 source 与 target **都**携 lease，由 AS2 在**同一临界区**内一并比对、通过才执行，任一不符回 `stale_state`、Web 重拉相关 snapshot。
  - **lease 形态**：AS2 维护有限生命周期记录 `session/openGeneration → container generation → slot → {occupantRef, kind, count/mergeKey, confirmProjection}`；token 只含 `sessionNonce + containerGeneration + slot`，对 Web 不透明、只缓存回显（禁止 Web 自算，避免跨语言规范化差异进并发判断）。AS2 友好存储 = 按 generation 存平行数组（`refs[]/counts[]/keys[]/confirm[]`），不建复杂哈希表。
  - **O(1) 校验**：对象物品 `currentItem === lease.occupantRef`；primitive/dict stack `currentValue == lease.expectedValue`；空槽 `currentItem == null`。直接发现：槽被另一物品替换 / 物品被移走 / 目标空槽被占 / source 被删 / target stack 被换。同一物品对象内部字段变化时，再按下表具体命令检查相关字段，不做全字段比较。
  - **lease 失效边界（代码核查 2026-06-20 补强，落地必须守住）**：lease 按 **BaseItem 对象引用**锁——`add/getItem/纯 rebuildOrder` 全程零克隆、保身份，AVM1 `===` 按引用相等且 `InventoryIcon.collection === _root.物品栏.背包` 是生产先例。但公开排序入口 [ItemSortUtil.as](../scripts/类定义/org/flashNight/arki/item/ItemSortUtil.as) 的 `sortInventory` 必带 `mergeStackables → rebuildInventory → ArrayInventory.setItems → super.setItems → ItemCollection.setItems → BaseItem.createFromObject`（见 [ArrayInventory.as](../scripts/类定义/org/flashNight/arki/item/itemCollection/ArrayInventory.as) 与 [ItemCollection.as](../scripts/类定义/org/flashNight/arki/item/itemCollection/ItemCollection.as)），会**全量重铸所有物品对象身份（不止被合并项）**→ 任何经 `setItems`/排序的路径后，受影响容器 lease **整体失效**（失效钩子挂 `sortInventory/mergeStackables`；挂纯 `rebuildOrder` 只够应对 key 失配、不够身份失配）。注意两条 merge 语义相反：`Inventory.merge` 走 `addValue` 原地改值、目标身份**保持**；`ItemSortUtil.mergeStackables` 铸新对象、身份**替换**。又：`createFromObject` 重铸 BaseItem 外壳却**复用同一 `value` 子对象引用**，故 lease 必须锁 BaseItem 外壳、**不能锁 `value` 子对象**（否则 setItems 后假阳性命中）。
  - **按命令定义 stale 前置条件**（取代"所有写命令都比完整 descriptor"）：

    | 命令 | 前置条件 |
    |---|---|
    | `discard` | source occupant lease + 用户确认投影一致 |
    | `move` | source occupant lease + target occupancy lease |
    | `swap` | source、target occupant lease |
    | `merge` | source/target lease + mergeKey + 两端 count |
    | `equip` | source lease；等级/use/槽位规则现场重判 |
    | `sortAndMerge` | **通常无需 stale token**；对当前权威容器执行（客户端只上送 `methodName`、不上送待应用布局）。最多可选携 `containerEpoch` 做结构失效，**勿**因"玩家刚捡到一个物品后点整理"误判 `stale_state` |
    | `claim` | purchased-list token 或稳定 claimId（**不是** inventory fingerprint，见 §9.1/§9.2） |
    | `snapshot` | 无前置条件 |

  - **`confirmProjection` 显式版本化**：只含"若变化必须让用户重新确认"的字段——`{ itemKind, displayName, quantity, enhancementLevel, rarity }`，**不含** `lastUpdate` 等用户没看到的内部字段（确认框没展示的内部变更不应自动使确认失效）。
  - **拆开被一个 `expectedFingerprint` 混在一起的四种职责**：`snapshotSeq`（仅供 Web 忽略倒序回包）/ `slotLease`（验证 source/target 仍是快照中占用者）/ `containerEpoch`（可选，结构性变更与整容器缓存失效）/ 未来 `instanceId`（跨 snapshot、跨 session、可持久化的物品身份）。协议字段 `expectedFingerprint` 统一改名 **`expectedLease`/`slotLease`**。v1 不引入 `instanceId`；旧"完整存储态相同视为可互换"的身份语义被 lease 的"按运行态对象引用区分"取代（更严格，但只多一次 stale、不会误操作，且无需现在决定持久化 instanceId）。
  - **canonical encoder 降级**：从 inventory-domain 全局基础设施降为 **诊断工具 / 测试 oracle / 少数确需 exact-storage-equality 的命令专用实现**；其存储健康检查（`undefined`/函数/循环引用/NaN/Infinity）应放在加载、存档验证或诊断工具，**不应顺带成为每个 UI 命令的前置条件**（否则仓库里有一件遗留脏数据就能禁掉整个 `sortAndMerge`）。若某命令坚持无状态 token，次优方案是 operation-specific digest：`hash(versionTag, kind, identityFields, operationRelevantFields)`，固定字段序 + 长度前缀 + 流式计算 + 只传定长摘要，不构造完整 canonical 字符串、不遍历无关字段；因 Web 不算 token，这里不存在跨语言 canonical JSON 必要性。`BaseItem.toObject()`（锚点 `public function toObject():Object`）仍是物品级存储投影的来源，但只在需要 exact-equality 的诊断/测试路径用，不进每命令热路径。
  - **token 铸造边界**：所有 lease/digest token 均由 AS2 snapshot 生成，Web 只缓存并原样回显。
- **AS2 最终裁决**：版本、容器/槽位、token 以及所有业务合法性（容量、use 匹配、背包-only、金钱/K点、等级）都在执行时由 AS2 领域服务重判；C# 结构校验与 Web capabilities 仅是前置拒绝/UX 预检。

**v1 inventory cmd 负载形状**：

| Web cmd | `payload` 必需字段 | 并发覆盖 | 成功结果 |
|---|---|---|---|
| `snapshot` | `{v, containerIds}`（窗口化形态见下注） | 无；只读 | 请求容器的 snapshot + AS2 铸造 lease/snapshotSeq |
| `discard` | `{v, source}` | source 槽 | 刷新后的背包 snapshot |
| `move` / `merge` | `{v, source, target}`；v1 仅整槽移动/整组堆叠合并，拒绝 `count` | source + target 槽 | 受影响容器 snapshot |
| `swap` | `{v, source, target}`；整槽互换，拒绝 `count` | source + target 槽 | 受影响容器 snapshot |
| `sortAndMerge` | `{v, container, methodName}` | 完整容器 | 刷新后的容器 snapshot |

`methodName` 在 v1 必填，并由 AS2 领域 handler 对显式策略白名单做严格校验；未知值回 `unsupported_sort_method` 且不得写入。既有 `ItemSortUtil` 的 `private static function validateSortMethod` 会把未知值静默回退到默认策略（实测：未知/空 method 静默回退 `byType`，其"合法集"是 `STRATEGY_CHAINS` 八个键 `byType/byUse/byPrice/byLevel/byID/byName/byValue/byTime`、并非协议白名单），只能作为当前行为定位点，**不能直接充当协议验收器**；handler 校验通过后才可调用既有排序工具。每个 inventory-domain 写命令都必须在同一权威处理路径中完成“比对 lease → 重判业务 → mutate → dirtyMark → 生成回包”，不得把校验和写入拆到可被其他事件插入的两段。

**snapshot 窗口化（可选增量，非首战必做）**：即使废了 canonical，若后续每次 move 都重发完整 1200 格仓库，性能问题仍在。建议把 `snapshot` 协议形态留出窗口位——请求 `{v, requests:[{containerId, offset, limit}]}`、回包 `{containerId, capacity, snapshotSeq, offset, slots}`；写成功后 move/merge/swap 只回受影响槽位或当前可见窗口、`sortAndMerge` 使全部窗口失效但只重拉当前页、翻页按需请求，**不把 1200 格容器当成一个不可分割 snapshot**。现状距离：web 协议从未对 owned 容器做 range 快照（merc 的 `page/perPage` 是 list 分页、pet/arena `handleSnapshot` 全量无 offset/limit），属 **net-new 协议层**。首战背包仅 50 格不急；**建议现在就采纳 range-capable 的协议形状（别把 snapshot 设计成整容器不可分割），实现推迟到仓库步（§8 Step 4/5）**。

**原子性不能靠“AS2 单线程”想当然**：现有 [Inventory.as](../scripts/类定义/org/flashNight/arki/item/itemCollection/Inventory.as) 的 `move/merge/swap` 是多步便利方法，过程中 `add/remove/addValue` 会同步发布生命周期事件，`swap` 也未逐步检查失败并回滚；事件回调是可重入点，因此这些方法本身不是 inventory-domain 的事务边界。新领域 handler 必须先完整预检两端，在事务重入守卫下完成无部分失败的状态提交，再统一发布事件、`dirtyMark` 与 snapshot；若不能保证提交成功，必须保持两端原状。禁止把“直接调用现有 move/merge/swap”当作原子性证明。部分堆叠拆分会引入数量、时间戳、回滚与确认语义，留给后续显式协议版本，v1 拒绝 `count`。

白名单按迁移阶段启用：首战只开放 `snapshot` 与 `discard`，其余 cmd 在对应 AS2 handler、闭环表和测试全部落地前均回 `unsupported_cmd`。C# 只负责结构/版本/action 白名单；containerId、槽位、token 与业务合法性仍由 AS2 最终裁决。

**信封适用范围**（解决"统一信封"与"首战保留 KShop 字段形状"的冲突）：信封 + 双端 token 是**新命令族（inventory-domain、后续模态①）的契约**；KShop 的 `shopBulkQuery/saveCart/checkout/claim` 首战保持既有请求/回包字段形状、不回填信封，但其写命令新增 §9 所述 ShopTask 单飞与 `busy/reconcile_required` 前置拒绝行为。若日后要给 shop* 套信封，单列为协议迁移项。

#### 4.1.1 equip / unequip（模态①b）：槽寻址、payload 与活体副作用（2026-06-21 补，源 玩家信息界面 HUD 迁移交汇）

equip/unequip 现 doc 此前只有 §3 模态①b 一行 + 「各自携带专属业务」一句，属真空缺口；首战（KShop）/ Step3-4 都不碰 equip（§8 留到 Step5 装备槽 / Step6 技能·药剂快捷栏），故此处先补契约占位，等真正排到 Step5/6 再提进白名单、并按 §4.4 结论回填锚点。**装/卸是 owned 主库（背包）↔ 槽位化目标（装备栏 / 药剂栏 / 快捷技能栏）的领域写，不是同质转移**，故 payload 在 source lease 之外**必须显式携带 target 容器 + 槽号**（「装到几号槽」），不靠落点 hitTest 反推：

```js
// equip：背包某槽 → 目标槽位
{ task:"cmd", action:"equip", callId, v:1,
  source:{ containerId:"背包", slot, expectedLease },
  target:{ containerId, slot } }   // containerId∈{装备栏,药剂栏,快捷技能栏}
// unequip：目标槽位 → 背包（背包落点由 AS2 取 getFirstVacancy，不由 Web 指定）
{ task:"cmd", action:"unequip", callId, v:1,
  source:{ containerId, slot, expectedLease } }
```

| target containerId | slot 语义 | AS2 落点（一次性快照锚点） |
|---|---|---|
| `装备栏` | 装备类型**字符串键**（EquipmentInventory，键=use/装备类型，**非数字**） | `ItemUtil.moveItemToEquipment`：`index!=equipmentIcon.index` 拒、等级门 `data.level>_root.等级` 拒 |
| `药剂栏` | 数字索引 0..3（DrugInventory(4)） | `ItemUtil.moveItemToDrug`：冷却门 `!isCoolDown() → return false` |
| `快捷技能栏` | 数字槽号 1..12（无集合，写 `快捷技能界面.快捷技能栏N`） | 见下「活体副作用」+ §4.4（出图载体双 symbol） |

**槽寻址与并发约束**：
- `target.containerId` 由 AS2 领域注册表**显式白名单**解析，未知值回 `unsupported_container`；首战不开放 equip。
- **slot 必须显式**：v1 `equip` 拒绝「未带 target.slot 让 AS2 猜空位」；装备栏 slot 是**类型字符串键**不是数字（与背包/仓库的数字 physicalSlot 不同栏寻址，渲染绑定须分别保留）。`unequip` 落背包由 AS2 `背包.getFirstVacancy()` 定，不由 Web 指定；满则回 `inventory_full` 明确提示（沿用 EquipmentIcon/DrugIcon 现状），不静默失败。
- **并发**：`equip` 携 source lease；target 槽是否被占由 AS2 现场判（占用即按目标栏规则 swap 或拒绝，**不复用 inventoryMove 的 target occupancy lease**——装备栏/药剂栏/快捷技能栏是领域槽非同质 owned）。

**药剂 unequip 的两条非显式路径（无叉按钮，迁移须各自等价复刻，不可套通用 unequip）**：
1. **手动卸 = 冷却态点击（不是叉）**：`DrugIcon.Press` 仅在 `!locked && isCoolDown()`（药剂处于冷却中）才允许卸——`if (locked || !isCoolDown()) return`，再 `collection.move(背包,...)` 回背包 + 清镜像 `_root["快捷物品栏"+index]=""`。故 Web `unequip(药剂栏,slot)` **必须带冷却门控**（非冷却态拒绝，由 AS2 现场读 `药剂进度条.冷却` 重判，Web capabilities 仅作 UX 灰显）。
2. **耗尽自动卸**：药剂用尽时 `药剂控制器` 自动清栏（`addValue(index,-1)` 后 `if(!icon.item)` → `_root[控制参数]=""` + 「耗尽！」）。这是**战斗内事件驱动清栏**（§3 模态之外），不是 Web 命令；Web 侧药剂 snapshot 用药/耗尽后须被动重投影，**Web 不得对耗尽清栏发 unequip**（会与自动清栏竞态）。

**equip/unequip 活体副作用清单（AS2 必做、Web 不可代偿）**——装/卸不是纯数据搬移，落地后触发改在场战斗单位的派生态，全部留 AS2 重做：

| 目标栏 | 装/卸必做的 AS2 活体副作用 | 锚点（一次性快照，须按 §4.4 激活份校正） |
|---|---|---|
| **快捷技能栏** | 写 `主角技能表[i][2]/[4]`（装 true/卸 false）→ `_root.更新主角被动技能()` → 回写 `_root.gameworld[_root.控制目标].被动技能 = _root.主角被动技能` → `_root.排列技能图标()` | 装=技能图标.xml（新结构）；卸=快捷技能界面.xml `取消装备技能`（**仅被动技能改时才重算**） |
| **装备栏** | `_root.刷新人物装扮(_root.控制目标)`（外观）+ 音效 + `dirtyMark` | ItemUtil（equip）/ EquipmentIcon（unequip） |
| **药剂栏** | `物品栏.药剂栏` move/addValue 改权威 + 同步 `快捷物品栏N` 镜像 + `dirtyMark` | ItemUtil.moveItemToDrug / DrugIcon / 药剂控制器 |

- **技能装/卸的「活体」核心 = 被动技能重算并回写战斗单位**（`更新主角被动技能()` + `gameworld[控制目标].被动技能`），直接影响在场角色战斗数值，**绝不能由 Web 算或跳过**；且**卸=仅当 `[3].indexOf("被动")>-1` 才重算、装=无条件重算**，迁移须保留这条不对称。
- **dirtyMark 责任**：装备/药剂路径 AS2 既有代码已 `dirtyMark`；**技能装/卸路径当前不显式 dirtyMark（真实缺口）**，新 handler 须在成功后补标脏（对齐 §6 纪律）。
- **重投影范围**：equip 影响 source（背包）+ target 栏 + 派生态（被动/装扮），Web 须重拉受影响容器 snapshot + HUD 派生字段，不局部 patch。

### 4.2 渐进

- **v1 用 round-trip 确认**（快车道低毫秒级，够用）；乐观更新 + snapshot 对账作为第二步增量，不在首版背回滚复杂度。
- **重投影**：AS2 mutate 后，受影响 collection（携当前 `snapshotSeq` + 新铸 slot lease）重新下发 snapshot 片段，Web 重渲染；写成功必失效该容器旧 lease（见 §4.1 失效边界）。
- **写命令不自动重试**：已下发写命令遇 timeout/断连/未知结果时，先拉权威 snapshot 对账；只有确认状态后，才允许用户重新发起新意图。禁止网络层静默重放 move/swap/discard/checkout 等非幂等写。`saveCart` 这类完整期望态覆写可在对账后按**当前最新期望值**构造一条新同步，但同样不得重放旧请求。

## 5. View 契约

每个 View 向壳/Broker 声明四件事：

1. **数据需求**：订阅哪些 snapshot 键（如背包订阅 `inventory.背包`）。
2. **作为 source 的能力（UX 预检）**：能"吐出"什么。例：`BackpackView` 声明自己可作 equip/sell/enhance/move 的来源——用于灰显/提示。**注意：这是展示层投影，不是权限 SOT**；"背包-only"等真实约束由 AS2 领域服务持有并最终裁决（§1.2），Web 能力来自 snapshot 下发的 capabilities。
3. **作为 sink 的能力（UX 预检）**：能接收什么 + 落点提示。例：`EquipmentSlotView` 预检 `use` 匹配；`CartView` 只收目录来的购买意图。同样仅预检。
4. **渲染与局部交互**：自身网格/列表/页脚，以及不经 Broker 的本地动作（翻页、displaySort/filter、tooltip）。

上述四件声明对应 §2 四层接口：第 2/3 项的 source/sink 能力即 View 的 `exportOffer(hit)`/`probeAccept(offer,hit)` 产出（不含 AS2 命令名）。**InteractionBroker 职责仅交互撮合**：拖拽/点击发起时用 Offer × Accept 做**本地预检**（不合则不发、给即时反馈），通过则生成中性 `OperationIntent`（如 `inventory.transfer`/`shop.addCartIntent`），**交由对应 DomainCoordinator 决定落 `move`/`merge`/`swap` 或 shop 命令**——Broker 自身不认识 AS2 命令名（否则退化成同时认识 `shopSaveCart/inventoryMove/equip/claim/sell/craft` 的中央 switch）。模态分层（①a/①b/②）由 Coordinator 据 OperationIntent + 当前目标态裁定。**最终合法性永远由 AS2 重判**；Broker/Coordinator 都不是权限或业务规则 SOT。Web capabilities 过期时，AS2 拒绝并回错误码，Web 重拉 snapshot。

## 6. GridContainerView primitive（复用基石）

一个网格容器组件，参数化三件：

- **item-model**：`owned`（真实拥有实例）/ `intent`（购买意图 `{catalogIdx, qty}`）/ `owned-待领`。
- **itemActions（逐格）**：单格出口/编辑动作。如 owned 格 = `move`/`equip`/`claim`/`discard`；intent 格（购物车）= `changeQty`/`remove`（走 `saveCart`，**非** checkout）。
- **viewCommitActions（整 View 批量）**：模态② 的整体提交。如购物车 = `checkout`、合成 = `craft`。**勿与逐格动作混为一谈**：逐格改的是缓冲内容，commit 才是带代价的整体结算。
- **sink-accept**：作为落点接受什么。

通用能力：网格渲染 + 拖拽/点击 + 筛选 + 批量动作 + 分页。

**避免 God-component（评审，2026-06-20）**：`GridContainerView` 同时担渲染 / item-model / itemActions / sink-accept / 批量 / 分页 / filter-sort / 拖拽点击，职责过载。落地至少拆三层——`GridRenderer`（DOM/虚拟化/physicalSlot/命中测试）、`ContainerViewAdapter`（item 投影 + `exportOffer`/`probeAccept` + 领域动作）、`ViewChrome`（filter/分页/页脚/commit bar）。`checkout` 等 `viewCommitActions` **不是 Grid primitive 的通用职责**，而是 `CartView` 的组合部件（commit bar 挂 ViewChrome、逻辑归 ShopCoordinator）。配套 code-review 硬门禁：`DualPaneShell` 内不得出现 `saveCart/checkout/claim`；`InteractionBroker` 内不得出现 `ShopTask/InventoryTask/callId`；`GridRenderer` 不得持 pending 表。

**排序必须拆成两个语义，勿混**（现有 `sortInventory` = `rebuildOrder` 重排槽位 + `mergeStackables` 合并堆叠，是权威写；见 [ItemSortUtil.as](../scripts/类定义/org/flashNight/arki/item/ItemSortUtil.as)，定位锚点 `public static function sortInventory(`）：

- **`displaySort` / `filter`**：Web 本地展示重排/过滤，**不改真实槽位**，无命令、无存档。
- **`inventorySortAndMerge`**：AS2 权威写命令（真实重排槽位 + 合并堆叠 + dirtyMark），走 §4.1 信封。Web 端"整理"按钮触发的是这个，不是 displaySort。**代码核查（2026-06-20）**：`ItemSortUtil` 全文不设 `dirtyMark`，新 handler 必须在成功后负责标脏并由测试断言；且 handler 须先对 `methodName` 做显式白名单校验——`validateSortMethod` 会把未知/空 method 静默回退 `byType`（其合法集是 `STRATEGY_CHAINS` 八键、非协议验收器，见 §4.1）。

每个渲染项必须始终携带 snapshot 下发的权威地址 `{containerId, physicalSlot}` 及对应 **slot lease**；`displaySort/filter` 只改变展示顺序/可见性，不得把 DOM 序号或过滤后的可见序号写回为 physicalSlot。点击、拖拽、丢弃和落点命令均从绑定模型读取权威地址，绝不从当前网格位置反推，否则本地排序后会操作错误槽位。lease 按对象引用绑定（§4.1），排序/`setItems` 后整容器 lease 失效、须重绑。

### 纠正线（必须守住，否则留债）

**只在视图/交互层统一，数据/所有权 + 出口命令按实例保持区分。**

仓库格子里是**已拥有的真实物品实例**；购物车格子里是**购买意图**（尚未拥有）。两者长得像、交互像，但语义与出口命令不同。"购物车是仓库的变体"——**是组件复用意义上的变体，不是数据所有权意义上的变体**。绝不可把购物车内容物建模成 owned 槽位，否则把"购买意图"语义漏进所有权模型。

### 容器分类法

| 容器实例 | item-model | itemActions（逐格） | viewCommit | sink-accept |
|---|---|---|---|---|
| 背包 | owned + 主库特权 | equip/sell/enhance/move/discard | — | move / claim / 购买结果 |
| 仓库 / 战备箱 | owned（容量/分页规则） | move | — | 背包来的 owned |
| **购物车** | **intent** | changeQty/remove（→`saveCart`） | **checkout** | 目录来的"加购意图" |
| **已购买** | owned-待领 / 回执 | claim→背包（模态①b）/ 只读 | — | — |

战备箱页数派生见 [物品栏UI.as](../scripts/展现/UI交互/UI交互_lsy_物品栏UI.as)，定位锚点 `_root.物品UI函数.计算战备箱总页数 = function`。

**资源箱暂排除**：资源箱库存数据在 [关卡系统_lsy_地图元件.as](../scripts/逻辑/关卡系统/关卡系统_lsy_地图元件.as) 的 `_root.地图元件.掉落物转换为物品栏 = function` 中瞬态 `new ArrayInventory(null, cap)`（cap=row×col）即时构造，**库存数据对象无稳定 containerId、不入任何领域注册表**（其 UI MovieClip 创建入口以 [物品栏UI.as](../scripts/展现/UI交互/UI交互_lsy_物品栏UI.as) 的 `_root.物品UI函数.创建资源箱图标 = function` 定位；UI 层虽有 `外部UI列表` + 场景转换时 `卸载外部UI()` 粗粒度清理，但数据对象本身无 per-container 寻址能力），不满足 §4.1 命令寻址前提。近期**不纳入**可寻址转移模型；将来纳入须设计 session-scoped containerId + 失效错误（容器已关闭时拒绝命令）+ 关闭清理。其视觉网格仍可复用 GridContainerView 渲染，但**不接入 Broker 转移**。

## 7. 交互模型：点击为主 + 拖拽加速

- **点击为主**：点选源格 → 点目标栏/格（或双击直送对侧、Ctrl 快移）。可靠、round-trip 语义干净、与既有命令模型同构。
- **拖拽加速**：pointer 式手写拖拽作为锦上添花，失败可降级回点击。

### CDP 可行性（已确认）

输入经 InputShield → CDP `Input.dispatchMouseEvent` 注入。**结论**：

- HTML5 原生拖放（`draggable`/`dragstart`/`drop`）在合成事件下**不触发**——全仓零使用，禁用路线。
- **pointer 式手写拖拽完全可用**，生产级证据：lockbox 的 `setPointerCapture`（[lockbox-panel.js](../launcher/web/modules/minigames/lockbox/lockbox-panel.js)）、jukebox 的滑块/进度条鼠标拖动模式（[jukebox.js](../launcher/web/modules/jukebox.js)）、map 的 `pointermove` 命中（[map-hotspot-hitcapture.js](../launcher/web/modules/map/map-hotspot-hitcapture.js)）。定位均直接搜索上述 API/事件名。
- 拖拽实现走 `pointerdown → setPointerCapture → pointermove(更新拖拽镜像) → pointerup → releasePointerCapture`；无 touch 事件（CDP 只模拟鼠标）。
- 注意 InputShield 在 hitRect 收缩后有 **800ms move 抑制窗口**（源码称「收起冷却」：hitRect 缩小后直接短暂禁止 CDP `mouseMoved`，防 mouseenter 重新展开振荡；见 [InputShieldForm.cs](../launcher/src/Guardian/InputShieldForm.cs)，定位锚点 `收起冷却：hitRect 缩小后短暂禁止 CDP mouseMoved`）。代码核查（2026-06-20）：**800 是硬编码字面量（无命名常量、无 ± 容差）；是全量抑制（直接 return）非限流；仅抑制 `mouseMoved`——`mousePressed/Released/wheel/leave` 不受此门控**。故开合 UI 时勿在拖拽中触发。
- **拖拽超时兜底（评审雷区）**：因冷却期抑制 move，若玩家开面板瞬间极快拖拽，前端 JS 会收到 `pointerdown`（捕获静止坐标）但**永远收不到 `pointermove`**。须给 `setPointerCapture` 加防抖/超时兜底：按下后一定时间内无 move 也无 up/cancel，强制销毁自绘镜像并 `releasePointerCapture`，避免满屏残留幽灵元素阻挡点击。

拖拽镜像**已定走 Web 自绘**（不续用 AS2 `_root.鼠标.物品图标容器`），与"AS2 UI 全面外迁"长期战略一致。

本节按机制描述 KShop 加重版，施工边界以 §8 为准：Step 2 只冻结 Shell/ViewHost/Grid/request mux/ShopCoordinator；Step 3 才引入 `InventoryTask` + 背包 `snapshot/discard` + claim 重投影；Step 4 背包↔仓库薄片才是 owned↔owned Broker/Gate A2 验收点。读本节时不要把所有机制压成同一个 PR。

### 9.0 现有 KShop 真实数据生命周期（确权，勿凭直觉）

现有 KShop（[kshop.js](../launcher/web/modules/kshop.js)）内部已是两区：目录网格 + sidebar（购物车 + 结账 + 已购买），固定画布 1024×576。其权威生命周期见 [商城系统_WebView.as](../scripts/逻辑系统分区/商城系统_WebView.as)：

| 阶段 | gameCommand | 改什么 | 注意 |
|---|---|---|---|
| 加载 | `shopBulkQuery` | 只读：目录 + `_root.商城购物车` + `_root.商城已购买物品` + K点(`_root.虚拟币`) + **玩家等级 `_root.等级` + 逆向被动等级 `_root.主角被动技能.逆向`** | 购物车有**服务端恢复影子**（供重开恢复），会话内意图主居 Web |
| 存购物车 | `shopSaveCart` | 写：用 `params.cart` 覆写恢复影子 `_root.商城购物车` | |
| 结账 | `shopCheckout` | 写：结算 **Web 上送的 `params.cart`**（**不读**持久影子）+ 扣 K点 + →`商城已购买物品` + 清空影子 + `强制存盘` | **不碰背包**；K点判定**严格 `_root.虚拟币 > total`**（非 `>=`）——恰好等额时回 `insufficient_kpoints` 失败，迁移须原样照搬勿静默改 |
| 领取 | `shopClaim` | 写：按 `params.purchasedIdx` 从 `商城已购买物品`→背包 + 强制存盘；**三条显式失败路径**：索引无效→`item_not_found`、背包无空位→`inventory_full`、`singleAcquire` 返回 false→`acquire_failed`；**回包只含 `purchased`，不含背包** | **真正入背包在此，非 checkout**；`purchasedIdx` 是会随前序 claim 删除而变化的位置索引，v1 必须单飞 |

上表四行均以 [商城系统_WebView.as](../scripts/逻辑系统分区/商城系统_WebView.as) 中对应 `_root.gameCommands["shopBulkQuery"]` / `shopSaveCart` / `shopCheckout` / `shopClaim` 为定位锚点。

`acquire_failed` 可按“明确未写入”处理的当前依据是 [ItemUtil.as](../scripts/类定义/org/flashNight/arki/item/ItemUtil.as) 的 `public static function acquire(itemArray:Array):Boolean`：先以 `require` 完整预检，失败直接返回，之后才进入 mutate。该顺序是 claim 错误语义的回归契约，重构时必须由测试守住。**2026-06-20 对抗复核坐实**：`require` 内 `getVacancies` 容量门不足即 `return null`，`acquire` 见 `list==null` 在任何 `_root` mutate（首个写 `_root.金钱+=`）之前 `return false`，`singleAcquire` 只是 `acquire([{name,value}])` 薄包装——故 `acquire_failed`/`inventory_full` 路径**原子无副作用**。外部评审"`singleAcquire` 可能部分写入、`acquire_failed` 需 `needs_reconcile` 强制对账"的质疑被代码**证伪**：KShop claim 收到这两个码后回 idle/释放锁是安全的，`needs_reconcile` 只留给 timeout/断连/畸形回包等真正结果不确定的情形（§9.1）。（残余风险仅 AS2 无 try/catch：若提交段运行时异常会停在中途且无回滚，但那属"运行时异常"而非"逻辑部分写入"，且容量预检已消除最常见溢出型中断。）

当前 `shopClaim` 在调用 `singleAcquire` 前先做统一 `getFirstVacancy()==-1` 物理空格悲观检查，因此即使物品理论上可合并到现有堆叠，满背包也会先回 `inventory_full`（拒绝时未调 `singleAcquire`、无副作用，属安全的过度拒绝）。首战按现状保留；若要放宽（满包仍允许领取可合并的可堆叠物品），须把前置检查从 `getFirstVacancy` 改为基于 `singleRequire` 的可行性预检，并作为**独立游戏行为变更评审**，不能在 UI 迁移中顺手改掉。

**checkout 边角（迁移须连带审视）**：`total` 仅累加通过 idx/qty 校验的条目，空购物车 `total=0` 时严格 `虚拟币>0` 对任何持正 K 点玩家为真，会走成功分支——扣 0 K 点、push 0 条已购、清空购物车、`record 商城结账次数+1` 并强制存盘。无 K 点经济偏差，但"空结账"会让成就计数 +1 + 触发一次存盘，前端改判定时勿误触发。

由此修正先前文档错误：① 购物车并非"纯 Web 本地"也非"纯服务端持久"——它是 **会话内 Web 主态 + saveCart 恢复影子**，且 **checkout 结算的是 Web 上送的 `params.cart` 而非读取影子**（二者可能不一致）；② **背包改变发生在 claim，不是 checkout**；③ "批量领取"语义属于**已购买 View**（owned-待领 → 背包），购物车（intent）的逐格动作是增删/改量，不是领取；④ **claim 回包不含背包**，背包刷新需另建通道（§9.1/§9.2）。

**首战不改这套生命周期**（仍是"请求 cart 结账"模型），只在其上叠加双栏壳 + 容器视图 + 背包并排；沿用 `shopBulkQuery/saveCart/checkout/claim` 原语义。**是否迁移为"AS2 持久 cart 结账"**（checkout 改读 `_root.商城购物车`）留作单独立项——两者的并发、恢复、失败语义不同，首战不动。

### 9.1 加重版叠加（保留既有 shop* 请求/回包形状）

- **槽位**：左∈{目录, 购物车} / 右∈{背包, 购物车}，视图单例不变量产三配置（§11）；默认进面板态 = (目录, 购物车)。
- **视图**：CatalogView=现有目录逻辑包成 View；CartView=GridContainerView(model=intent, itemActions=changeQty/remove→`saveCart`, viewCommit=`checkout`)；**已购买是独立容器**（model=owned-待领, itemActions=claim），v1 UI 可沿用现有 sidebar 把 购物车/已购买 两区并置于同一槽（CartView 复合），也可拆独立可换视图——布局自由，但**数据与 egress 必须分开**。
- **目录→购物车** = sink 接受"加购意图"（可拖可点，**理想的低风险拖拽沙盒**：错了零成本、不碰真实库存）→ 走 `shopSaveCart` 写恢复影子。
- **KShop 写协调（不可继续沿用各按钮各自 pending）**：`saveCart` 会覆写恢复影子，`checkout` 会扣 K 点、追加已购买并清空影子，`claim` 会删除已购买并入背包；三者必须共享一个 ShopTask/Web 写协调器。Web 对高频 `saveCart` 只保留“单个在途 + 一个最新期望 cart”的合并槽，不逐次排队；`checkout` 发出前必须等待 save pipeline 空闲并阻止新 save，避免旧 saveCart 在 checkout 清空影子后迟到回写。`checkout/claim` 均为非幂等写，只允许单飞且不自动重发。正常关闭也必须进入协调器：停止新编辑，等待当前 shop 写终止或完成对账，再只 flush 一次最新 cart 后发送 close；不得像当前 `requestClose()` 那样在已有写 pending 时并发补发 saveCart。强制关闭可跳过等待，但重开必须先对账。Host 在任一 shop 写 pending 时拒绝第二个 shop 写为 `busy`。
- **checkout 不确定结果对账**：当前 Web 仅用 `_checkingOut` 防同页双击，timeout 后会重新开放重试；这不足以防重复扣 K 点。已下发 checkout 遇 timeout/途中断连/未知或畸形回包时，保持结账与领取入口禁用，先成功 `shopBulkQuery`，以返回的 K 点、购物车影子与已购买列表重投影后才允许新意图，禁止直接重发原 checkout。明确 `insufficient_kpoints` 是当前唯一已列明的未写入失败，可直接结束该 checkout。发送前拒绝没有写入歧义。
- **claim 单飞 + 背包重投影（不可拆开）**：现有 AS2 用会漂移的 `purchasedIdx`，现有 [kshop.js](../launcher/web/modules/kshop.js) `function onClaim(e)` 只禁用被点按钮；若连续点击两行，首个 claim 删除数组元素后，第二个旧索引可能领取错误物品。v1 保留既有 shop* 请求/回包形状，但必须同时建立两层约束：共享 ShopTask 写 gate 在一个 shop 写 pending 时拒绝后续 claim；Web 设置**全局 claim single-flight 锁**并禁用全部领取入口，锁一直持有到 `shopClaim` 回包及其链式 `domain=inventory cmd=snapshot` 成功或终止失败。由于现有请求没有已购买列表 token/稳定 claim ID，Host 无法识别“首个响应后才送达”的陈旧 `purchasedIdx`；v1 正常路径依赖工作台 mux + Web 单飞杜绝此类延迟意图，不能把 Host pending gate 宣称为完整陈旧意图防线。**稳定 claim ID/已购买列表 token 由本次评审升级为 §9 明确必修项（非"若需则做"的可选项）**：代码核查证实 `shopClaim` 按 `purchasedIdx` 直接索引活数组并 `splice(claimIdx,1)`，删除后数组左移、旧索引若仍 `< 新 length` 会**静默领取相邻错误物品**（非越界报错，危害更大），`qty` 取 `item[length-1]` 也随之错配；即便加了 Web 全局单飞，Host 仍按位置盲取、新 callId 重发陈旧 `purchasedIdx` 即绕过 callId pending gate。**第二条漂移源（跨命令族）**：claim 在途期间若 checkout 回包刷新 `_purchased` → `renderClaimed` 全量重编号 `data-pidx`，在途 claim 携带的旧编号即漂移——故单飞锁在禁用全部领取入口期间**也须屏蔽 checkout 触发的已购买区重渲染**。该协议迁移建议在 Gate A2 之前或并行排期，不要拖到全部界面迁完。
- **claim 结果与刷新失败分开处理**：claim 成功后即视为不可重试的已提交写，再链式补发 inventory snapshot。若 snapshot 失败，提示“领取已成功但背包刷新失败”，只允许重试 snapshot，不得重发 claim；若**已下发 AS2** 的 claim timeout/途中断连或收到未知/畸形结果，保持领取入口锁定，先用 `shopBulkQuery` + inventory snapshot 对账，完成后才允许下一次 claim。发送前即因未连接被拒绝则没有写入歧义，可直接失败。`inventory_full`/`acquire_failed` 是当前代码的明确未写入失败，可不发 snapshot 并释放锁；`item_not_found` 也未写入，但表明已购买列表陈旧，须先成功 `shopBulkQuery` 再启用领取。面板初次打开/重开时，领取入口也必须等 `shopBulkQuery + inventory snapshot` 均成功后才启用，避免关闭面板丢失 Web 内部 refresh 状态。
- **背包满时领取的功能边界**：claim 需背包空位，满时**拦截+明确提示**，不静默失败（沿用 AS2 现有 `getFirstVacancy()==-1` 判定）。(购物车/已购买, 背包) 同屏即为此场景。v1 腾格子工具箱仅限背包自身 egress：至少提供经明确二次确认的 `inventoryDiscard`；卖出/挪仓库需对应目标/视图存在，留到 NPC 商店、仓库步骤。既有 Flash UI 丢弃路径可在 [InventoryIcon.as](../scripts/类定义/org/flashNight/arki/item/itemIcon/InventoryIcon.as) 以 `// 垃圾箱` 定位，但 Web 迁移必须走新增领域命令而非复用 UI hitTest。
- **v1 背包写串行化**：工作台层维护一个 backpack write gate；claim 从发出到链式 inventory snapshot/对账终止、discard 从发出到其刷新 snapshot 终止，均独占该 gate。其他背包写直接显示 busy、**不排队**；与当前 owner 无关的背包 snapshot 请求延后/合并到 gate 释放后，只有 owner 的刷新/对账 snapshot 可结束本轮写。关闭/强制关闭不能视为取消已下发写；若关闭时 gate 有 owner，清理 Web pending 后只忽略迟到回包，不重发、不推断结果，重开时所有背包写入口继续禁用，直到新的 inventory snapshot（claim 另加 bulkQuery）完成重投影。这样既避免陈旧意图延迟执行，也避免跨 ShopTask/InventoryTask 的回包倒序使 Web 投影回退。该 gate 只是首战投影顺序约束，不能替代 AS2 slot lease 与最终裁决；出现第二个可同时写背包的 Web 面板时，应提升为共享 inventory-domain 协调器或引入 AS2 铸造的 snapshot epoch，禁止复制互不知情的 panel-local gate。
- 拖拽镜像 Web 自绘（§7）。

Web 侧先以 shop 写协调器串行化 `saveCart/checkout/claim`；其中 claim 子状态机固定为：`idle → claim_pending → inventory_refresh_pending → idle`；`inventory_full/acquire_failed` 直接回 `idle`；`item_not_found` 进入 `shop_refresh_required`，成功 bulkQuery 后回 `idle`；claim timeout/途中断连/未知结果进入 `reconcile_required`，仅 `shopBulkQuery + inventory snapshot` 均成功后回 `idle`；claim 已成功但 snapshot 失败进入 `refresh_required`，仅重试 inventory snapshot，禁止重发 claim。Host 只维护请求级 shop 写 gate 与 `needs_reconcile`，不负责 inventory 刷新状态。

定位诚实声明：KShop 加重版是**壳 + GridContainerView + 模态② + claim 重投影 + discard 单向领域写**的试车；模态①双向 Broker（owned↔owned）的完整证明在 §8 Step 4（背包↔仓库 Gate A2）。

### 9.2 协议闭环表（七列，遵循护栏 §2）

**闭环表**遵循护栏 §2（迁移闭环表）；**C#/AS2/Web 接入清单**遵循护栏 [§3/§4/§5](../agentsDoc/as2-web-panel-migration.md)（护栏 §2 是闭环表、非接入清单，勿混）。商城现有业务请求只带 `type/cmd/callId/...`，**不带 `panel=kshop`**；仅 close 请求带 `panel=kshop`。Host 以全局 cmd 分支把下列业务命令交给 `ShopTask`，其再机械派生 `action="shop"+Cap(cmd)`，回包 task=`shop_response`：

| Web cmd | C# action | AS2 handler | AS2 resp task | C# panel_resp | JS handler | 写状态 |
|---|---|---|---|---|---|---|
| `bulkQuery` | `shopBulkQuery` | `shopBulkQuery` | `shop_response` | `panel_resp` + `callId`（ShopTask 仅剥 `task`、置 `type/callId`，**不盖 panel/cmd**） | `_pendingReq[callId]`（纯按 callId 匹配） | 读 |
| `saveCart` | `shopSaveCart` | `shopSaveCart` | `shop_response` | 同上（仅 callId） | 同上 | 写(购物车) |
| `checkout` | `shopCheckout` | `shopCheckout` | `shop_response` | 同上（仅 callId） | 同上 | 写(K点+已购买+存盘) |
| `claim` | `shopClaim` | `shopClaim` | `shop_response` | 同上（仅 callId） | 同上 | 写(已购买→背包+存盘) |

首战保持上表请求/回包字段形状；扩展 shop 写协调的 Host 前置拒绝错误词汇。Web callback 必须区分 `busy` 与 `reconcile_required`：`busy` 表示已有 shop 写在途，不得据此重发，也不得让被拒的重复请求释放当前 owner 锁；`reconcile_required` 表示 Host 持有不确定 shop 写结果，必须先 `shopBulkQuery`。claim 还须区分 `item_not_found`、`inventory_full` 与 `acquire_failed`；`item_not_found` 触发已购买列表重拉，后两者是明确未写入失败。checkout 的 `insufficient_kpoints` 是明确未写入失败。

**共享背包单独立 domain/Task，勿绑进 shop_response**（背包是后续所有面板共享的领域；若复用 `shop_response` 会把共享容器永久绑死在商城）。新增：

| Web cmd | C# action | AS2 handler | AS2 resp task | C# panel_resp | JS handler | 写状态 |
|---|---|---|---|---|---|---|
| `snapshot`(domain=inventory) | `inventorySnapshot` | 新 `handleInventorySnapshot` | **`inventory_response`** | `panel_resp domain=inventory cmd=snapshot` | 背包视图 callback | 读 |
| `discard`(domain=inventory) | `inventoryDiscard` | 新 `handleInventoryDiscard` | **`inventory_response`** | `panel_resp domain=inventory cmd=discard` | 丢弃 callback + 背包刷新 | 写(背包删除+dirtyMark) |

**saveCart/checkout 闭环**：购物车编辑只更新 Web 期望态并触发“单个在途 + 最新值合并槽”的 save pipeline；checkout 必须等该 pipeline 空闲后才发送，并在终止前冻结新的 save。saveCart 结果不确定时先 bulkQuery；若返回影子与当前 Web 最新期望 cart 不同，对账完成后发送一次**新的最新值同步**，不得重放旧请求。checkout success 用回包中的 `newBalance/purchased` 清空 Web cart；`insufficient_kpoints` 是明确未写入终态；已下发 checkout 的 timeout/途中断连/未知结果只允许 `shopBulkQuery` 对账，禁止重发。正常 close 等待该闭环空闲后才做最后一次 saveCart；force_close 不补发。测试必须覆盖 saveCart 在途时点击 checkout/关闭、连续编辑只落最终影子、saveCart timeout 后对账并仅同步最新值、旧 saveCart 不得在 checkout 后回填影子、checkout 快速双击/绕过 UI 第二次发送、timeout 后仅 bulkQuery 对账、force_close 后重开对账。

**claim→背包刷新的链式闭环**（方案 A）：`claim`(shop_response, success) → Web 回调内发 `snapshot`(domain=inventory) → `inventory_response` → 背包视图刷新。两条独立通道串接，AS2 claim 回包不变；Host shop 写 gate 防在途重入，Web 全局锁覆盖整条链。claim timeout/途中断连/未知结果后 Host 将 shop 写 gate 置为 `needs_reconcile`，后续 shop 写一律拒绝，直到成功完成 `shopBulkQuery` 对账；Web 仍须再完成 inventory snapshot 才解锁 UI。测试必须覆盖快速点击两条已购买记录、绕过 UI 直接发第二个 claim、三条显式失败均不修改背包/已购买、claim 成功但 snapshot 失败、claim timeout 后对账。

实现该 gate 需要扩展**当前** [ShopTask.cs](../launcher/src/Tasks/ShopTask.cs) 的内部 pending：它现在只记 flash↔web callId，无法在 timeout/回包时区分写命令与 bulkQuery。首战须改为记录 WebCmd（或增加等价 shop write coordinator），维护 `idle/write_pending/needs_reconcile`，其中 `write_pending` 记录 `saveCart/checkout/claim` owner。`write_pending` 拒绝后续 shop 写时回 `busy`；`needs_reconcile` 拒绝 shop 写时回 `reconcile_required`，但必须允许新的 bulkQuery 用于对账。saveCart 收到 success、checkout 收到 success 或 `insufficient_kpoints`、claim 收到 success 或已列明的三种未写入失败，才回 idle；未知/畸形回包、已下发 shop 写的 timeout/途中断连都必须先转 `needs_reconcile` 再清传输 pending，不能按普通请求直接回 idle；发送前拒绝则不进入该状态。特别是 `OnSocketDisconnected`：若存在已下发 shop 写，`ClearPending()` 只能释放 timer/映射，必须保留 `needs_reconcile`，不能顺手复位 gate。只有进入 `needs_reconcile` **之后新发起且成功**的 bulkQuery 才能清除 Host 状态，旧 pending 回包不得误清；重连重开后，Web 仍按初始门槛完成新的 `shopBulkQuery + inventory snapshot` 才重新启用写入口。pending、timer、owner gate 与重放缓存的读取/终态转换必须受同一把锁保护，每个请求只能由 response/timeout/disconnect 其中一条路径完成一次，迟到路径不得释放或覆盖后续 owner。该 Host 内部行为变更**不改变**现有 shop* 请求字段和 `shop_response/panel_resp` 形状。

**discard 闭环**：Web 在用户明确确认（展示物品名与数量；确认投影 = `{itemKind,displayName,quantity,enhancementLevel,rarity}` 版本化子集，§4.1）后发送 `domain=inventory cmd=discard`，payload 携背包 source slot lease；C# 白名单映射 `inventoryDiscard`；AS2 v1 仅允许 `containerId=背包`，重新校验 source slot lease（`currentItem === lease.occupantRef`）、槽位存在性与领域丢弃策略后删除**整个槽/整组堆叠**、设置 `dirtyMark`，并在同一 `inventory_response` 回包中带刷新后的背包 snapshot。v1 不接受 partial count；以后若要部分丢弃，单列协议版本。删除失败不得局部更新 Web；`stale_state` 后只能刷新并要求用户重新确认。发送前拒绝可直接失败；已下发 discard 的 timeout/途中断连/未知回包一律视为结果不确定，只允许拉 inventory snapshot 对账，**禁止自动重发破坏性写**。测试覆盖取消确认、陈旧 source、空槽/非法槽、非背包容器、策略拒绝、夹带 count、成功删除并标脏、成功回包刷新、timeout 后仅对账。

**Web 回包复用边界**：当前 [bridge.js](../launcher/web/modules/bridge.js) 会把同一 `type` 的消息广播给全部监听器，KShop 的 [kshop.js](../launcher/web/modules/kshop.js) 又只按 `callId` 查 `_pendingReq`。首战同时存在 legacy shop 回包（无 domain/cmd）与 inventory 回包（有 domain/cmd），故 KShop 工作台必须只有一个 request mux，并生成带 workbench 命名空间、WebView session nonce、openGeneration 与单调序号的 callId，避免与其他 panel 模块撞号或 WebView 重载后复用旧 ID；pending 项记录 `{channel, cmd, callback, openGeneration}`，并由同一协调器持有 backpack write gate。mux 在发送前拒绝注册重复 callId。回包先按 callId 取 pending，再校验预期形状：shop 项拒绝带 inventory domain 的回包；inventory 项必须匹配 `domain=inventory` 与原 cmd。只有形状校验通过的回包才能消费 pending；错形/错通道回包记录并丢弃但不得删除等待项，已完成请求的重复回包直接忽略。关闭/强制关闭清空 pending 并递增 generation，迟到回包不得落到新一轮视图；若被清项含已下发写，其结果按 §9.1 的重开对账规则处理，绝不能因 pending 被清就视为失败或取消。禁止 GridContainerView/InventoryView 各建一套可能撞号的独立 pending 表。

**本地拒绝也必须可路由**：InventoryTask 自身产生的 `unsupported_cmd/unsupported_version/invalid_payload/timeout` 以及**发送前** `disconnected` 均须回 `panel_resp + domain=inventory + 原 cmd + callId`；Host 对未知 domain 的拒绝须回显原 `domain/cmd/callId`，不得让 mux 只能等超时。已下发请求后的途中断连不是普通本地拒绝，按上文强制关闭 + 重开对账处理。工作台生成的 callId 必须匹配 `^[A-Za-z0-9._-]{1,96}$`；缺失、超长或非法字符在创建 pending 前拒绝。ShopTask/InventoryTask 还须按完整 Web callId 维护活动集合与有界近期重放缓存，并与 pending/timer 在同一锁内转换：活动 callId 的重复请求不得下发 Flash，也不得回一个可能被 owner callback 误消费的同 ID `busy`；已完成/已拒绝 callId 的重放不得再次下发写命令；timeout、关闭或断连清理 active pending 时先把这些 ID 移入近期缓存，`ClearPending()` 不能顺手清空重放历史。新增/改造的 C# 本地错误回包用 `JObject` 构造，禁止把 callId 直接拼接进 JSON 字符串。即使有重放缓存，攻击者用**新 callId** 重发陈旧 `purchasedIdx` 仍无法被 Host 识别，完整防线仍需已购买列表 token/稳定 claim ID。

**路由顺序（必修，否则落错 Task）**：代码核查（2026-06-20）坐实当前 [WebOverlayForm.cs](../launcher/src/Guardian/WebOverlayForm.cs) 的 `private void HandlePanelMessage` 是 `switch(cmd)`、`case "close"` 为第一个 case（已最优先），裸 `snapshot`（panel 缺失/未知）经 `panel==` 白名单逐一不匹配后落末尾 `else → _mapTask.HandleWebRequest`，且**全文不读 `domain`**（domain 早分流是 net-new）。⚠**caveat**：落 MapTask catch-all 的不止 `snapshot`——该 case 组里所有非 drop-list（`enter/jump_frame/return_frame/open_stage_select` 在未知 panel 时被显式丢弃）的未知-panel cmd 都会落 MapTask，故 domain 早分流不能只拦 `snapshot`、要拦整个 `domain=inventory` 白名单（detail/refresh 等同理）；另 `_mapTask` 有 `!= null` 守卫，插 InventoryTask 在前须保持守卫一致。改造顺序必须是：①保留通用 `cmd=="close"` 最优先；②若消息携 domain，则 `domain=="inventory"` 只把白名单 cmd 交 `InventoryTask`，其他 domain 回 `unsupported_domain`，**不得**回落既有全局 cmd；③仅无 domain 的 legacy 请求进入既有 panel/cmd 路由。`InventoryTask` 不处理 close，KShop close 仍由 Host 通用关闭路径解析为 `shopPanelClose`。路由测试至少覆盖：

- `{cmd:"snapshot", domain:"inventory", panel:"kshop"}` 命中 InventoryTask，不落 MapTask。
- `{cmd:"close", domain:"inventory", panel:"kshop"}` 仍走通用 close，不被 InventoryTask 吞掉。
- `{cmd:"unknown", domain:"inventory"}` 回 `unsupported_cmd`，不派生或下发任意 action。
- `{cmd:"claim", domain:"unknown"}` 回 `unsupported_domain`，不得回落 ShopTask。
- Web 夹带 `action`/`task`/`op` 不能覆盖 C# 白名单映射；InventoryTask 只下发经验证并解包的 payload。
- InventoryTask 本地拒绝/timeout 与 Host `unsupported_domain` 回包均保留可供 mux 校验的原 `domain/cmd/callId`；非法 callId 不创建 pending。
- mux 不注册重复 callId；Host 收到活动/近期已见 callId 时不再次下发 Flash，且活动 ID 的重复请求不能用同 ID 错误回包干扰 owner。

新增 `InventoryTask` + 注册 `inventory_response`（C# 接入面照护栏 §3：Program.cs 创建注入、TaskRegistry 注册、WebOverlayForm 字段 + close 后的 domain 分支 + OnSocketDisconnected 清传输 pending、xUnit 覆盖 cmd→action 白名单/负载解包/回包重写/断连/路由优先级；其余 §3 标准项——Task 自身 timeout/`Dispose()`、`SetInventoryTask()` 注入方法一并照搬）。**`.csproj` 无需手改**：主 csproj 与 `launcher/tests/Launcher.Tests.csproj` 均为 SDK glob 风格（无 `<Compile Include>`），`InventoryTask.cs` 落 `src/Tasks/`、新测试落 `tests/Tasks/` 即自动纳入编译。断连会强制关闭面板；若 discard 已下发而回包未确认，重开时的初始 inventory snapshot 承担对账，任何层均不得自动重发该 discard。**注**：现有 `ShopTask/MapTask` 桥本身并无专用 xUnit（`launcher/tests/Tasks/` 无 `ShopTaskTests`）；首战须同时新建 InventoryTask 桥测试与 ShopTask shop 写 gate/重对账状态测试，骨架可参照 `PetTaskTests`（恰 6 个 `Fact`）/`MercTaskTests`（4 方法、含 `Theory` 7 `InlineData`、执行 10 例），四类断言=断连错包 / 已知 cmd→可信 action（含 Web 夹带 `action/task` 反污染）/ 未知 cmd 错包 / 回包还原 web callId。不要为 InventoryTask 复制 panel close 语义。

> 备注：上表 `shop*` 命令在现有 KShop 已存在；首战保持其协议形状，但会扩展 ShopTask 内部 pending/shop 写 gate。另新增 `InventoryTask`/`inventory_response`/背包视图，以及双栏壳与容器视图的 Web 层。`shop_response` 与 `inventory_response` 是两条独立回包通道。
>
> **模板选型（重要，决定 InventoryTask 怎么写）**：当前 `ShopTask` 是现有桥中不在 `panel_resp` 盖 `panel/cmd` 的最小桥；见 [ShopTask.cs](../launcher/src/Tasks/ShopTask.cs)，定位锚点 `public sealed class ShopTask` 与 `msg.Remove("task")`。即使为 shop 写 gate 扩展其内部 pending，shop 回包仍保持只按 callId 匹配的既有形状。`InventoryTask` 的回包要带 `domain=inventory cmd=<原 cmd>`，故应照 [MapTask.cs](../launcher/src/Tasks/MapTask.cs) 的 `private sealed class PendingRequest` 范式记录 WebCmd——**但注意 MapTask 现状盖回的是 `panel=map`/`cmd`、回包并无 `domain` 字段**（核查纠偏：勿误以为 MapTask 已有 domain 可直接复制）；InventoryTask 是把 PendingRequest 的 `panel` 位置换成 `domain`、盖回 `domain/cmd` 的**新形状**，且 action 映射必须改成 inventory 专用显式白名单，**勿机械复制 ShopTask 的字符串拼接**。

## 10. 债务防线小结

1. 容器只在视图层统一，所有权/命令按实例区分（§6 纠正线）。
2. 背包-only 约束的 SOT 在 **AS2 领域服务**；视图能力声明只是 snapshot 下发的 UX 预检，不是权限 SOT（§1.2、§5）。四层职责：Shell 只认 View 生命周期 / InteractionBroker 仅产中性 `OperationIntent`、不认识 AS2 命令名 / DomainCoordinator 落命令与并发 / AS2 最终裁决业务规则（§2）。
3. 模态② UI 视图本地，第二消费者出现才抽壳级托盘，命名抽取点但不提前泛化（§3）。
4. 命令分层（§4.1）：①a 同质 owned 用 `inventoryMove/merge/swap`；①b 装备/卸装/claim/discard/sell 用显式领域命令（带专属业务）；②批量用 checkout/craft；不存在吃下一切的 `itemTransfer`。逐格 `itemActions` 与整 View `viewCommitActions` 分开（§6）。
5. 命令寻址用**稳定 containerId**（非展示槽）。并发防线 = **AS2 snapshot-local slot lease（O(1) 按 BaseItem 对象引用判定 + 按命令前置条件）**，按写范围覆盖 source/target；排序/`setItems` 后整容器 lease 失效。**已废除"全量 canonical descriptor 作普遍防线"**（语义过度 + AVM1 性能自杀）；canonical encoder 降为诊断/测试 oracle/少数 exact-equality 命令专用。`expectedFingerprint` 改名 `expectedLease`，并把 `snapshotSeq`/`slotLease`/`containerEpoch`/未来 `instanceId` 四职责拆开。Web `cmd` 只经 C# 白名单映射可信 action，信封带版本 `v`，且**仅适用新命令族，不回填 KShop shop\***（§4.1）。
6. 排序拆 `displaySort`(Web 本地) vs `inventorySortAndMerge`(AS2 写)，勿混；新 handler 成功后显式 `dirtyMark`（§6）。
7. AS2 权威不动；Web 禁用钮≠安全校验；写操作 AS2 重校验（§4、护栏 §7）。
8. 共享背包走独立 `InventoryTask`/`inventory_response`；Host 先保留通用 close，再路由白名单 `domain=inventory`。Web 工作台以单一 request mux 隔离 legacy shop 与 inventory 回包，勿只凭可撞号 callId 跨通道分发（§9.2）。
9. KShop 以共享 shop 写 gate 串行化 `saveCart/checkout/claim`，checkout/claim 不确定结果先 bulkQuery 对账；claim 另以 Web 全链单飞约束位置索引并显式建链至 inventory snapshot；discard 走独立确认/校验/标脏/刷新闭环。v1 工作台以共享 backpack write gate 串行化 claim/discard 两条背包写，防跨 Task 投影倒序；AS2 slot lease 仍是 inventory-domain 权威防线。**claim 的 `purchasedIdx` 漂移**：稳定 claim ID/已购买列表 token 升为 §9 明确必修项（位置索引 splice 后会静默领错相邻物品，且 checkout 回包重排 `_purchased` 是第二条漂移源，§9.1/§9.2）。
10. 资源箱等瞬态容器**暂不接入 Broker 转移**（库存数据对象无稳定 ID、不入领域注册表；UI 层虽有粗粒度清理，数据对象本身不可寻址，§4.3/§6）；纳入前补 session-scoped containerId + 失效错误 + 关闭清理（§6）。
11. **首战 KShop 只冻结 Shell/ViewHost（Gate A1），不冻结 Broker**；真正的双向 Broker（owned↔owned）验收在 Step 4 背包↔仓库最小纵切（Gate A2），通过后才放行后续界面。迁移拆 Step 0（独立修 ShopTask 写协调）/ Step 1（提取 request mux），避免一个 PR 同改 UI 架构 + Shop 并发 + Task 路由 + inventory 协议（§0.0 决议 A、§8）。Gemini 的 `acquire_failed → needs_reconcile` 因 `acquire` 原子无副作用被证伪、不采纳（§0.0、§9.0）。

## 11. 已定细节（2026-06-15）

- **双栏槽位 UX**：**左∈{目录, 购物车} / 右∈{背包, 购物车}**，由**视图单例不变量**（§2）自动产出且仅产出三种配置，双购物车被排除：
  - (目录, 背包) 逛+看空间 ｜ (目录, 购物车) 逛+攒单（**面板初始进入态**）｜ **(购物车, 背包) 背包满时腾格子领取**。
  - **「默认」消歧**（两义勿混）：右槽的**结构默认 View = 背包**（displaced 时的回落目标）；**面板初始进入态 = (目录, 购物车)**（进面板首屏，右槽初值=购物车）。
  - 当右=购物车、在左点"购物车"，单例规则把购物车移入左槽；左槽原先的目录不允许挂到右槽，因此右槽 displaced 后回落其**结构默认 View（=背包）**，正好落到 (购物车, 背包) 腾格子态。比购物车专属状态机更省，对后续背包+仓库同屏腾挪沿用同一冲突解析规则。
  - 取舍：切右栏不打断浏览；背包+购物车同屏是**背包满才能领取**的刚需（非锦上添花）。早期"左=目录⇄购物车 / 右=背包"的窄方案已被本方案取代。
- **设计画布**：**1024×576**，对标既有面板/Flash（Stage 尺寸本作动态，落到实处即沿用 PanelScale 标准画布）。好处：AS2 网格几何可原样搬（图标 28px、背包 10 列=280px）。约束：每栏可用约 480–500px，施工保持布局紧凑。
- **乐观更新**：v1 全程 round-trip 确认；**功能完成后再补**乐观更新 + snapshot 对账。
- **拖拽镜像**：**Web 自绘**（不续用 AS2 `物品图标容器`），与"AS2 UI 全面外迁"长期战略一致。
