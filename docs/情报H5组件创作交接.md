## 数据契约

每个 H5 文件形如：

```json
{
  "schemaVersion": 1,
  "itemName": "资料",
  "skin": "paper",
  "pages": [
    {
      "pageKey": "1",
      "blocks": []
    }
  ]
}
```

规则：

- `itemName` 必须精确匹配字典项，文件名必须是 `<itemName>.json`。
- `pages[].pageKey` 必须与 `information_dictionary.xml` 中该情报的 PageKey 顺序完全一致。
- 锁定页由 C# 截断，不向 Web 下发 `blocks`。
- JSON 中不得出现自由 HTML、自由 JS、事件属性、任意 asset 路径或项目根路径。
- 新增组件必须同时更新 C# 白名单、Node validator、Web renderer、CSS、harness 和本文档。

## 组件语义

### 文本结构

- `paragraph`：普通正文段落，可承载 inline token。
- `heading`：页内标题；避免滥用大号标题，保持旧 Flash 情报窗的阅读密度。
- `list`：条目化信息。
- `table`：档案字段、工单、材料清单、对照表；单元格可使用 inline token。
- `quote`：引用、转录、二手记录。
- `divider`：轻量分隔，不承担叙事。

### 档案与批注

- `note`：中性备注框。
- `handwritten`：手写批注、档案人注记、铅笔/红字旁注。它表示“材料上后来留下的人工痕迹”，不要拿来代替普通正文。
- `annotation`：左右并排的正文 + 注释结构，适合需要在旁边解释、质疑或补充的段落。
- `stamp`：封存、作废、机密、已核验等档案章。

### 破译与损毁

- `decryptText`：段落内局部密文。**真隐藏**，未达 `decryptLevel` 时 C# 后端剥离 `content` 字段，前端不渲染明文、不挂 hover/click（相比 redaction 的"hover 偷看"更严格，对得起逆向技能投入）。视觉按 `level` 自动分档：
  - `1-3` → 撕缺米色小纸条（tier1）
  - `4-6` → 涂改液白底 + 蓝黑笔划线（tier2）
  - `7-9` → 焦痕碎块（tier3）
  - `10` → 黑色油墨晕染（tier4）
  - `encryptedText` 是必需的占位字段，`content` 只在解锁时下发。
- `decryptBlock`：整段或整组内容受解密等级控制，适合"密钥破译全文乱码"。
- `redaction`：有真实 reveal 明文的遮盖内容（叙事性涂抹）。**保留 hover/click 揭示行为，不受技能门槛约束**。没有真实明文时不要用它。
- `damageText`：不可恢复占位，如 `[数据损毁]`、`[涂抹]`、`[已编辑]`、`[缺失]`、`[已删除]`、`[签名模糊]`。它不应该弹出明文。

`redaction`、`decryptText`、`damageText` 的边界必须清楚：技能门槛锁死的是 `decryptText/decryptBlock`（不可偷看），叙事可揭的是 `redaction`（可 hover），永远缺失的是 `damageText`。

### 场景化材料

- `terminalLog`：终端、控制台、日志提取。
- `hardwareExtract`：硬件介质、芯片、黑匣子、破损设备提取；当前为 v0 交互，可展开步骤和结果。
- `blueprint`：图纸、材料、改造路线。
- `timeline`：时间线、事件链。
- `surfaceMark`：纸面叙事装饰，如污渍、水痕、血手印、折痕、破损。它应服务于文本氛围，不应遮挡核心阅读。

### 叙事容器（field-notes skin 专用）

资料类条目可走"碎纸条钉到笔记本上"的统一隐喻；这套容器与 `field-notes` skin 配合使用。

- `paperStage`：纸条容器，必带 `layout`：
  - `stack`：纵向自然流，纸条上下堆叠（主用，连续叙事）。
  - `scatter`：绝对定位散落（彩蛋页用，每张纸条要给 `x/y/w/rotate/opacity`）。
  - `fragments` 数组**只放 `paperFragment`**，不要混其他 block。
- `paperFragment`：单张纸条；属性：
  - `tone`：`aged`（米黄旧纸）/ `burnt`（焦痕）/ `torn`（撕缺）/ `ink`（蓝印记）/ `carbon`（复写纸）。
  - `pin`：`none`（无）/ `pushpin`（黄铜图钉，主条目）/ `tape`（米色胶带，短结论）/ `clip`（金属回形针，延伸观察）。
  - `content`：inline 数组（常见纯文本纸条）。
  - `blocks`：block 数组，用于在纸条上嵌入 heading/annotation 等复合结构（如 pageKey=40 的"研判摘录"模式）。

不要把 `paperFragment` 用作纯装饰散落；那是 `surfaceMark` 的职责。`paperFragment` 永远承载正文，是表达力载体而非纸面纹饰。

## Inline Token

- `text`：普通文本。
- `strong`：强调。
- `underline`：下划线。
- `colorToken`：语义颜色，使用 `danger/info/warning/success/material-* / faction-* / biohazard` 等 token，不写任意颜色。
- `pcName`：玩家名占位。
- `damageText`、`redaction`、`decryptText`：见上文边界。

## 逐篇手工流程

1. 选 1-3 篇情报，不要一次改全量。
2. 阅读 legacy txt、H5 JSON、世界观文档，先写一句目标，例如“强化水厂外勤档案的封锁现场感”。
3. 只改 `data/intelligence_h5/<itemName>.json`，除非确认现有组件表达不了目标。
4. 优先复用现有组件：局部加密用 `decryptText`，不可恢复缺损用 `damageText`，手写痕迹用 `handwritten`，纸面痕迹用 `surfaceMark`。
5. 每页保持可读性：组件应该分层信息，而不是把所有段落都装饰化。
6. 修改后跑 strict validator 和 harness；窗口 / 全屏布局问题优先在 CSS 或组件参数上修，不要靠删内容绕过。

建议每篇交付时记录：

- 目标情绪或表达力。
- 使用了哪些组件。
- 是否新增组件。
- 是否改变原文本含义。
- 是否需要世界观复核。

## 何时新增组件

只有同时满足以下条件才新增组件：

- 至少 3 篇情报会复用。
- 现有组件组合无法自然表达。
- 能用 schema 明确描述，不需要内容侧自由 JS。
- 能写出 harness 用例。
- 能给出移动 / 窗口 / 全屏下的稳定布局规则。

单篇强风格需求优先通过 `skin`、现有组件参数或手工排版实现，不急着扩 API。

## KimiCode 使用边界

KimiCode 适合做粗迁移、表格化、重复结构整理和候选方案，不负责最终审美决策。批量任务必须限制写入范围到 `data/intelligence_h5/` 的目标文件；不得允许它修改 renderer、CSS、C# schema、validator 或 README。

推荐用法：

- 给 KimiCode 一篇或一小批原文、PageKey、组件规范和目标风格。
- 让它输出候选 JSON。
- 人工/Codex 审阅语义和节奏后落地。
- 每批通过 validator 和 harness 后再进入下一批。
