## 1. 诊断日志分层原则（三层）

新增诊断日志前先归层。**层之间的语义不同，混用会让玩家版要么漏样本、要么背负无谓开销**。

### Tier 1 — 永久"账本"（事件 / 阈值触发，绝不开关化）

| 日志前缀 | 触发 | 体量 | 来源文件 |
|---|---|---|---|
| `[Activation]` lost/gained | WM_ACTIVATEAPP | 一次 3h 会话 ~27 行 | [`launcher/src/Guardian/GuardianForm.cs`](../launcher/src/Guardian/GuardianForm.cs) WM_ACTIVATEAPP 分支 |
| `[PerfGate]` closed/opened | 失活 / 最小化门控转换 | 仅状态转换时 | [`launcher/src/Guardian/PerfDecisionEngine.cs`](../launcher/src/Guardian/PerfDecisionEngine.cs) |
| `[FgWatchdog]` vacuum detected | 真空 ≥2 tick 后回收，2s 节流 | 稀有 | [`launcher/src/Guardian/GuardianForm.cs`](../launcher/src/Guardian/GuardianForm.cs) `OnForegroundWatchdogTick` |
| `[FocusRestore]` | 主动回收前台（panel close / watchdog） | 每次 panel idle 1 次 | [`launcher/src/Guardian/WindowManager.cs`](../launcher/src/Guardian/WindowManager.cs) `RestoreFlashInputFocus` |

**原则**：事件 / 阈值触发，健康路径零行；偶发卡顿复盘**唯一数据源**。"关掉 = 拆黑匣子"，故**不暴露开关**。

### Tier 2 — 永久"探针"（默认 ON + env 急停）

| 日志前缀 | 触发 | 体量 | 急停 env | 来源文件 |
|---|---|---|---|---|
| `[UIFreezeProbe]` ui_stale_enter/exit, hung_window, vacuum_seen | 后台线程 1Hz 轮询 + 阈值触发 | 健康路径 0 行 | `CF7_DIAG_FOCUS_PROBE=0` 跳过线程启动 | [`launcher/src/Diagnostic/UiFreezeProbe.cs`](../launcher/src/Diagnostic/UiFreezeProbe.cs) |
| `[RevealProbe]` setready.* / readyWiring.total | boot 期一次性 | ~10 行 | 无（成本可忽略） | [`launcher/src/Program.cs`](../launcher/src/Program.cs) readyWiring |
| `PerfTrace.Scope` reveal 段 | reveal 各段 | 6 行 / 次 reveal | 无 | [`launcher/src/Guardian/GameLaunchFlow.cs`](../launcher/src/Guardian/GameLaunchFlow.cs) DoPerformReveal |

**原则**：后台轮询 / 启动期一次性，玩家版作为永久监控。开销极低（探针 1Hz + 3 个 P/Invoke），但保留 env 急停作为逃生通道——疑似探针自身在特殊机型 / 杀软环境交互不可预期时用。**不暴露 toml 开关**避免玩家误关导致漏样本。

### Tier 3 — 基线测量器（默认 OFF，按需开）

| 文件 | 角色 | 需要 admin? | toml 开关 |
|---|---|---|---|
| [`launcher/src/Diagnostic/LayerAuditDump.cs`](../launcher/src/Diagnostic/LayerAuditDump.cs) | EnumWindows 快照所有 top-level HWND + WS_EX 标志 | ❌ | `diagLayerAudit` |
| [`launcher/src/Diagnostic/UlwCommitMonitor.cs`](../launcher/src/Diagnostic/UlwCommitMonitor.cs) | OverlayBase.CommitBitmap 调用频率 + p50/p95/p99/max 延迟 | ❌ | `diagUlwMonitor` |
| [`launcher/src/Diagnostic/DwmEtwMonitor.cs`](../launcher/src/Diagnostic/DwmEtwMonitor.cs) | 原生 P/Invoke ETW 订阅 Microsoft-Windows-Dwm-Core 事件计数 | ✅ | `diagEtwDwm` |
| [`launcher/src/Diagnostic/DiagnosticsBootstrap.cs`](../launcher/src/Diagnostic/DiagnosticsBootstrap.cs) | 集中 wiring + Program.Run 启动钩 | — | — |

**原则**：高频连续采样 / 需 admin / 大尺寸 dump。默认 OFF 是为了玩家版零成本，按需开收集 PR 前后基线。

OverlayBase.CommitBitmap 已嵌探针（[`launcher/src/Guardian/OverlayBase.cs`](../launcher/src/Guardian/OverlayBase.cs)）。禁用时静态读 + 一次分支，**生产路径零开销**。

PanelHostController.OpenPanel / ClosePanel 末尾自动 dump LayerAudit，**仅 `diagLayerAudit=true` 时执行**。

OnFormClosing 三条分支全部带 `reason=` (CloseReason)，可区分 UserClosing / Alt+F4 / TaskManagerClosing / ApplicationExitCall / WindowsShutDown / FormOwnerClosing。

## 3. 使用流程

### 3.1 基本采集

1. **开三件诊断**（推荐 config.toml 而非 env，避免遗漏）
2. **以管理员启动** —— 否则 ETW 直接 skip，得到 `[EtwMpo] requires Administrator` 一行警告
3. **稳定到游戏内后**：跑你要测量的场景（panel 开关 / 战斗 / 切场景）。每场景至少 30 秒 = 6 个报告周期
4. **退出后立即冻结日志**：
   ```powershell
   Copy-Item logs/launcher.log logs/baseline-pre-B0-2026-05-24.log
   Copy-Item logs/perf-latest.jsonl logs/baseline-pre-B0-2026-05-24.perf.jsonl
   ```
   `launcher.log` 2MB 自动轮转到 `.log.1`，长 session 不冻就丢

### 3.2 启动期 sanity 检查

启动后第一时间在日志里搜：

```
[Diag] enabled: layerAudit=True ulwMonitor=True etwDwm=True interval=5s
[EtwMpo] struct sizes: EVENT_TRACE_LOGFILEW=448 TRACE_LOGFILE_HEADER=280 EVENT_TRACE=88 (expected x64: 448 / 280 / 88)
[EtwMpo] started provider=Microsoft-Windows-Dwm-Core interval=5s session=CF7ME-Diag-DWM
```

三个 struct sizes **必须** = `448 / 280 / 88`。对不上 → **不要继续用 ETW 数据**，可能是 .NET 升级或 Windows API 改了布局，参考 [`memory/etw-struct-layout-trap.md`](../../../../C:/Users/Administrator/.claude/projects/e--steam-steamapps-common-CRAZYFLASHER7StandAloneStarter-CrazyFlashNight/memory/etw-struct-layout-trap.md) 排查。

如果没看到 `[EtwMpo] started`、只有 `requires Administrator` → launcher 没以 admin 启动。

### 3.3 PR 前后对比

B0 工程或任何渲染合成相关 PR：

1. 改前跑一遍同一 playthrough 脚本，冻结为 `baseline-pre-<feature>.log`
2. 改后跑同一脚本，冻结为 `baseline-post-<feature>.log`
3. 对比四类数字：
   - `[LayerAudit] layered=` —— 结构指标，应该减
   - `[LayerAudit] layered_visible=` —— 稳态合成压力，应该减
   - `[UlwMonitor] commits/s` —— commit 频次
   - `[UlwMonitor] p99=` —— DWM stall 尾延迟，应平
   - `[EtwMpo] events/s` —— DWM 总活动量，应该减

注意 `layered_visible` **随状态变化**（panel-open 时是另一档），对比要对齐同一游戏状态。

## 6. 局限 / 已知盲点

- **ETW 需 admin** —— Steam 启动一般无 admin，玩家版数据收不到。Dev 测试需手动 admin 启动。
- **UlwMonitor 不覆盖 WebOverlay** —— WebView2 走自己合成路径，CommitBitmap 不经过它。这是测**我们 GDI+ overlay** 那条线的工具。
- **ETW callback 只 ++counter，未解析 payload** —— 当前给出"DWM 活动总量"，不区分 plane state change / present complete。要细分在 `OnEventRecord` 加 EventDescriptor.Id 分类即可。
- **LayerAudit 推断 form 身份靠 class name + size + pos** —— 同 class 同尺寸的两个 form 区分不开。未来若需精确 hwnd→Form 映射，在 OverlayBase / form 构造时记字典。
- **高配开发机看不到完整 MPO 病** —— 平均帧时间被压平，只有 p99 / 事件频次能看出 stall。真实弱机用户体感的可玩性下降必须在弱机实测。

- InputShieldForm vis=Y 1600x900 LAYERED — Ghost Input 命中
- **WebOverlayForm vis=Y 2560x1440 TransparencyKey（无 LAYERED）** — WebView2 视觉
- **NativePanelBackdrop vis=Y 2560x1440 TransparencyKey（无 LAYERED）** — opaque backdrop

### 关键发现：NativePanelBackdrop 是已经做了的 B0 优化, 不是新候选

读 [NativePanelBackdrop.cs:13](../launcher/src/Guardian/NativePanelBackdrop.cs#L13) 原注释：
> 故意不继承 OverlayBase（layered window）：普通不透明 Form 让 DWM 按 opaque surface 合成, 避免 per-pixel α traversal。**这是消除 ~15pp iGPU floor 的核心一环。**

机制：panel 开启瞬间拍 Flash 当前帧 + dim 蒙版 → 静态 Bitmap → opaque Form paint。DWM 按 opaque surface 合成（最便宜路径）, Flash 可暂停, WebView2 只渲染 panel 矩形。

**如果"合并 WebOverlayForm + NativePanelBackdrop"**：WebView2 持续 CSS 渲染全屏 backdrop, **iGPU 反而升高 ~15pp**, 把 backdrop 设计的目的完全反掉。这是反向优化。

### B0 候选最终清单

**不可合并（每个都有 deterministic 设计理由）**：
- WebOverlayForm + NativePanelBackdrop = 反 alpha 合成的核心设计
- WebOverlayForm + InputShieldForm = Ghost Input airspace 隔离
- DesktopCursorOverlay 独立 = desktop 级 topmost 跟随光标
- NativeHudOverlay 独立 = HUD 显示主体

**可挖的轻量空间**：
- HitNumberOverlay 懒创建（省句柄不省 GPU, 微小）

### 严肃结论

**当前 layered=5 是反复优化后的结果**, 每个 form 都有具体设计理由。"B0 合并"在当前状态下 **没有水可挤**。如果 iGPU 仍有瓶颈, 原因更可能在：
- WebOverlayForm 自身 WebView2 GPU 占用（独立大头）
- Flash SA 自己的渲染（独立大头, 跟 quality 设置和 hwAccel 相关）
- DWM 跨进程合成本身开销（架构限制, 解决方案 = C# 自有合成器, 见 `docs/launcher-渲染架构-长期决策-2026-05-21.md`）

**B0 工程已收**, 资源转其他方向。原 `docs/launcher-渲染架构-长期决策-2026-05-21.md` 关于 "B0 通过减少合成层数解决 MPO churn" 的假设**需要重新审视** —— 当前 layered_visible=1~2 已 minimal, B0 在合成层维度不再有空间。
