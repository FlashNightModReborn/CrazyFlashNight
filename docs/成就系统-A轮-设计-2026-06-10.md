## 0. 决策记录(2026-06-10 拍板,原文照录,本设计必须遵守)

**D1 计数型成就从0**: 新计数器(经济次数等)在 ext 中 undefined→0 初始化(参照 PetPanelService.as:978 宠物购买次数 isNaN 守卫范式); killStats 系成就用基线快照——成就系统首次初始化时记 基线=当前 killStats.total(及设计用到的有界 byType 键), 进度=max(0, 当前-基线); ⚠基线字段 undefined→快照【当前值】, 绝不能→0(否则老档秒解锁); 状态型成就(基建/技能等级/任务完成/物品拥有)不设基线、追溯解锁(老玩家基建满级应直接亮「基建大王」)。

**D2 A轮范围**: 基建成就+击杀里程碑+任务完成/链进度+技能等级/物品拥有(全现成数据) + 经济「次数型」累计(成功购买/强化/宠物领养·进阶等, 仅 .as 成功分支埋点; 合成/改装执行点在 物品改装界面.swf 帧脚本→本轮降级为状态型「拥有改装产物」或排除; 金额型累计花费/收入暂缓, 除非与次数埋点同站点零额外成本可顺带记录)。单局挑战(SessionStats)/好感度=后续计划, 文档只留占位不设计细节。

**D3 领取制**: unlocked/claimed 分离; 玩家在 web 面板逐个点击领取(远程领取=天生 claim 动作, 沿 finish_remote→claim:{mode:npc|remote|auto} 泛化, 成就用 remote, 无奖励成就可考虑 auto); 领取幂等(claimed 位图); 背包满→领取失败保持 unlocked-unclaimed 可重试(天然化解旧档批量补发雪崩; 参 FinishTask 通信_鸡蛋_任务系统.as:289 背包满 return false 先例)。

> v1.1 注: D3 的 auto 为「可考虑」项。评审证实 v1.0 对 auto 只有 schema 无执行点(死字段),A 轮砍掉 auto——全部 remote(无奖励成就也手点确认,UI 零分叉),枚举位与启用方案留 §11/§12,不违背拍板。

### 1.3 后续计划占位(本轮不设计细节)

## 3. 数据设计

### 3.1 内容层 schema — `data/achievement/achievements_a.json`(顶层 `{"achievements":[...]}`),manifest `data/achievement/list.xml`(`<root><achievement>achievements_a.json</achievement></root>`,标签名与任务 `<task>` 刻意不同作第二道防线)

每条字段: `id`:Number(90001 起独立号段) · `category`:"链名#序号"(沿任务 chain 语法;链名=页签: 基建/击杀/任务/技能/物品/经济) · `title`/`description`:字面量短文本(**禁 $ 前缀**) · `hidden`:Boolean(缺省 false) · `rewards`:["物品名#数量",...](可空数组;**禁「经验值」**,§5-⑧) · `claim`:{"mode":"remote"}(A 轮唯一合法值;auto/npc 留枚举注释位不实现;**禁 finish_remote 字段**,出现即派生器 fail) · `objective`:{"type":枚举(§1.1),"params":{...},"target":Number}。

示例三条(基建/击杀/经济):
```json
{"id":90101,"category":"基建#2","title":"百业兴旺","description":"建成 5 处基建设施",
 "rewards":["金币#20000"],"claim":{"mode":"remote"},
 "objective":{"type":"infraBuiltCount","params":{},"target":5}}
{"id":90201,"category":"击杀#3","title":"百人斩","description":"成就启用后累计击杀 100 名敌人",
 "rewards":["K点#50"],"claim":{"mode":"remote"},
 "objective":{"type":"killTotal","params":{},"target":100}}
{"id":90501,"category":"经济#1","title":"精打细算","description":"在商店成功购买 10 次物品",
 "rewards":[],"claim":{"mode":"remote"},
 "objective":{"type":"economyCount","params":{"counter":"购买物品次数"},"target":10}}
```
注: killTotal 文案必须写「成就启用后」(基线语义对玩家诚实);单条 rewards 禁含两条同名货币(ItemUtil.as:323-325 覆盖语义);无奖励成就(如 90501)同样走 remote 手点领取确认,UI 零分叉。

### 3.2 状态容器 — `_root._saveExt.成就`(随 mydata.ext 透传,SaveManager.as:1225-1226 pack/1480 unpack;禁用 reserved,L1227 每次重置)
```
{ v: 1,
  base: { kt: Number },                // killStats.total 基线(undefined→快照当前值,绝不→0)
  cnt:  { "<指标键>": Number },        // AchievementMetrics 唯一写入者,undefined→0
  unl:  { "<成就id字符串>": 1 },       // 解锁锁存位图(v1.1 新增,棘轮语义)
  claimed: { "<成就id字符串>": 1 } }   // 领取幂等位图
```
**unl 锁存位图(v1.1)**: 「unlocked 纯现算推导」对 itemOwned 等非单调事实违反棘轮语义(达成 toast 后卖掉物品→check() 回 false→面板回退+拒领)。改为: 任一锁存写点(§4.1: scanTick / handleClaim ⑥)首次 check()==true 时 `unl[id]=1`+dirtyMark,**只加不减**;对外 unlocked 语义=`unl[id]==1 || check()`(状态型追溯解锁仍天然成立,锁存只是把达成事实固化)。锁存粒度=10s scan,极端窗口(达成后 10s 内卖出且未开面板)未锁存属可接受边界。toast 触发=锁存写入沿(替代会话 memo,顺带消灭跨会话重复通知与 memo/容器重建生命周期问题)。
**有界键约束**: unl/claimed 键=catalog 成就 id;cnt 键∈AchievementMetrics.VALID 白名单(§4.3)。禁实例名/动态拼接键(真档 byType 实测 108 键污染,1791B)、禁自由文本(launcher fffd 修复静默丢 ext 未知字段→文案全由 catalog 查表)。
**体积**: 50 条全解锁+全领取最坏 ~1.3-1.5KB,典型 <0.6KB,对实测 10-19KB mydata stringify +3~8%,可忽略,不做位图压缩。

### 3.3 多存档槽语义
slot 每 Flash 进程固定(bootstrap 握手写 `_root.savePath`,通信_fs_bootstrap.as:35),ext 随 mydata per-slot 往返→**多槽独立天然成立**。破口=同进程「→新建角色」路径的 ext 残留: ① 删档→新建(deleteSlot 清理清单不含 _saveExt);② 坏档→新建(unpackGameState L1480 写 ext **先于** L1485 角色名校验 return false,不经过 deleteSlot)。两条路径最终都汇入 `SaveManager.newCharacter()`(:1069 段),它在「存档数据」步 `_root.mydata = packGameState()` 时把残留 _saveExt 原样打包进新档(现状已致宠物购买次数跨角色泄漏)。

**三道防线(v1.1 重排,主防线移到真正的单一咽喉)**:
1. **主防线**: `newCharacter()` 在首行 `_root.允许存档 = true;` 旁补一行 `_root._saveExt = {};`——新建角色的 ext 无条件为空,同时覆盖「删档→新建」与「坏档→新建」。副作用=宠物购买次数等既有 ext 命名空间一并清零,即顺手修复已知跨角色泄漏 bug(per-character 语义本就正确),施工记录须注明。
2. **纵深**: deleteSlot 清理块(与 killStats/任务重置并列)补 `_root._saveExt = {};`(namespace 无关单站点,非 killStats-others 式四处硬编码)。
3. **兜底**: ensureInit 单调自愈(§4.1,base.kt>当前 total 时整体重建)。⚠已知盲区: 原档零击杀时 kt==0 残留探测不到——有防线 1 后该盲区仅剩防御纵深意义,不再是唯一依赖。

**触发模型=惰性权威+低频轮询锁存**(否决纯事件驱动:killStats 无聚合事件,EnemyKilledEventComponent.as:48-59 只累加;否决每帧轮询):`handleState` 开面板时全量现算(≤50条×读全局比阈值,亚毫秒,纯只读);`scanTick`(10s 循环任务,两次触发间零每帧成本)为**锁存+通知唯一写点**:
```actionscript
private static function scanTick():Void {
    if (!ensureInit() || !_dataReady) return;
    var a:Object = _root._saveExt.成就;           // 现场解引用
    var newly:Array = [];
    for (var id:String in defs) {
        if (a.unl[id] == 1) continue;
        if (check(defs[id])) { a.unl[id] = 1; newly.push(id); }
    }
    if (newly.length > 0) {
        _root.存档系统.dirtyMark = true;
        if (!_suppressToastOnce) for (var i:Number=0; i<newly.length; i++) _root.发布消息("成就解锁: " + defs[newly[i]].title);
    }
    _suppressToastOnce = false;
}
// _suppressToastOnce(v1.1): 容器新建/unl 补缺/自愈重建后的首轮 scan 仅静默锁存不发通知——
// 老档追溯解锁可达十几条,首扫即 toast 轰炸;正常会话(unl 已存在)首扫照常通知(补上上会话末尾未及锁存的解锁,体验正向)。
// 埋点站点可选调 requestScan() 置 flag 缩短延迟。
```

## 11. 任务系统获得的表达力增强

1. **claim:{mode} 泛化(收敛为文档承诺,v1.1)**: 成就域 A 轮单一写法 `claim:{mode:"remote"}`,**finish_remote 出现即派生器 fail**(§5-⑥)——原「归一化函数两种写法都收」造成双权威(AS2 读 raw JSON 无归一化,web catalog 经 derive 归一化,两端语义分叉)且违反 data-schemas.md:217「finish_remote 留 AS2 不进 catalog」纪律。「finish_remote 是 claim:{mode:remote} 的别名」仅作为未来**任务** schema 迁移的文档承诺,届时在任务管线内实现归一化,不进成就 derive 代码。`auto` 枚举位保留: 启用方案已蹚平——AS2 权威落账(scanTick 检出 unlocked && mode==auto → claimed[id]=1+dirtyMark+toast,跳过 acquire)+derive 补「auto⇒rewards.length==0」校验;acquire([]) 空奖励路径已实核安全(getVacancies(0)→[]/require 非 null,ArrayInventory.as:184-191/ItemUtil.as:387-396)。`npc` 枚举位=现行任务交付语义。
2. **量化进度展示范式**: progressOf 的 `{cur,target}` 封顶契约(永不 null)+四态徽章+进度条组件,可反哺任务面板对 get_requirements(收集 N 个 X)的量化展示——目前任务面板仅布尔「可交付」。
3. **后续可选项**: 任务奖励 web 内渲染(摆脱「成功必关面板看 AS2 弹窗」的体验妥协,成就已蹚通回包 rewards 渲染路径);derive 跨域闭包校验反向扩展(任务引用成就作前置);economyCount 指标池(含已顺带落盘的金额型)为后续金额成就/商人好感度免回填;hidden 脱敏+按需回传范式可反哺任务 catalog 进一步收紧。
