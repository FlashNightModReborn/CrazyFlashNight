## 1. 背景

launcher 有两个**独立**的性能问题，本文档统一处理：

**问题 A — 合成路径卡顿（MPO churn）。** launcher 当前是一个 6~7 个 DWM 顶级 HWND 的窗口栈（GuardianForm + 嵌入的 Flash 子窗口 + WebOverlayForm + WebView2 自己的 Chromium 子 HWND + InputShieldForm + DesktopCursorOverlay + NativePanelBackdrop）。DWM 的 MPO（Multiplane Overlay）planner 面对这种透明窗口层叠栈会反复 promote/demote 叠加平面，每次重配置在 present 队列插一个 stall → 帧时间尖刺、WebView2「假死」。已诊断确认。

**问题 B — Flash 本体单核吃紧。** Flash 是 AVM1，单线程、单核，无多核能力。实测把 Flash 的 `_quality` 从 medium 降到 low 是**目前最有效的提帧杠杆**，单核估算可省 50%+ 开销。

两个问题的解都指向同一件事：**把合成收回 launcher 自己管，并把能 offload 的渲染工作移出 Flash 单核。** 因此合并决策。

> 相邻决策（已固化，不在此展开）：shipping launcher **绝不**写 `OverlayTestMode` 等全局注册表项来治 MPO（杀软/SmartScreen 启发式命中 + 反盗版 Guardian 叠加风险 + Steam 口碑）。生产修复必须**进程作用域**。本架构正是那个"进程作用域修复"。详见 `web-overlay-perf-harness` memory 的 MPO 段 / `tools/mpo/`。

## 2. 锁定的约束（决策前提，不再讨论）

这三条是地基。后面所有决策都站在它们上面。

### 2.1 Flash 运行时永久 = Adobe 独立播放器（SA）

- **Ruffle 出局**：Ruffle 官方明确无意支持多 SWF 文件加载（社区已有大量重复 issue），而 CF7 运行期会加载多个 SWF。
- **插件版嵌入出局**：① 大量数据以 SA 为前提加载，迁移数据源时间成本高；② 插件嵌入是进程内，会与 AVM1 主线程抢同一个 CPU 核 —— 直接砸掉问题 B 想保的那个上限。
- **推论**：任何"换运行时就能解决"的方案（尤其"Ruffle 渲染到自有透明贴图"）**不存在**，别再设计。

### 2.2 Adobe SA 播放器的舞台不透明

- `wmode=transparent` 是浏览器插件的概念；SA 放映器的窗口是普通不透明窗口，播放器内部**根本不计算逐像素 alpha**。
- 因此**拿不到 Flash 的真 soft-alpha**。差值抠像（同帧渲两次不同背景色）实时做不到；Flash 内 `BitmapData.draw` 到透明位图会把成本砸回 AS2 核 —— 全部不可行。
- **推论**：Flash 这一层在合成器里**永远是不透明贴图**。

### 2.3 最低支持 OS = Win10

- → `Windows.Graphics.Capture`（WGC，Win10 1803+）可用。这是抓取 D3D 加速窗口内容的现代正解。

## 3. 目标架构

### 3.1 一句话

**塌缩成 1 个 DWM 可见的不透明 flip 窗口。** 由 C# 自己持有合成器（DirectComposition visual 树 或 D3D flip-model swapchain），内部分多少层都行 —— DWM 看不见内部层，不收钱。

塌缩的**单位是顶级 HWND**，不是脑子里的"逻辑渲染面"。单个不透明全屏 flip 窗口是 MPO 的**理想输入**（Independent Flip、零 churn）—— 这不是"打败 MPO"，是把 MPO 唯一能完美处理的东西喂给它。

### 3.2 Flash 层 = WGC 抓帧

- Flash 在合成器里是一张**不透明贴图**，来源 = WGC 抓帧。
- **嵌入模型要改**：Flash 从现在的 `SetParent` → `WS_CHILD` 改成 **off-screen 顶级 `WS_EX_TOOLWINDOW` 窗口**（放屏幕外，无任务栏/无 alt-tab）。WGC `CreateForWindow` 要求顶级窗口；屏幕外不影响 DWM 给它 redirection surface（只要不最小化）。用户永远看不到这个窗口，只看到合成器里那张贴图 —— 视觉伪装不破，反而比 `WS_CHILD` 重定位那套更干净。输入仍走现有的 PostMessage 转发（`WindowManager`），不变。
- **为什么不是现有的 GDI 抓帧**：当前 `GpuRenderer.cs` / `FlashSnapshot.cs` 用 `GetDC` + `BitBlt`。GDI 读的是窗口的 GDI 设备上下文；Flash 开硬件加速后画面走 D3D swapchain present，GDI DC 基本是空的 → **黑帧**。`FlashSnapshot.IsLikelyBlackFrame` 和 `GpuRenderer` 的 10 帧失败 fallback 就是这个问题留下的疤。WGC 抓的是 DWM 合成输出，能正确拿到 D3D 内容。
- 换 WGC 后，删掉 `GpuRenderer` 里的 staging texture CPU 回读（`CopyResource` → `MapSubresource` → `CopyMemory`）—— WGC 帧本就是 GPU 贴图直达。

### 3.3 抽象

- 合成器把 Flash 层当一个"贴图源"接口（`IFlashLayerSource` 之类），合成器只管把若干层叠起来。
- 但**只有 WGC（不透明）一个 provider** —— 因约束 2.1/2.2，"Ruffle 透明 provider"那一档不会存在，别预留。

### 3.4 瓶颈在哪（CPU 还是 GPU）

**都不是吞吐瓶颈。**

- 合成本身（2~4 层全屏 alpha blend）是 GPU 工作，极廉价 —— 即便弱核显也扛得住，这本就是 DWM 今天替你做的事。
- 真正的成本与风险是 **帧步进 / 多时钟域同步**：Flash（SWF 帧率，常 30）、WebView（cap 60）、显示器（可达 240）三个时钟域。自建合成器后你就拥有了这个问题 —— 这是 **CPU 侧的调度问题**，不是吞吐问题。
- 抓帧获取必须 **latest-frame-wins、绝不阻塞**；阻塞式等捕获 fence 会自造 stall（和 MPO churn 同一类病）。
- 代价 = **+1（偶尔 +2）帧延迟**。

**这笔交易的本质**：把一个看不见、不可控的 DWM 调度问题（MPO churn），换成一个看得见、可 profile、可修的进程内调度问题（帧步进）。这是好交易 —— 你能修自己的调度器，修不了 DWM 的 MPO planner。

## 4. 三大杠杆

### 杠杆 1 — `_quality` = LOW + GPU 后处理（头号 CPU 杠杆）

这是**合成器存在的第一理由**。实测 `_quality` medium→low 是最强提帧杠杆、单核省 50%+。

- `_quality` 只影响光栅化器，**不碰 AVM1**。所以省的是主线程里光栅化那一段；收益最大的恰好是视觉密集场景（密集伤害数字、战斗 VFX）—— 也正是会卡的那些。
- 思路：Flash 跑 LOW（省核），GPU 侧后处理把画质补回来 —— AA 这件事从 Flash 单核搬到 GPU。GPU 后处理一个 pass 在 1080p 下约 0.1~0.3ms，弱核显也无所谓。**CPU 轴上 50%+ 一分不还。**
- **诚实的限度**：后处理是"重建"不是"还原"。AA 是创造信息（亚像素覆盖率），`_quality=LOW` 时这个覆盖率根本没被算出来。后处理只能检测边缘 + 猜测性平滑。**LOW + 后 AA ≈ 介于 LOW 和 MEDIUM 之间**，不等于 MEDIUM。能否接受要弱机肉眼验收。

> **关键纠正**：现有 `GpuRenderer` 的着色器是 **CAS（锐化）**。`_quality=LOW` 的病是**锯齿**，锐化会**放大**锯齿 —— CAS 单用对这个杠杆是**错的工具**。
>
> 两个独立杠杆，要配不同的后处理：
> | 杠杆 | 伪影 | 对的后处理 |
> |---|---|---|
> | `_quality` → LOW | 原生分辨率锯齿 | **后 AA：SMAA / FXAA** |
> | 降低渲染分辨率 | 模糊 / 块感 | **升采样 + 锐化：EASU + CAS（FSR1 配方）** ← 现有 CAS 属这条 |
>
> 最可能的最优解是**两杠杆叠加**：Flash 渲染又小又 LOW → GPU EASU 升采样（升采样滤波本身糊掉一部分锯齿）→ CAS 锐化。具体按实测定。

- **文字 / 伤害数字迁到 web overlay**：`_quality=LOW` 最难看的是文字（无字体 AA），而后 AA 对小字恰恰最弱（FXAA 会把小字糊坏）。把伤害数字、UI 文字迁到 web overlay 渲染 —— 那边不受 Flash `_quality` 影响、永远清晰；顺带卸掉 Flash 的 glyph 光栅化负载，与 50% 杠杆叠加。

### 杠杆 2 — 外部背景（边界受限）

意图：把复杂后景的渲染移出 Flash 单核，让 C#/web 在别的核/GPU 上渲。

- **硬约束**：玩家飞行能力极强，一般关**无法**把游戏 sprite 限制在某个区域之外。而"外部背景透出 gameplay"需要 gameplay 的逐像素 alpha matte —— 由约束 2.2，拿不到。
- **结论**：
  - **一般关**：放弃"外部背景透出"。背景留在 Flash。注意 —— 杠杆 1（`_quality=LOW`）逐帧作用、与背景在不在 Flash 无关，**不受影响**，省核大头照拿。
  - **外部丰富背景只限演出关**（见第 5 节）—— 那里禁用飞行能力合情合理，背景区与角色活动区可分离。

### 杠杆 3 — 接缝（外部层与 Flash 层如何拼）

仅在演出关/有外部层时涉及。两条诚实的路：

- **作者区域 mask**：外部渲染只在场景作者标注的有界区域可见（电梯舷窗外、墙上一块屏）。边界是不透明 Flash 美术框，硬边、**零抠像伪影**。约束：角色不能进入该区域。
- **抠像（chroma key）**：外部层透到所有 sprite 之后。`_quality=LOW` 让矢量边变硬 → 矢量内容键得干净；但**位图 sprite 的烘焙 alpha、辉光/加色 VFX 仍会镶边**（CF7 有位图素材）。背景又暗又乱时 halo 不显眼。属"流媒体级"质量，需肉眼验收 + 美术规避。

## 5. 演出层（延后）

把丰富/动态背景做成"演出"—— 甚至用玩家的战斗操作实时驱动另一款开源游戏（如 FreeDOOM）当背景，给演出"升维"。

- **定位**：演出关、禁用飞行。演出为代入感服务，**不考核玩家操作**，因此可松耦合、延迟可接受。
- **选定传输**：RTMP/FLV 直播流入 Flash（AS2 内建 `NetConnection`/`NetStream` + `Video` 对象）。
  - **为什么是它**：`Video` 是 AS2 原生显示列表部件 —— 背景能与 AS2 UI 自由交错、由 Flash 自己的渲染器正确合成（真 alpha 混合，无抠像伪影），相当于"背后放一块大屏幕显示器"。这种集成性是 C# 侧合成给不了的。RTMP/FLV 是 AS2 成熟可靠的内建路线。
  - **代价/待测**：RTMP 是带缓冲的流媒体传输，结构性带延迟（实时端到端几百 ms 起步）—— 松耦合下可接受。**关键待测项**：Flash 侧实时视频解码成本落在 Flash 进程，会吃问题 B 正在优化的那个核。因此演出层**必须**排在游戏本体优化之后，且届时实测解码成本。
- **被否的替代**：把外部游戏做成 C# 合成器的原生层（低延迟、不吃 Flash 核）—— 技术上更优，但牺牲了"AS2 原生部件集成"。鉴于演出可松耦合，作者选择了集成性优先。若未来某个演出需要**跟手**的强耦合，再单独评估这条。

## 6. 被否决的方案及理由（防止重新捡起）

| 方案 | 否决理由 |
|---|---|
| Ruffle 替换运行时 | 官方不支持多 SWF 加载，CF7 多 SWF（约束 2.1） |
| Flash 插件版嵌入 | 数据迁移成本高 + 进程内抢 AVM1 核（约束 2.1） |
| 从 Flash 抠真 soft-alpha | SA 舞台不透明，alpha 从未被计算；可行的抽取法都吃 AS2 核（约束 2.2） |
| GDI BitBlt/PrintWindow 抓 Flash | 读不了 D3D 加速内容 → 黑帧（§3.2） |
| 一般关做外部背景透出 | 玩家飞行能力使 sprite 区无法限制（杠杆 2） |
| shipping launcher 写 OverlayTestMode 治 MPO | 全局/持久/未公开设置，AV 启发式 + Guardian 叠加风险（§1 注） |

## 7. 施工顺序与关键判据

1. **游戏本体性能优化优先**，演出层延后。
2. **spike A**：验证 off-screen 顶级窗口 + WGC 抓帧干净（无黑帧、无冻结、延迟可控）。
3. **spike B（成败命门）**：`_quality=LOW` 的 Flash + SMAA（或 低分辨率 + EASU/CAS）在**弱机**上跑、肉眼验收。后处理若压不住 LOW 的伪影 → 50% 不可落袋 —— 这是整个架构 ROI 的命门，**要最早证伪**。
4. **零成本预检**（可先于一切）：直接把 `_quality=LOW`、不加任何后处理，在弱机上肉眼看一眼。横版动作运动量大，裸 LOW 或许已勉强可接受 —— 若如此，不赌后处理就能先落袋一部分。

## 8. 开放问题

- 演出关的接缝形态：外部层是**有界视口**（作者 mask，干净）还是**整屏背景**（抠像，有损）？决定杠杆 3 怎么做。未定，待具体演出设计。
- Flash 侧实时视频解码是否硬件 offload（影响演出层是否真的省核）—— 待施工期实测。
- 多时钟域帧步进的具体策略（外部层与 Flash 抓帧如何对齐）—— 待 spike 后设计。
