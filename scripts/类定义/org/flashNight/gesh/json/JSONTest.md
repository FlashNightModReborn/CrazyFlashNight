## 说明

这个文件只保留测试入口和测试设计说明，不再混入历史跑数日志。

## 当前测试目标

1. 验证 `JSON`、`LiteJSON`、`FastJSON` 的常规正确性。
2. 显式记录三套实现的当前差异行为。
3. 提供更可信的 parse / stringify 性能基准。
4. 对 benchmark 自身的变体与缓存假设做最小自校验，避免测错对象。

## 重点差异

- `LiteJSON` 当前定位是无缓存、`parse` 优先的启动期解析器，为性能裁剪了部分 JSON 特性。
- `LiteJSON` 不处理任何转义序列（`\n`、`\”`、`\\`、`\uXXXX` 等均原样保留含反斜杠的文本），遇 `\”` 会导致字符串提前闭合并返回 `undefined`。
- `LiteJSON` 不支持科学记数法（`1e2` 等返回 `undefined`）。
- `LiteJSON` 对非法输入不抛错，返回 `undefined`；拒绝根值后的 trailing token、尾逗号、未闭合结构等。
- `FastJSON.parse()` 对同一字符串会返回同一对象引用。
- `FastJSON.stringify()` 对同一对象实例会命中身份缓存，可能返回旧字符串。
- `FastJSON` 对“对象尾逗号”有容错行为，但对多数其他非法输入仍会抛错。
- `JSON` 的非法输入测试走“记录错误但不触发异常上报副作用”的路径。
- `1e` 这类“不完整指数”仍作为单独差异行为记录，因为三套实现并不一致。

## 性能基准原则

- 三套实现都走等价循环路径，避免把闭包开销混进某一方。
- parse / stringify 都有基线扣除。
- `FastJSON` 明确区分三种语义：`失配`、`严冷`、`热缓存`。
- `FastJSON` 冷路径使用“有限变体循环 + 批量冷操作聚合”的方式维持 cache-miss 负载。
- 自动放大迭代次数，尽量避开 `getTimer()` 分辨率陷阱。
- 冷路径规模校准基于重复采样的中位数，而不是单次试跑。
- 在进入性能区前，套件会先自校验字符串变体和对象变体确实不同。
- 只有双方都达到高置信度时才输出倍率，避免低置信度样本误导结论。
