# Web Panel 沉浸全屏化 — 全量审计与施工方向（2026-06-11）

**文档角色**：把所有运行时 Web Panel 在「固定 16:9、不做多分辨率响应式」约束下统一为**沉浸式全屏**（对标 tasks 任务面板）的审计结论 + 确认方向 + 波次施工计划。一次性过程/取舍/踩坑记录留本文；协议与目录变更同步 [launcher/README.md](../launcher/README.md) Panel System 节与 [agentsDoc/as2-web-panel-migration.md](../agentsDoc/as2-web-panel-migration.md)。

**审计方法**：21-agent workflow（每面板 1 审计 + 1 对抗证伪 + 1 机制 ground truth），全部结论引用 `file:line`。

---

## 1. 背景与目标

- 用户已确认：Web Panel 保持**全屏沉浸**体验最好（tasks 是黄金标准——铺满整个游戏视口、无黑边、无浮窗卡片感、可操作与视觉空间最大）。
- 痛点：部分面板（尤其 map）不铺满，靠暗 `#panel-backdrop` 把居中卡片浮在「黑屏」上——既割裂沉浸感又浪费屏幕面积。
- 约束：**保持固定 16:9 长宽比，不做多端分辨率响应式 reflow，只做整体等比缩放**（明确的成本约束）。

## 2. 机制 Ground Truth（两级全屏开关 + 16:9 不变量）

**anchor 恒 16:9 是硬不变量**：`FlashCoordinateMapper` 永远以 `1024f×576f` 构造（[WebOverlayForm.cs:331](../launcher/src/Guardian/WebOverlayForm.cs)），`CalcViewport` 用写死的 `stageAspect=1024/576` 居中 letterbox-strip（[FlashCoordinateMapper.cs:31-52](../launcher/src/Guardian/FlashCoordinateMapper.cs)）。用户把窗口拉成 4:3/21:9/竖屏，anchor 只会居中变小、露 Flash SA 自己的黑边，**anchor 矩形本身永远严格 16:9**。主 SWF 原生舞台实测 1024×576（FWS 头 nbits=16）。⟹「1024×576 画布 + inset:0 + `Math.min(w/1024,h/576)` 等比缩放 → 零 letterbox 无损铺满」对任何窗口比例永真。

**两级全屏开关**：
1. **C# WebView 覆盖面** — [PanelLayoutCatalog.cs](../launcher/src/Guardian/PanelLayoutCatalog.cs) `GetRect()`：默认 `return anchorScreenRect`（全 anchor 满屏，:41）；仅 jukebox(`Centered(880,620)`)、arena(`Centered(1024,720)`) 走居中子矩形（:29-40）。⚠️ :44-55 的 switch 在 `#pragma 0162 unreachable` 段内是**死代码占位**，永不执行。
2. **CSS 内容框** — `#panel-content` 的 `inset`（[panels.css:156](../launcher/web/css/panels.css)）：默认 `inset:4% 6%`（居中卡片）+ `#panel-backdrop` 暗色径向渐变铺满（:154-155）= 「黑屏浮窗」。tasks 覆盖 `inset:0`（:280-282 黄金标准）。

**第三层黑屏源（C#）**：`PanelHostController.CaptureBackdrop` 抓 Flash 截图按 `dimAlpha`(普通160/黑帧220)压暗铺满 anchor（[PanelHostController.cs:345-396](../launcher/src/Guardian/PanelHostController.cs)）；截图失败降级纯黑 `#08080c`。⟹ 露出区呈暗色 = 即使 web backdrop 透明也黑。**消除黑屏必须 inset:0 + 面板自身铺满 16:9 底图**（让这三层全被遮住）。

**vestigial 机制**：`panel_viewport_set` / `--panel-w/--panel-h` 管道存在（[panels.js:145-153](../launcher/web/modules/panels.js)）但**全 web 树 0 处 CSS 消费**它——沉浸全屏不依赖也不应依赖它。

**沉浸全屏标准配方**（照搬 tasks）：
1. C# `PanelLayoutCatalog` 该 panel 走全 anchor（多数已是，jukebox/arena 需改）。
2. CSS `#panel-container[data-panel="<id>"] #panel-content { inset:0 }`。
3. 固定 16:9 画布 shell：`width:1024px; height:576px; transform:scale(var(--scale)); transform-origin:top left`（仿 [task_panel.css:12-32](../launcher/web/css/task_panel.css)）。
4. JS：`scale = Math.min(el.clientWidth/1024, el.clientHeight/576)` 写 CSS 变量 + 绑 resize/ResizeObserver（仿 [task-panel.js:382-389](../launcher/web/modules/tasks/task-panel.js)）。
5. 全铺底图：`background-size:100% 100%` 覆盖整 16:9（本项目先用 CSS 渐变/底色铺满，留 `background-image` 插槽后补图）。
- **不要**：用 panel_viewport_set 做 reflow；保留 `height:100%` flex 流式（reflow 病灶）；让内部 canvas 走 `Math.min(widthScale,heightScale)` content-fit（美术 aspect≠16:9 会二次 letterbox）。

## 3. 关键洞察：真正的判别维度是「布局引擎」

「切 inset:0」这个看似廉价的修复是陷阱——它只对两类面板无损。按布局引擎分五类：

| 引擎 | 面板 | 全屏可行性 |
|------|------|-----------|
| **固定 1024×576 画布 + transform:scale** | tasks, team(pet/merc) | A — 已是/差一行 inset:0 就无损 |
| **纯文档流（flex+文字）** | help | A — inset:0 + 去切角，文档流自然铺满（配 max-width 防行宽过长）|
| **fluid-flex 固定 px（无 scale）** | kshop, arena | **B** — 放大只 reflow + 密度漂移（违反硬约束），须 retrofit scale-shell |
| **canvas/正方形玩区 content-fit；非16:9 固定画布** | map, stage-select, minigames, intelligence | **B** — 美术/玩区比例≠16:9，须重排坐标/底图/空间 |
| **工具型伴随浮窗（无底图无内容量）** | jukebox | 本判 C，用户选择升 B（需补全画幅电台美术，先 CSS 底）|

**高杠杆基建**：tasks/pet/merc 各自重复实现了同一套 `1024×576 + transform:scale + Math.min(w/1024,h/576)`。所有 B 类 retrofit 本质都是「套这层 shell」。⟹ Wave 2 先抽**共享 `.panel-scale-shell` primitive（CSS 类 + JS helper）**，后续每个 B 面板 = 包一层 + 出底图。

### 三条改变结论的证伪修正
1. **kshop A→B**（最大修正）：fluid-flex 全是固定 px、无 transform:scale，inset:0 后放大只 reflow（grid 2→4 列）+ 字不放大 + 视觉密度漂移，正踩「不做响应式」红线（[panels.css:485-627](../launcher/web/css/panels.css)）。
2. **minigames = web-prototype 非 shipped**：唯一入口 notch「其他▸」dev 测试菜单，全 `mode:dev`，`minigame_session` 仅 `LogManager.Log` 不接经济/存档（[WebOverlayForm.cs:3536-3551](../launcher/src/Guardian/WebOverlayForm.cs)），lockbox kicker 自标 `// WEBVIEW PROTOTYPE //`。这是用户问的「未实际上线可放弃」精确落点。
3. **jukebox C = 放弃全屏化非弃面板**（已上线）；用户最终选择仍升 B 全屏化。

- **Wave 3 — B 类按成本升序**（每个：包 scale-shell + inset:0 + CSS 底铺 16:9 + 留底图插槽）：
  1. `intelligence`（low-med）：画布 1180×790→1404×790（=790×16/9 保密度）；清 [panels.css:5326-5345](../launcher/web/css/panels.css) 遗留两栏死块 + 两处 `@media(max-width:900px)`(5583-5597, 1412-1417)；横向变宽空间分配。

  3. `stage-select`（med 偏上）：固定 1024×576 scale-shell 取代独立 contain（maxScale 1.65）；header 改舞台之上覆盖 HUD；**16 帧底图高度归一**（552/555/556/614/768 五种 → 统一 576/object-fit:cover）；保留 `.is-runtime` 门控不破坏 dev/harness。

- **暂缓**：`minigames`（先建 AS2 正式入口 gameCommand + `minigame_session` 回流接经济/存档，再投全屏）。
