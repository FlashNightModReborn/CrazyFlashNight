# 地图界面 WebView 迁移路线图

**文档角色**：地图界面迁移 canonical doc。  
**最后核对代码基线**：commit `9f8f0c225`（2026-04-20）。

## 1. 结论

`flashswf/UI/地图界面` 应按长期替代路线迁移到 Launcher WebView `panel` 层。

这次迁移的核心目标不是单纯“把一个 Flash UI 复刻成 Web 页面”，而是：

- 持续降低重位图 UI 对 Flash / AS2 运行时的拖累
- 延续既有 UI 迁移已验证的帧率稳定性收益
- 提升地图界面的打开与重载响应体验
- 让地图后续迭代脱离时间轴脚本和重资源常驻模式
- 为更高上限的视觉表达、预览器和可视化构建铺路

当前默认判断：

- 收益预期：高
- 首期实现难度：高
- 长期维护价值：高
- 长期双轨维护价值：低

默认路线不是“长期保留 Flash 地图作为正式回退”，而是“开发期保留临时切换口，稳定后完成替代并清理旧路径”。

## 2. 立项目标

### 2.1 主要目标

- 把地图界面迁入 WebView `panel` 运行时，脱离 Flash 舞台常驻重位图
- 把热点、NPC 点位、任务闪光、楼层切换、跳图入口改造成可维护的数据驱动表达
- 把地图迭代从“改时间轴 + 重编译”转向“改 manifest / 资源 / panel 逻辑”
- 为后续视觉升级保留更强渲染能力

### 2.2 次要目标

- 形成一条可复用的“热点 UI 迁移”方法论
- 为 preview / harness 和未来可视化构建器提供稳定 schema
- 把地图目录下的资产、导出规则、命名约束收敛为显式文档

### 2.3 非目标

- 不在首期追求通用低代码 UI 编辑器
- 不在首期重做整个地图的美术设计语言
- 不以“总进程内存绝对值下降”作为首要成功标准
- 不长期维护 Flash / Web 两套正式地图实现

## 3. 背景判断

当前地图界面属于典型的高价值迁移对象，原因不是理论推测，而是已有项目经验已经给出方向性证据：

- 既有 UI 迁移已稳定提升帧率表现和帧率稳定性
- 重加载型 UI 的打开响应已有改善
- 地图界面使用频率高，且仍有持续施工与扩展需求
- 地图界面重位图占用和时间轴逻辑耦合，天然放大了 Flash 侧压力

由于 AS2 运行时缺乏可靠、低成本、持续可用的精确定量观测，本路线图默认接受如下事实：

- 成功验收以体验代理指标和行为等价为主
- 定量指标可做辅助，但不作为是否继续推进的唯一门槛
- 迁移价值更多体现为“持续收益 + 可维护性收益 + 视觉上限收益”

## 4. 成功标准

### 4.1 一期成功标准

- 地图可以通过 WebView `panel` 独占打开与关闭
- 关键楼层、热点、跳图路径、NPC 点位与任务闪光行为达到可用等价
- 地图主要更新不再依赖 Flash 时间轴改动
- 旧地图只保留开发期临时切换口，不作为长期正式回退

### 4.2 二期成功标准

- 全图行为 parity 达成
- 新增热点、移动点位、替换背景的修改成本明显低于当前 Flash 工作流
- 视觉升级不再受 Flash 时间轴与舞台表达能力明显限制
- preview / harness 可以承担日常校准与回归检查

### 4.3 体验代理指标

以下指标作为默认验收代理，而不是硬门槛式 KPI：

- 打开地图时的主观流畅度更稳定
- 长时游玩后的地图打开体验更稳定
- 其他重加载 UI 受地图资源常驻影响更小
- 地图相关迭代在修改、复核、回归上的往返成本更低
- 常规施工不再要求回到 Flash 时间轴找实例与脚本

## 5. 目标架构

地图迁移后的推荐边界如下：

| 层 | 责任 |
|------|------|
| `flashswf/UI/地图界面` | 旧 XFL 资产、布局真相源、导出参照 |
| XFL / DOM XML | 位置、层级、实例、帧态的原始真相源 |
| FFDec / 导出脚本 | 位图导出、linkage 对照、缺图补齐 |
| Manifest / Schema | 背景、热点、NPC 点位、闪光点、楼层状态、跳图目标 |
| Web Panel | 渲染、交互、动效、状态切换、视觉升级 |
| Host / Bridge | 打开关闭、状态同步、跳图命令、输入与生命周期管理 |

迁移的推荐数据流：

1. 从 XFL / DOM XML 读取布局和命名
2. 通过 FFDec 或脚本导出位图与必要符号参照
3. 生成或维护 manifest
4. Web panel 按 manifest 渲染地图
5. Host / Bridge 负责与游戏运行时交换状态与命令

## 6. 真相源约定

在 preview / harness 和构建器成熟前，必须保持真相源清晰：

- `XFL / LIBRARY XML` 是历史布局与逻辑真相源
- 导出的 PNG 只是资源产物，不是位置真相源
- Web 侧的 `manifest` 是迁移后运行时真相源
- 任何“只改 DOM 绝对坐标、不回写 manifest”的做法都视为临时调试，不是最终结果

默认约束：

- 不直接以 FFDec 导出结果推断最终布局
- 不手写一份与 XFL 无法对照的大型 HTML 绝对定位页面
- 不把 Flash 时间轴脚本照搬成不可维护的前端散逻辑

## 7. 施工策略

首期遵循“先替代、后升级、再构建”的顺序。

### 7.1 Phase 0：立项与真相源梳理

建议周期：`1-2` 个工作日

目标：

- 固化迁移范围和首期边界
- 盘点地图界面的楼层、热点、跳图、任务态和点位类型
- 明确哪些行为属于必须 parity，哪些行为允许延后

产出：

- 本路线图定稿
- 地图目录 README
- 地图 manifest 草案
- 行为清单与迁移优先级

完成标准：

- 一期边界固定，不再把 builder、重设计、全量特效绑进首期

### 7.2 Phase 1：Panel 壳与桥接协议

建议周期：`2-4` 个工作日

目标：

- 建立独立地图 panel
- 建立专用 `map` 协议，不复用仅适合 HUD 的扁平 `UiData`

最低产出：

- `open / close` 生命周期
- `map_snapshot` 或等价初始化数据通道
- `map_command` 或等价命令通道
- 跳图与关闭行为链路
- 开发期临时切换口

完成标准：

- Web panel 可以独立开关，且不依赖最终全量 manifest 即可联调

### 7.3 Phase 2：资产导出与 Manifest 建模

建议周期：`2-5` 个工作日

目标：

- 建立“背景 + 热点 + 点位 + 状态”的数据表达
- 打通 XFL / FFDec 到 manifest 的最小导出链

推荐 manifest 维度：

- 背景层
- 楼层或场景节点
- 可点击热点
- NPC 头像 / 任务点位
- 闪光提示
- 显示条件
- 跳图目标
- 交互状态

完成标准：

- 至少一个试点分区可以完全由 manifest 驱动渲染

### 7.4 Phase 3：试点分区迁移

建议周期：`3-5` 个工作日

目标：

- 先迁一个高频、结构相对清晰的分区
- 验证交互 parity、资源策略和维护体验

推荐试点选择原则：

- 热点密集但范围可控
- 覆盖典型功能
- 能反映未来重复工作量

完成标准：

- 试点分区实现 1:1 布局、热点、跳图、状态反馈
- 确认路线可扩展到全图

### 7.5 Phase 4：全图 Parity 迁移

建议周期：`2-3` 周

目标：

- 扩展到全地图
- 迁移主要逻辑与状态表现
- 把常规地图维护切到 Web 路径

完成标准：

- 全图主路径行为可用
- 新增或调整热点不再依赖 Flash 时间轴编辑
- Flash 地图从主路径降级为开发期临时兜底

### 7.6 Phase 5：默认切换与旧路径清理

建议周期：`2-4` 个工作日

目标：

- 把 Web 地图切为默认路径
- 清理长期双轨维护成本

旧路径移除条件：

- 全图主路径稳定
- 关键操作无阻塞级 bug
- 近一轮地图施工可以在新链路独立完成

完成标准：

- 正式默认使用 Web 地图
- 旧路径不再作为长期正式实现

### 7.7 Phase 6：Preview / Harness

建议周期：`1-2` 周

目标：

- 建立最小可视化预览器，服务日常校准
- 让位置与状态调整脱离手改坐标

最低能力：

- 读取 manifest
- 显示背景与层级
- 显示热点与点位
- 调试显示条件
- 快速校准与导出

完成标准：

- 日常地图迭代可以先走 preview 再进正式 panel

### 7.8 Phase 7：可视化构建器

建议周期：二期单列

目标：

- 让地图作者在更接近所见即所得的界面中维护热点与点位

说明：

- 它是长期投资项，不应阻塞首期替代
- 只有在 manifest 与 preview 稳定后再启动才划算

## 8. 实施顺序与依赖

推荐依赖顺序如下：

1. 先确定 schema 和协议
2. 再做试点分区闭环
3. 再扩展到全图 parity
4. 再删除长期旧路径
5. 最后做 preview / builder 深化维护体验

不推荐的顺序：

- 先手写完整 HTML / CSS 地图再补数据模型
- 先做通用编辑器再证明地图迁移成立
- 在 parity 之前大范围重设计视觉

## 9. 工作流分工

建议至少按四条工作流并行组织：

| 工作流 | 内容 |
|------|------|
| 运行时工作流 | Panel 壳、协议、Host / Bridge 接入、输入与生命周期 |
| 资产工作流 | XFL 盘点、位图导出、命名清洗、资源归档 |
| 数据工作流 | Manifest 设计、状态建模、点位与热点映射 |
| 体验工作流 | parity 校验、视觉升级、预览器与后续构建器 |

## 10. 风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| 布局真相不清 | 仅靠导出位图无法复原层级与状态 | 以 XFL / DOM XML 为布局真相源 |
| 逻辑迁移失控 | 时间轴脚本直接散翻译为前端脚本 | 先抽命令与状态，再写结构化运行时 |
| 维护体验倒退 | 手改 DOM 坐标比 Flash 更痛苦 | 尽早引入 manifest 与 preview |
| 视觉升级挤占 parity | 在替代前过度追求重设计 | 首期只做必要优化，重升级后置 |
| 双轨维护拉长 | 旧 Flash 路径长期不删 | 明确旧路径移除条件 |

## 11. 里程碑验收

| 里程碑 | 验收问题 |
|------|------|
| M1：壳与协议 | 地图 panel 是否能稳定开关并接收专用状态 |
| M2：试点分区 | 是否已经证明 manifest 路线可行 |
| M3：全图 parity | 主路径是否已能脱离 Flash 时间轴运行 |
| M4：默认切换 | 是否已不需要长期双轨维护 |
| M5：preview | 日常施工是否已明显比旧工作流轻 |

## 12. 默认决策

- 默认把地图迁移视为长期技术投资，不再按“是否值得试一次”讨论
- 默认首期目标是替代 Flash 主路径，而不是证明总内存绝对值
- 默认不建设长期正式回退方案，只保留开发期临时切换口
- 默认以 manifest 驱动为中心，而不是 HTML 复刻为中心
- 默认 preview 先于 builder
