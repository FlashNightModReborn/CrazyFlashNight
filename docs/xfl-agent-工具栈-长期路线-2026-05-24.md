## 1. 为什么要长期投资

| 事实 | 含义 |
|---|---|
| FLA / XFL 是项目美术资产**唯一**编辑入口 | 这块流程出问题没有第二条路绕 |
| AS2 + Flash CS6 是**冷门栈** | agent 训练数据稀薄，每次新操作都要从零验证 |
| XFL 是 XML，**可机器读写** | agent 不需要 GUI 就能做大量整理类工作 |
| 手工 FLA 编辑越频繁，错误累积越多 | 命名失序、孤儿符号、linkageId 撞车、Include 失同步、`恢复_*.swf` 残留污染 asset_source_map…… 这些都是已发生过的 bug 类型 |
| AS2 已有 [as2-anti-hallucination.md](../agentsDoc/as2-anti-hallucination.md) 护栏 | XFL 还没有等价的 agent 护栏 |

**投资目标**：让 agent 能逐步、安全地做更多 XFL 操作，把人的注意力从机械整理释放到艺术决策；同时为冷门领域沉淀可复用的"机器可读的项目记忆"。

**反目标**（不要做的）：
- 不替代 Flash CS6 GUI 做艺术创作（DOMShape 这层 XFL 文档稀缺，机器生成蓝图代价过高）
- 不重复 [FlashCS6自动化编译.md](../scripts/FlashCS6自动化编译.md) 已覆盖的编译 / smoke 链路
- 不做"AI 生成 icon"类 agent 能力外的需求

## 2. Layer 划分

### Layer 0 — 校准层（已施工 · 完成度 1.0）

**角色**：safety net；每次 FLA 施工后跑一遍，保证 XFL 内部一致 + asset map 干净。不做任何"创造性"操作。

**已交付**：
- [scripts/tools/xfl/audit.py](../scripts/tools/xfl/audit.py) — 7 项只读检查
- [scripts/tools/xfl/rename_a_class.py](../scripts/tools/xfl/rename_a_class.py) — A 类 `Symbol NNN` → linkageId
- [scripts/tools/xfl/fix_includes.py](../scripts/tools/xfl/fix_includes.py) — DOMDocument include 修复
- [tools/linkage_scanner/scan_linkage.py](../tools/linkage_scanner/scan_linkage.py) — 重写 [asset_source_map.xml](../data/items/asset_source_map.xml)

**验证基线**：commit `624dbc8d5` 的 things.swf 整理 + 本轮 things2 / 收割者无人机 / 公共素材位0 多 XFL 抽测。

**Layer 0 的边界**：只识别明确无歧义的修复（A 类 Symbol NNN 重命名、itemID 兜底的 Include 修复）。**遇到 linkageId 撞车（历史 copy-paste 错误，如 M134 / M202火箭发射器）一律不动**，必须人工去 CS6 修——理由：自动改会把美术内容关联错，错误代价远大于人工成本。

---

### Layer 1 — Inspection（建议次步）

**触发条件**：当人 / agent 需要"这个符号在哪用了 / 这个 XFL 有什么"这类查询，**3 个月内手工 grep + 推理超过 5 次**。

**最小可交付**（只读，三个独立脚本）：
- `where_used.py <xfl_root> <symbol_name_or_lid>` — 反向引用查询，覆盖 libraryItemName + AS attachMovie 字符串（后者跨 XFL 全仓 grep）
- `inspect.py <xfl_root> <symbol_name_or_lid>` — 单符号详情：层 / 帧 / 子符号 / itemID / linkageExportForAS
- `dead_symbols.py <xfl_root>` — 列出"A 类但无引用"的候选孤儿（需配套 AS 全仓字符串 grep，结果只作候选不直接删）

**风险**：低（只读）。

**校准测试**：在 things 上跑 3 个已知答案的查询，结果与手工 grep + 人工判定一致。

**何时不建**：如果上述查询都能用 [tools/linkage_scanner/scan_linkage.py](../tools/linkage_scanner/scan_linkage.py) 的输出 + `grep` 解决，就别建。

---

### Layer 2 — Refactor（中期，不急）

**触发条件**：
- 发生第二次"想拆分 things.swf 但不敢动手"
- 或：跨 XFL 资产搬迁需求出现（如 things → things-new 整理）

**最小可交付**（写操作，强制 `--dry-run` 默认）：
- `rename_symbol.py` — 通用 rename（不限于 Symbol NNN）
- `move_folder.py` — LIBRARY 文件夹批量挪 + 引用同步
- `migrate_symbol.py` — 跨 XFL 符号搬迁（含冲突检查 + itemID 重生策略）

**风险**：中（多文件写；引用图错一处就连锁失同步）。

**校准测试**：操作后必跑 Layer 0 audit + linkage_scanner 复检 + 在 CS6 打开目标 FLA 确认无报错 + 编译 smoke。

**前置**：Layer 1 已稳定运行 3 个月以上。

---

### Layer 3 — Generation（延后）

**触发条件**：发生**实际**的批量生成需求（如一次造 30 个色变 icon、批量给所有武器配套生成图标 wrapper）。**不能因为"听起来酷"就启动。**

**风险**：高（DOMShape / DOMBitmapInstance 层 XFL 官方文档稀缺，机器生成蓝图正确性难保证）。

**校准测试**：生成结果能在 CS6 打开 + 编译 + 视觉验收 + Layer 0 audit。

**何时永久放弃**：若 6 个月内未出现 3+ 次明确批量生成需求，标记为不建。

---

### Layer 4 — Semantic Diff / Merge（延后）

**触发条件**：FLA 协作分支出现 / FLA 改动的代码评审成为瓶颈。

**单人开发期可能永远不需要**。

## 3. 投资节奏 / 决策协议

| 规则 | 说明 |
|---|---|
| 每个 Layer 启动前 | 明确 1 个**真实发生过**的 use case，不投机性建设 |
| 每个新脚本 | 必须先 `--dry-run` + 在 1 个 small XFL 上验证 + 写校准测试 |
| 每个 Layer 跨越前 | 把已建脚本在 **3+ 不同 XFL** 上跑一遍无 regression |
| 每个 Layer 之间的隔离 | 上层失败不能污染下层；Layer 0 必须永远能独立跑通 |
| 每年一次回看 | Layer N 是否真用，没用就降级标记为「不建」并写明理由 |
| 触发条件未满足 | **明确拒绝**建设，写入本文档作为决策 |

**冷门栈特别约束**：
- 不信赖 LLM 关于 XFL / AS2 的"通识"，所有断言都要在仓库内或 CS6 GUI 验证
- 每个新发现的 XFL 行为坑写入 [agentsDoc/as2-anti-hallucination.md](../agentsDoc/as2-anti-hallucination.md) 的兄弟文档（或单独 `xfl-anti-hallucination.md`，待 §4 累积到阈值时建）

## 5. 相关但分层独立的能力

### 5.1 FLA 二进制层 — rig overflow 紧急修复

[scripts/优化随笔/FLA-rigPropagationMatrix-溢出导致元件不可编辑.md](../scripts/优化随笔/FLA-rigPropagationMatrix-溢出导致元件不可编辑.md)（commit `681bbecd0`，2026-05-23 入库）

- **失效层**：FLA（zip 包，未解开），不是 XFL（已解开的文本树）
- **触发**：Adobe Animate Asset Warp Rig 在大角度旋转 + 大缩放 motion tween 编辑下，`rigPropagationMatrix` 16.16 定点缓存累积溢出
- **症状识别器**：「元件坐标轴歪斜 + 编辑闪退 + 无法另存 XFL」三件套同时出现
- **修复**：doc 内含 `fix_rig.py` + `pack_fla.py` 两段可复用脚本（解包 → 剥离损坏分量 → 重打 zip）
- **使用频率**：罕见紧急（实测一次 922 条损坏修复）；不进入本路线图 Layer 划分，按 emergency 治理
- **与 Layer 0 的边界**：Layer 0 是「每次施工后定期跑」的 XFL 文本层体检；rig overflow 是「FLA 编辑过程中触发后才修」的 FLA 二进制层应急。两者操作的文件树不同（zipped vs unpacked）、调用时机不同、风险不同——不要合并

如果未来 rig overflow 类紧急修复出现第二种、第三种（如 bin 缓存损坏、ZipEntry 表损坏），再考虑在 [scripts/tools/](../scripts/tools/) 下新建 `fla/` 子目录把 emergency 脚本从 doc 内嵌代码块抽出来。**仅一种时不抽**：doc 自包含可复用价值已经足够。

### 5.2 不在本路线图覆盖范围
