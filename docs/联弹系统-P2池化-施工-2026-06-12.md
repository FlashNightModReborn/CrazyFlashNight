# 联弹系统 P2 池化施工记录 + P3 去MC化审计（2026-06-12）

> 主线背景与 P0/P1（配置治理 + 纵向补弹速率）见会话记忆 `chain-bullet-modernization.md`。
> 本文记录 P2（单元体池化）施工与 P3（去影片剪辑化）审计结论，供验收与后续施工使用。

## 一、P2 施工内容

### 1.1 架构

- **新类** [ChainUnitManager.as](../scripts/类定义/org/flashNight/arki/bullet/BulletComponent/Chain/ChainUnitManager.as)：
  - 共享层 `gameworld.子弹区域.联弹单元体层`（深度 1048000，懒建；gameworld 重建后自动重建并重置组注册表）
  - 按 linkage（`单元体-{子弹种类}`）池化：自由表 + `_visible` 切换，无 attachMovie/removeMovieClip 周转
  - 组注册表 + 单一 `layer.onEnterFrame` 统一 tick（替代旧实现每发联弹一个 onEnterFrame 闭包）
  - 组删除/单元体删除均为 swap-with-last + pop（O(1)）
  - teardown 双保险：消失帧脚本显式 `removeGroupByBullet` + tick 检测 `area/bullet._parent == undefined` 兜底
- **帧脚本重写** [战斗系统_fs_联弹管理.as](../scripts/逻辑/战斗系统/战斗系统_fs_联弹管理.as)：
  - 五类型（横向/横向拖尾/纵向/滑翔/爆炸）全部迁移为 `初始化(clip)`（建组+生成单元体）+ `X更新(group)`（统一 tick 分发）
  - **模拟保持在子弹本地坐标系**，数学公式与旧实现逐式等价；单元体为纯数据 `{mc,x,y,rot,...}`
  - **渲染**：本地坐标经子弹仿射矩阵（`_x/_y/_rotation/_xscale/_yscale`，支持镜像兜底分支）映射到共享层；共享层与子弹同为 子弹区域 直接子级，无需 localToGlobal
  - **area 子剪辑仍是碰撞代理**：包围盒公式不变 → FLAG_CHAIN 碰撞管线零改动
  - **FLA 零改动**：`X联弹初始化(this=area)` / `联弹消失(this=子弹)` / `爆炸联弹消失(this=子弹)` 入口签名不变
  - 旧 `_root.创建单元体` / `_root.回收单元体` 退役（外部零引用已验证）

### 1.2 有意行为差异（验收口径）

1. 衰竭回收 `splice(j--,1)` 的旧实现每次删除会多跳过一个单元体一帧更新（疑似笔误），新实现 swap-pop 不跳帧，视觉更平滑
2. 消弹直删等不经消失帧的销毁路径，单元体由 tick 兜底回收，**至多 1 帧视觉残留**
3. 单元体渲染层级固定在 子弹区域 顶层（旧实现跟随各自子弹深度）
4. `爆炸联弹预处理` 首次被正确注册——旧文件中它在 `_root.联弹系统 = {}` 重置**之前**赋值，实际从未注册（爆炸联弹此前预处理静默未执行）
5. `爆炸联弹消失` 的逐单元体爆炸定位修正为真实单元体位置（旧代码对单元体自身调 localToGlobal 又传入其自身坐标，偏移叠加两次）
6. 统一 tick 与子弹自身 onEnterFrame 的帧内相位可能与旧实现存在亚帧级次序差（包围盒为本地坐标计算，不受影响）

### 1.2b 复审修复（同日第五轮，compile 0错）

1. **爆炸联弹重复结算**：恢复消失帧中 `clip.area.removeMovieClip()`（在 shoot传递 之前）——area 跨越消失帧，不移除会被 BulletQueueProcessor 预检查（仅检测 area 属性）再次入队，原联弹碰撞与新爆炸碰撞重叠
2. **场景切换静态引用泄漏**：ChainUnitManager 订阅 `SceneChanged`（EventBus，订阅一次）主动 `resetAll()` 清空组注册表，不再等到下次联弹层懒建才释放旧 gameworld/旧池引用
3. **池化时间轴重置契约**：release 时 `stop()`（隐藏的高水位实例不再逐帧空转），acquire 复用时 `gotoAndPlay(1)`（与全新 attachMovie 等价，多帧单元体如 单元体-铁枪能量子弹 不得从任意帧续播）
4. **渲染继承**：单元体每帧继承父子弹 `_xscale/_yscale/_alpha/_visible`（复刻旧子剪辑视觉继承；非等比缩放+子旋转的斜切无法用 scale/rotation 表达，取最近似分解，常规子弹 100% 等比不受影响）

## 二、P3 去MC化审计结论

### 2.1 MC 依赖面（FLAG_CHAIN 路径硬绑定，三处）

| # | 依赖 | 位置 | 含义 |
|---|------|------|------|
| 1 | `bullet.onEnterFrame = BulletQueueProcessor.createNormalPreCheck(target)` | BulletLifecycle.bindFrameHandler | 帧泵入口：子弹靠自身 MC 帧事件进入队列处理。纯对象无帧事件 → 需在 BulletQueueProcessor 增设对象子弹泵列表（对标 `_rayBullets`，但多帧存活） |
| 2 | `bullet.gotoAndPlay("消失")` | BulletQueueProcessor 单出口收尾（VANISH/击中地图） | 击杀路径依赖 FLA 帧标签（消失帧脚本承载死亡行为）。可仿 `bindSafeRemove` 在对象上安装 `gotoAndPlay` 方法垫片转发到 `_root.联弹系统.联弹消失` |
| 3 | `factory.createFromBullet(target, target.area)` + 旋转联弹多边形惰性创建 + CoverageAABB | NormalBulletLifecycle.bindCollider / ChainDetector / BulletQueueProcessor | AABB/Coverage/Polygon 碰撞器全部以 area MC 为输入。需新增 `updateFromChainData`（从组本地极值+子弹矩阵直接算 OBB/AABB 点集）并在 ColliderFactory 三处接线 |

辅助事实：`bindSafeRemove` 已有透明子弹（纯 Object）双模先例；`shouldDestroy`/`checkMapCollision`/movement 字段均可在纯对象上工作；DepthManager 不介入子弹。

### 2.2 施工方案（押后至 P2 真机验收通过后执行）

1. BulletFactory 联弹分支：`_root.对象浅拷贝(Obj)` 创建对象子弹（保留 xmov/ymov/movement 绑定），注册进新泵列表
2. BulletQueueProcessor：新增 `_chainObjectBullets` 多帧泵（进入主扫描的入口与 MC 子弹同构），对象子弹安装 `gotoAndPlay`/`removeMovieClip` 垫片
3. 碰撞器：`AABBCollider.updateFromChainData(group)` + Polygon 同名路径 + ChainDetector 工厂选择改造
4. 帧脚本：初始化入口从 FLA onClipEvent 迁到 BulletFactory 分支内直接调用（按 `子弹种类` 前缀查模板注册表），9 个 FLA 元件 + `_root.联弹系统` 帧标签语义退役
5. 每步 compile 门控 + 真机回归（消弹三模式/穿透/击中地图/爆炸联弹全路径）

**降级决策依据**：①~③ 均处 BulletQueueProcessor 热循环及其契约层，盲改（无真机验证窗口）与 P2 未验收叠加的风险不对称；P2 已消除主要性能开销（attachMovie 周转 + 每弹闭包），薄壳 MC 的边际成本 = 每发一个两帧空壳 + area 子件。

### 2.3 P3 已施工（P2 验收通过后同日实施，compile 0错0警）

按 §2.2 方案完整落地，**六个枪式联弹模板（横向/纵向 × 基础/机枪/手枪）全部对象化**；
拖尾系（trail 矢量绘制依赖 area 画布）与 滑翔/爆炸（帧标签+子弹区域area 联动）暂保留 MC 壳。

**改动清单**：

1. **碰撞器数据路径**：`AABBCollider.updateFromChainObject`（组本地碰撞盒四角 × 子弹仿射矩阵 → 子弹区域 localToGlobal → gameworld AABB，与 `area.getRect(_root.gameworld)` 等价）；`PolygonCollider.updateFromChainObject`（盒中心 + 固有半宽高 12.5 + 子弹旋转构造 OBB，与 MC 路径"取 detectionArea 自身坐标系矩形（固有 25×25 不随实例缩放）"的语义等价）；三工厂（AABB/Coverage/Polygon）各加 `createFromChainObject`（扩展方法不在 IColliderFactory 接口内，调用方经无类型引用分发）
2. **ChainObjectLifecycle**（继承 NormalBulletLifecycle）：bindCollider 走数据路径（工厂仍由 ChainDetector 按旋转角选 AABB/Coverage）；bindFrameHandler 空实现；bindSafeRemove 安装对象式 removeMovieClip（回收双碰撞器 + 回收单元体组 + `__chainDead` 标记）与 gotoAndPlay/gotoAndStop/stop 垫片（队列单出口 `gotoAndPlay("消失")` 对两形态统一生效）
3. **联弹管理**：横向/纵向初始化拆出「组装核心」（MC/对象双模共用）；更新函数碰撞盒写入组字段（盒x/盒y/盒宽/盒高），area 存在时镜像；`对象化模板` 注册表（FLA 授权常量：横向系 [-5,5]²、纵向系 [7,17]×[-5,5]，源自壳元件 联弹area 实例矩阵 25×25×0.4）；`对象联弹初始化`/`对象联弹消失`（复刻消失帧语义：霰弹值耗尽或击中地图→销毁，否则置 flag 继续飞行）
4. **BulletFactory**：`isChainObject` 分支（联弹 ∧ 非透明/射线/单位子弹 ∧ 注册表命中）→ 浅拷贝纯对象 + 显示属性默认值（_xscale/_yscale/_alpha/_visible）+ 普通子弹同款运动绑定 + ChainObjectLifecycle；组构建先于 bindCollider（碰撞器需读组盒）
5. **统一泵**：ChainUnitManager.tick 对象分支承担 MC 子弹 onEnterFrame 预检查职责（越界标记 STATE_HIT_MAP → 数据路径更新 AABB → `BulletQueueProcessor.add`）；processQueue 由 frameEnd 帧末统一消费，入队时序与 MC 同构
6. **processQueue 唯一热循环改动**：多边形惰性创建点（isPointSet 段）按 `子弹区域area || area` 存在性分支——存在走原 MC 路径逐字不动，缺失走 `createFromChainObject/updateFromChainObject`

**等价性时序考古（第一版结论曾写反并导致左射视觉反向，真机回归确认后已修正）**：
AVM1 中 attachMovie 的**时间轴子剪辑在首次渲染才实例化**，area 的 onClipEvent(load)
实际晚于工厂的 xmov 赋值执行——`运动方向系数 = xmov<0 ? -1 : 1` 的生产行为就是
条件值（左射=-1）。对象路径的组装同样在 xmov 赋值之后调用，两模式按条件式一致。
教训：对"载入时序"类考古结论，必须用左右双向真机表现做反例验证，不可仅凭代码顺序推断。

**左射锚点修复（真机回归第二轮）**：纵向补弹落点公式 `系数×localDeltaX×插值` 中
localDelta 本身已是带方向的本地位移（在枪管轴上的投影恒为 +|v|），再乘系数使左射
落点反向展开——新单元体出生于枪口后方（枪管内），斜射时沿对角线放大。旧 MC 实现
靠"原始坐标捕获于一帧位移后"（area 首渲染才实例化）的时序巧合补偿；对象路径在
工厂内捕获（位移前）暴露了该反向。已改为方向无关的物理正确插值（去掉系数），
任意射角落点均沿行进方向展开。注意：若回退 纵向系模板 到 MC 壳，左射落点将比
原版提前约一帧（方向仍正确）。

**回退方式**：删除 联弹管理.as 末尾对应的 `注册对象化模板(...)` 行即整型回退该模板到
MC 壳路径（FLA 元件均未删除），逐模板可灰度。

## 二点五、第六轮全系列复审（同日，覆盖 fb75607→3612b3f 五提交）

逐提交 diff 审阅 + 旧版（fb75607^）公式逐式对照 + 消费端契约交叉验证。结论：

**对照确认等价/正确的关键点**（曾列为疑点，已排除）：
- `击中地图` 属性在 `对象联弹消失`/`联弹消失` 中可用——`BulletInitializer.installStateFlagsAccessors` 为含纯对象在内的所有子弹安装了映射 STATE_HIT_MAP 的 addProperty 访问器
- CoverageAABBCollider 仅重写 checkCollision，update 路径继承 AABBCollider → 对象路径 `updateFromChainObject` 对两种 AABB 工厂等价生效
- 对象子弹的运动（updateMovement）/射程（shouldDestroy）/销毁单出口全部由 processQueue 统一驱动，tick 只补齐 MC onEnterFrame 预检查职责，时序同构
- `#include "../macros/STATE_HIT_MAP.as"` 与 Queue/Lifecycle/Init 七处既有文件同一约定（基准为 .fla 类路径），compile 0 错验证
- isPointSet 的 `rot != 0 && rot != 180` 判别对未归一化角度（对象路径）与 MC 归一化角度分类一致
- handleBounce/handleCancel 对纯对象子弹兼容（纯属性运算 + gotoAndPlay/removeMovieClip 垫片）
- 横向/纵向/拖尾/滑翔/爆炸五类型公式与旧版逐式等价（含衰竭阈值随霰弹值动态、`Math.max(基准*-2, …)` 最小盒、splice 跳帧差异等已记录项）
- 影子记账窗口（霰弹值负编码）只存在于 processQueue 单弹迭代内，tick 阶段读写 霰弹值 无窗口重叠

**本轮修复**：tick 的组分支判别由 `g.area == null` 改为显式 `g.isObject` 标记
（对象联弹初始化 落标）。原因：MC 壳组（拖尾/滑翔/爆炸）存在不经消失帧的死亡路径
（REMOVE 消弹、超射程 priority-4 直删——后者每发飞满射程的拖尾联弹必经），此时组靠
MC 分支 `_parent == undefined` 兜底回收；悬挂 MC 引用与 null 的 loose equality 在
AVM1 中无可靠语义，若判真则死组误入对象分支 → `__chainDead` 恒 undefined → 组永不
注销且每帧向碰撞队列泵入死子弹（僵尸组慢性累积，性能劣化难定位）。显式标记在两种
语义下都正确。compile 0 错 0 警告，SWF 已刷新。

**记录在案的轻微残留**（不修，知悉即可）：
- `__pendingFireInterval` 若写入后射击失败未被消费，可能被下一发不同路径的 fillrate 武器消费到 stale 值——量级毫秒级、仅影响单发补弹密度，自愈
- 池化后单元体渲染相位较旧"子级自动跟随"晚一帧子弹位移（碰撞盒与单元体一致，子弹壳无视觉，命中判定不受影响）——差异⑥的全帧化表述

## 二点七、P4 增强（同日，P2/P3 真机验收通过后）

### A. fillrate 普及化 + 分数补弹率（自适应任意射速）

**动机**：XM556-Preview（interval 235ms ≈ 7帧，霰弹值 5）装 磁稳贯穿弹（压制机枪系
interval (I-50)×1.8 → 333ms ≈ 10帧）后，旧"每帧补1发"在 4 帧内打完弹链再空转 6 帧——
弹链节奏远快于名义射速，破绽明显。

**方案**：
1. **盖戳普及**：去掉 `补弹对齐射速` opt-in 门控（ShootCore 写 pending / WeaponFireCore
   盖戳 发射间隔毫秒 改为无条件，每次射击各多一次属性写，可忽略）；`fillrate=auto`
   降级为历史兼容写法（等同缺省），数值 fillrate = 显式补弹率（允许小数，去掉双 ceil）
2. **分数补弹率 + 累加器**：组装期 `率 = (霰弹值-1)/间隔帧数`（间隔帧数不再 floor 取整、
   不设 1 发/帧下限）；更新期 `acc += 率; n = acc|0; acc -= n` ——
   率<1 自然得到 2,3,2,… 的不均匀隔帧补弹（残差跨帧滚动，长期均值精确等于率），
   率>1 帧内多发沿当帧位移插值（(s+1)/n 恒 ≤1）。n=0 的帧只推进已有单元体
   （子弹仍锚定枪口、位置照常复位，弹链间距=空帧数×相对位移，几何自洽）
3. **回退语义**：技能/装备函数直调 子弹区域shoot传递 的路径无间隔戳 → 率=1（旧行为）

**行为变更面（需手测确认观感）**：所有经射击链路的纵向联弹武器弹链节奏改为与实际
射速对齐（此前仅 XM214 opt-in）。横向联弹（齐射）不受影响。

### B. 热路径性能改造（纪律对齐 agentsDoc/as2-performance.md 与 BQP 既有实现）

每帧热路径（tick → 组更新 → 渲染 → 碰撞器数据路径）全量过 §2a/§2b 硬规则：

| 项 | 改造 | 依据 |
|----|------|------|
| 单元体数据对象 | ChainUnitManager.acquireUnitData/releaseUnitData 自由表池化（mc 回收置 null） | S03/new=847ns |
| 单元体三角函数 | 生成时一次缓存 u.sin/u.cos（枪式/滑翔/爆炸 rot 终身不变；拖尾逐帧改 rot 时跟写缓存） | Math.sin 255ns×每单元体每帧 → 0 |
| 渲染矩阵 | group 级缓存 ma/mb/mc2/md + rcos/rsin，仅 旋转/缩放/透明度/可见性 变化时重算 | H01/H02 |
